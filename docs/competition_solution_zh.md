# TrialCompiler 方案说明

## 1. 方案整体概述

本方案拟构建 **TrialCompiler：可编译、可验证、可继承专家经验的临床文档智能系统**，面向临床试验方案与研究报告篇幅长、专业协作复杂、撰写审核周期长、跨章节内容易不一致以及历史经验难以复用等问题。系统以临床试验目标和结构化事实为统一约束，将传统静态 Word/PDF 文档转化为具有章节、事实、证据和依赖关系的临床文档图，利用多智能体协作完成资料检索、分章节起草、缺陷识别、修改建议、质量复核和正式报告生成。人工审核产生的接受、拒绝与修改记录将进一步沉淀为可追溯的组织经验，并在后续相似项目中自动调用，实现从“辅助撰写一份文档”到“持续提升组织临床文档生产能力”的转变。

## 2. 整体架构与核心功能模块

系统由五个核心模块组成。第一，**临床文档解析与事实图谱模块**，解析方案、报告、表格及历史材料，提取研究目标、主要终点、样本量、访视安排和统计分析等关键事实，形成统一的 Trial Fact Sheet。第二，**知识检索与证据管理模块**，连接监管规范、企业 SOP、历史模板和专家经验，为每项生成内容和审核结论提供可定位的证据来源。第三，**多智能体文档编译模块**，由上下文 Agent、知识 Agent、医学写作 Agent、审核 Agent 和报告 Agent 协同完成分章节起草与循环修订。第四，**文档测试与修改影响分析模块**，自动执行目标与终点一致性、样本量一致性、时间窗冲突和版本残留等检查，并在核心事实变化时识别所有受影响章节。第五，**经验知识复用模块**，将人工审核结果提取为包含适用条件、修改理由和证据的经验卡片。系统以飞书文档作为协作编辑界面，以飞书多维表格管理问题、状态与指标，并通过飞书 Aily 触发智能体工作流和人工审核。

## 3. 核心创新点：技术、模式与流程创新

技术上，本方案提出“**临床文档编译器**”思路，不再将数百页文档视为普通长文本，而是将其构建为可执行、可测试的事实依赖图；当主要终点、样本量或访视时间发生变化时，系统能够进行修改影响传播，定位所有需要同步更新的章节、表格和关联文件。模式上，系统将监管知识、企业规范、项目事实与专家经验分层管理，避免普通 RAG 仅依赖语义相似度检索。流程上，系统建立“AI 起草—自动审核—循环修订—人工确认—经验提取—后续复用”的闭环，将专家每次修改转化为可验证、可继承的组织能力。同时，项目将建设 TrialDocBench，从事实提取、缺陷识别、修改传播和经验迁移四个层级进行量化评价，证明系统在审核效率、缺陷召回、证据追溯和经验复用方面相较通用 LLM 与单 Agent 方案的实际提升。
