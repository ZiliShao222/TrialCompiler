# 医学资料到 TrialCompiler 系统模块的映射说明

## 简短结论

医学同学提交的资料包不是单纯的参考文献集合，而是 TrialCompiler 的医学侧“需求约束层”。这些资料帮助我们把比赛方案从“AI 写临床方案”收敛为“以临床设计事实为核心的文档编译、测试、追踪与增量更新系统”。

资料包可以映射到四个核心系统模块：

1. **法规与伦理资料**：定义系统不能越过的专业责任边界和人工审核门。
2. **试验设计与统计资料**：定义 Trial Fact Sheet 需要管理的关键事实字段。
3. **协议审查与 AI 治理资料**：定义多 Agent 输出如何被审计、解释和限制。
4. **模板与工具资料**：定义 Document Graph 中应覆盖的章节、表格、关联文件和执行系统节点。

## 1. M1：法规伦理与报告规范

M1 资料主要来自 ICH、FDA、EMA、NMPA/CDE 等官方或准官方来源，用于约束 TrialCompiler 的边界。

对应系统模块：

- **Rule Constraint Layer**：记录法规、指南、模板要求和适用范围。
- **Human Review Gate**：区分 AI 可辅助的候选提取、候选文本生成、风险提示，与必须由医学、统计、注册、伦理或质量人员确认的判断。
- **Audit Trail**：记录事实来源、规则版本、模型输出、人工修改和最终批准状态。

对方案表述的影响：

- TrialCompiler 不能声称自动完成医学合理性、统计合理性或注册策略判断。
- 系统输出应定位为“候选事实、候选文本、风险提示和证据定位”，正式版本仍由授权人员批准。
- 当前阶段只使用公开资料和合成案例，不采集真实患者数据、企业内部文件或真实临床项目文件。

## 2. M2：试验设计与统计

M2 资料用于定义 TrialCompiler 需要追踪的关键设计事实。

对应系统模块：

- **Trial Fact Sheet**：管理主要终点、次要终点、评估时间点、样本量、研究人群、入排标准、对照组、随机化、盲法、分层因素、给药方案、访视安排、安全性评估、停药规则、分析集、统计方法、缺失值处理和敏感性分析。
- **Fact Versioning**：记录当前值、历史版本、来源、提出人、审核人、生效状态和适用范围。
- **Change Impact Analysis**：当关键事实变化时计算受影响章节、表格、关联文件和执行配置。

对方案表述的影响：

- TrialCompiler 不是对自然语言做简单摘要，而是把关键设计事实抽取为可计算对象。
- 事实变化不是“改一句话”，而是会影响多个文档单元和执行节点。
- `Week 12 -> Week 16` 这类时间点变化应作为核心演示案例。

## 3. M3：协议审查与 AI 治理

M3 资料用于定义 AI 系统在临床文档场景中的可解释性、可追踪性和风险控制要求。

对应系统模块：

- **Agent Trace**：记录每个 Agent 读入了什么、判断了什么、输出了什么、是否触发返工。
- **Quality Judge**：对候选修订进行独立复核，避免生成 Agent 自评。
- **Experience Compiler**：只把经过授权确认的审核意见沉淀为可复用经验，避免把一次性项目意见错误泛化。
- **Evidence Pack**：为每个风险提示和候选修订提供来源、引用位置和适用范围。

对方案表述的影响：

- 技术创新不应只写“多 Agent 协作”，而要强调职责分离、证据可追踪、人工确认和审计闭环。
- 经验复用不能直接保存原始聊天或批注，而应转化为带触发条件、适用范围、反例和审核状态的 Decision Capsule。

## 4. M4：模板与工具

M4 资料用于定义 TrialCompiler 的文档节点和输出形态。

对应系统模块：

- **Clinical Document Graph**：覆盖 Protocol、Synopsis、SoA、ICF、SAP、CRF/eCRF、IWRS/IRT、药物手册、实验室手册、影像手册、注册字段和 CSR 相关章节。
- **Document Unit Tests**：检查数值一致、语义一致、统计一致和执行一致。
- **Redline and Impact Matrix**：输出红线稿、影响矩阵、证据表和待人工确认项。

对方案表述的影响：

- 我们的对象是“临床文档与执行文件体系”，不是单一 Word 文档。
- Document Graph 需要包含执行侧节点，例如 CRF/eCRF 字段、EDC edit checks、IWRS/IRT 配置和手册流程。
- 系统价值应体现为减少跨文件漏改、旧值残留和执行文件未同步，而不是只提升写作速度。

## 5. 对当前三份比赛材料的修改建议

### 方案整体概述

需要突出：

- 临床试验方案是核心源文件，但它会驱动多类关联文件和执行系统。
- TrialCompiler 通过 Trial Fact Sheet 和 Clinical Document Graph 管理事实传播。
- 当前只使用公开资料和合成案例验证，不使用真实企业案例。

### 整体架构与核心功能模块

需要突出：

- 架构中应有事实层、规则层、文档图层、测试层、变更传播层和人工审核层。
- Document Graph 的节点不只包括章节，也包括 SoA、SAP、ICF、CRF/eCRF、IWRS/IRT 和手册。
- 文档测试应分为数值、语义、统计、执行四层。

### 核心创新点

需要突出：

- TrialCompiler 是临床文档编译器，而不是普通 AI 写作工具。
- 创新来自“事实结构化 + 依赖图 + 单元测试 + 增量变更 + 人工治理经验”的组合。
- 竞品对比应避免只说“我们用了 AI”，而要说明普通 RAG/LLM/Copilot 缺少受控事实层和跨文件依赖测试。

## 6. 后续可交给医学同学复核的问题

1. 当前 Trial Fact Sheet 字段是否覆盖方案设计阶段最常见的关键事实？
2. CRF/eCRF、IWRS/IRT、药物手册、实验室手册、影像手册是否适合作为第一版 Document Graph 节点？
3. 四层一致性检查中，哪些属于医学可判断，哪些属于统计可判断，哪些属于执行/运营可判断？
4. `Week 12 -> Week 16` 是否适合作为比赛 Demo 的主线，是否还需要补一个样本量或入排标准变化案例？
5. 公开资料中哪些可以作为高置信规则来源，哪些只能作为背景材料或竞品参考？
