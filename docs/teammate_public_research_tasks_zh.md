# 队员公开资料复核任务表

## 使用原则

当前阶段不能采集真实企业案例、真实临床项目文件、患者数据、内部 SOP、内部邮件或真实访谈记录。因此队员的工作应集中在公开资料复核、资料摘要和人工判断，不需要写代码。

每条资料复核建议记录：

- 来源标题；
- 官方链接；
- 是否可打开；
- 资料类型；
- 与 TrialCompiler 哪个模块相关；
- 该资料能支持什么判断；
- 哪些内容不能过度推断；
- 是否建议进入比赛 PPT 或方案正文。

## A. 医学同学优先任务

### A1. 复核 Trial Fact Sheet 字段

目标：确认 `references/metadata/trial_fact_field_dictionary.tsv` 中的字段是否覆盖方案设计阶段的关键事实。

重点检查：

- 主要终点、次要终点；
- 终点评估时间；
- 样本量和统计假设；
- 入组/排除标准；
- 对照组设计；
- 随机化、盲法、分层因素；
- 给药方案；
- 访视安排和时间窗；
- 安全性评估；
- 停药、暂停、终止规则；
- 分析集和统计方法；
- CRF/eCRF 数据采集字段；
- IWRS/IRT 配置；
- 方案偏离规则。

建议输出：指出“缺少字段 / 字段名称不准确 / 需要合并 / 需要拆分”的地方即可。

### A2. 复核 Document Graph 节点

目标：确认第一版 Clinical Document Graph 是否应覆盖以下节点。

候选节点：

- Protocol；
- Synopsis；
- SoA；
- ICF；
- SAP；
- CRF/eCRF；
- IWRS/IRT；
- EDC edit checks；
- 药物管理手册；
- 实验室手册；
- 影像手册；
- 注册字段；
- CSR 相关章节。

建议输出：标记每个节点为 `第一版必须 / 第二版再做 / 不适合当前范围`。

### A3. 复核四层一致性检查

目标：确认四层一致性的医学表述是否准确。

四层：

1. 数值一致性；
2. 语义一致性；
3. 统计一致性；
4. 执行一致性。

建议输出：每层给 1-2 个真实但不涉及内部项目的公开/通用例子。

## B. 商业/竞品同学优先任务

### B1. 人工打开直接竞品官网

目标：核验竞品公开能力，避免我们过度宣称。

优先来源：

- HumanTrue；
- Cori Clinical；
- Clinials；
- Evinova Study Document Assistant。

每个竞品需要回答：

1. 它公开说自己解决什么问题？
2. 它是否提到 protocol structure / quality / review / source tracing / version / redline / dependency / cross-document consistency？
3. 它的主要入口是 Word、网页、平台还是 API？
4. 它是否强调企业部署、安全、权限、审计？
5. 它和 TrialCompiler 最大重叠点是什么？
6. TrialCompiler 还能讲出的差异是什么？

### B2. 人工打开相邻平台官网

目标：确认我们不是要替代大型平台，而是作为语义治理层与其互补。

优先来源：

- Veeva；
- Medidata；
- IQVIA；
- 太美医疗；
- Oracle Clinical One；
- MasterControl；
- Montrium。

每个竞品需要回答：

1. 它更像上游方案设计、临床运营平台、受控文档平台、质量平台，还是 disclosure/registry 工具？
2. 它是否直接做 protocol authoring？
3. 它是否公开强调跨文档一致性或修订影响？
4. TrialCompiler 应该与它竞争、补充，还是接入？

## C. 飞书/产品同学优先任务

### C1. 复核飞书可实现能力

目标：判断 TrialCompiler 哪些交互能用飞书实现，哪些必须放在独立后端。

需要核验：

- 飞书 Aily 是否适合做任务入口；
- 飞书多维表格是否适合展示 Trial Fact Sheet；
- 飞书审批是否适合做人类确认门；
- 飞书 Wiki 是否适合展示证据包和审核报告；
- 文件上传、权限和审计是否需要独立后端补充。

建议输出：形成 `飞书可做 / 需要后端 / 暂不做` 三列表。

## D. 当前不建议队员做的事

这些事情交给代码侧或后续授权阶段：

- 写爬虫批量清洗资料；
- 建数据库；
- 处理真实企业文档；
- 接触真实患者数据；
- 找真实药企员工访谈；
- 把未核验竞品资料写成确定性结论；
- 声称 TrialCompiler 已经在真实临床流程中验证。

## E. 最小交付格式

队员可直接填下面这种表：

```text
资料名称：
官方链接：
是否可打开：
资料类型：
相关模块：
能支持的判断：
不能过度推断的地方：
是否建议进入 PPT：
备注：
```

如果时间紧，只要完成“能支持的判断”和“不能过度推断的地方”两项即可。
