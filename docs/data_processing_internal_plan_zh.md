# TrialCompiler 资料处理与数据构建内部方案

- 版本：v0.1
- 日期：2026-07-17
- 负责人：AI 成员
- 输入：医学与商业成员提交的公开资料链接和原始文件
- 输出：可检索知识、结构化事实、依赖图、合成案例与 TrialDocBench

## 1. 分工原则

医学与商业成员只负责寻找公开资料并说明用途。以下工作全部由 AI 成员承担：

- 下载和归档公开文件；
- 文件去重、哈希、命名和目录整理；
- PDF、Word、网页和表格解析；
- 扫描 PDF 的 OCR；
- 元数据清洗和来源核验；
- 术语、缩写、单位和章节名称规范化；
- 区分项目事实、规则约束和专家经验；
- 生成 Trial Fact Sheet 候选字段；
- 构建事实与文档单元的候选依赖关系；
- 构造完全合成的测试案例；
- 生成金标准初稿和医学复核表；
- 导入知识库、运行检索和 Agent 测试；
- 记录数据版本、处理脚本和评测结果。

## 2. 原始资料接收

接收后先建立不可修改的原始资料层：

```text
references/inbox/
  medical/
  business/
```

每个文件计算内容哈希，并保留原始文件名、提交人、提交时间和来源链接。原始层不直接供 Agent 检索。

## 3. 来源登记与去重

将共享表格和原始文件统一整理为来源索引，补齐：

- 稳定 `source_id`；
- 正式标题；
- 作者或发布机构；
- 年份和版本；
- DOI 或官方 URL；
- 资料类型；
- 适用地区、文档类型和主题；
- 获取状态和许可边界；
- 内容哈希；
- 重复或替代关系；
- 核验状态。

相同 DOI、官方 URL 或内容哈希的资料合并为一条来源记录。旧版本不直接删除，而是建立 `superseded_by` 关系。

## 4. 文本提取与清洗

按文件类型处理：

| 输入 | 处理方式 |
| --- | --- |
| 可搜索 PDF | 提取正文、页码和标题层级 |
| 扫描 PDF | OCR 后与页面图像保留映射 |
| Word | 提取标题、段落、表格和批注结构 |
| 网页 | 保存正文、发布日期、更新时间和原始链接 |
| Excel/表格 | 保留工作表、表头、单元格和注释 |

清洗时不改写原意，只处理页眉页脚、乱码、重复换行、断词和明显版面噪声。每个文本块必须能够返回原始文件和具体页码、章节或单元格。

## 5. 知识分层

清洗后的内容按以下三层处理：

### 5.1 项目事实定义

从公开资料中提取“某类临床试验事实应如何表示”的候选字段，例如终点、评估时间、样本量、研究人群、入排标准、给药和访视。

公开资料只能帮助定义事实类型，不能产生真实项目事实。当前所有项目值均来自合成案例。

### 5.2 规则约束

将法规、指南、模板和公开规范处理为带来源、版本、适用范围和定位信息的规则候选。

只有经过来源核验并由医学成员确认理解无误的规则，才能进入可引用状态。

### 5.3 经验与案例

当前没有真实专家审核经验。公开论文和报告中的一般做法只能作为参考资料，不能直接标记为企业批准经验。

Experience Compiler 的真实经验写回功能暂时只使用合成演示数据，直到获得授权的人工审核轨迹。

## 6. Trial Fact Sheet 字段构建

AI 成员根据医学资料生成候选字段字典，包括：

- 字段名称；
- 数据类型和单位；
- 常见中文和英文表达；
- 可能来源位置；
- 需要的专业确认角色；
- 是否属于高风险事实；
- 与其他事实的候选关系。

医学成员只需在普通复核表中选择：

- 正确；
- 需要修改；
- 不确定；
- 不属于当前范围。

不要求医学成员直接编辑 JSON。

## 7. 文档依赖图构建

AI 成员从公开模板和指南中生成候选依赖边，例如：

```text
主要终点评估时间
  -> Protocol Synopsis
  -> Objectives and Endpoints
  -> Schedule of Activities
  -> Statistical Analysis Plan
```

每条候选边保留来源依据和置信度。医学成员通过普通表格复核是否合理；存在统计或注册专业边界时标记 `requires_expert_review`，不强制确认。

## 8. 合成案例构建

当前没有真实案例，因此所有 benchmark 输入都由 AI 成员构造，并满足：

1. 不复制或影射真实临床项目；
2. 不包含患者或企业信息；
3. 事实关系有公开规则依据；
4. 同时包含正确样本、缺陷样本和 hard negatives；
5. 显式标记 `synthetic: true`；
6. 可以追溯到用于构造该案例的来源；
7. 医学成员只复核医学和文档逻辑，不处理数据格式。

首轮目标：

- 2 个完全合成的方案小型项目；
- 40 个 Trial Fact 类型；
- 80 条候选依赖边；
- 30 个一致性缺陷；
- 至少 8 个 hard negatives；
- 15 个事实变更影响案例；
- 10 个 Agent 修订的人类复核标签。

## 9. 商业资料处理

AI 成员将商业成员提交的公开资料整理为：

- 公开工作流证据表；
- 角色与交接关系图；
- 公开效率和成本证据表；
- 竞品能力矩阵；
- 数据安全、部署和采购约束清单；
- 已证实事实、厂商主张和团队假设的分层记录。

公开资料没有给出可靠数字时保留 `unknown`。ROI 只能作为公开信息支持的情景模型，不能写成真实客户收益。

## 10. 人工复核方式

为了不要求队员使用代码，AI 成员将所有待确认内容导出为普通 Excel 或飞书表格。

医学复核字段只包括：

- 内容是否理解正确；
- 来源是否适用；
- 是否需要统计、注册或其他专家确认；
- 修改意见。

商业复核字段只包括：

- 来源是否为官方或可信公开来源；
- 产品能力是否被官方页面明确说明；
- 数字口径是否完整；
- 该结论是事实还是团队假设。

复核结果由 AI 成员再转换为系统需要的 JSON、知识条目和 benchmark 标签。

## 11. 质量检查

每批处理后检查：

- 来源 URL 可访问性；
- DOI、标题、年份和发布机构一致性；
- PDF 是否能够正常打开；
- OCR 结果是否保留页面定位；
- 是否误收真实项目或个人信息；
- 事实、规则和经验是否混层；
- 合成案例是否明确标记；
- 金标准是否有人类复核状态；
- 训练、验证和测试数据是否避免泄漏；
- 数据处理脚本和版本是否可复现。

## 12. 最终责任边界

两位队员交付原始公开资料，AI 成员负责把资料变成机器可用的数据。医学成员负责纠正医学和文档逻辑理解，商业成员负责纠正来源口径和产品能力理解。

任何结构化结果、合成案例、评测标签和模型输出都不能因为经过数据清洗就自动变成临床、统计、注册或商业事实。
