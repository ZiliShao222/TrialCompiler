# NCT04683926 测试案例包复核报告

## 结论

`NCT04683926 / OMNI-PAIN-103` 适合作为 TrialCompiler 第一套公开研究文档案例。
Protocol、SAP、ICF 和 ClinicalTrials.gov 注册记录均可从官方地址获得，且材料中存在事实抽取、
跨文档口径、有效映射、状态区分和变更传播等多种任务。

第二个 `Week 12 -> Week 16` ZIP 是仓库现有合成案例的运行封装，不应作为独立外部案例计数。

## 已确认的高价值测试点

1. **饮水指令冲突**：Protocol Synopsis 与 SAP 表述为给药前后 1 小时不得饮水，Protocol
   正文多处却表述为该时段只允许饮水。这是会影响现场执行的高风险指令冲突，系统应阻断
   自动确认并提交医学、运营和质量角色判断。
2. **给药间隔边界**：Protocol 同时出现 `>3 days` 和 `3 days`，给药日为 Day 1、4、7、10。
   系统应识别边界差异，但不能自行改值。
3. **双时间轴映射**：Protocol 使用 Day 1/4/7/10；SAP 明确说明 CRF 将四个处理期分别记为
   Period 1-4 的 Day 1。这是合法映射，不应被误报为冲突。
4. **计划、目标和实际人数**：32、24、24 分别是最大计划随机人数、目标 PK 完成人数和实际
   入组人数，必须保留语义状态。
5. **合成变更传播**：最终 PK 采样由 32 h 改为 36 h 的变更是人工合成测试资产，可测试
   Protocol、SoA、SAP、ICF、CRF 和结果登记的影响召回。

## 需要修正的原始标注

1. `11 days` 与 `about 12 days` 可能涉及 Day -1 到 Day 11 的包含式计数以及“on study”与
   “confinement”的定义差异，应标为需人工裁定的口径差异，而不是无条件硬冲突。
2. `<=30 days` 与 `up to 31 days` 可能来自包含式计数和受试者语言转换，应保留证据并交由
   运营/伦理判断。
3. SAP 明确声明其只处理 safety objective，因此 Protocol 的 evaluable population 与 SAP 的
   Safety/PP 表述不能脱离文档范围直接判成定义冲突。
4. 人工解压后的原始 Excel 中文内容正常，可以作为提交材料和人工审阅界面保留。仓库另行
   建立 UTF-8 TSV 与 reviewed JSON，是为了便于版本控制、程序化评测和记录复核后的标签，
   不是因为原始工作簿损坏。

## 当前系统能力差距

目前 CLI 的确定性规则主要处理“已确认 Week 事实与章节文本不一致”。它尚不能完整处理：

- 否定词造成的操作指令反转；
- `>`、`>=` 与精确日程之间的边界推理；
- 文档范围和分析人群定义；
- 跨文档时间轴映射；
- 通用事实变更的图传播。

因此，这一案例现在是下一阶段 benchmark，而不是已经通过的宣传结果。后续评测必须保存
模型预测、gold 对齐、误报/漏报和人工裁定记录。
