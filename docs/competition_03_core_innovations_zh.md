# 3. 核心创新点：技术、模式与流程创新

> 医学资料整合提示：TrialCompiler 的创新应表述为“临床文档编译器”而不只是“临床写作助手”。它把 Trial Fact Sheet、Clinical Document Graph、Document Unit Tests 和 Change Impact Analysis 组合起来，解决关键设计事实跨章节、跨表格、跨关联文件和跨执行系统传播时的遗漏与冲突。

> 竞品分析整合提示：跨文档一致性、协议结构化、修订影响分析和来源追踪在 HumanTrue、Cori Clinical 等直接竞品中已有公开迹象，因此 TrialCompiler 不能把“发现受影响位置”单独宣称为独有壁垒。更稳妥的创新定位是：面向中国药企的临床方案事实与变更治理层，以已确认事实层、变更任务闭环、规则与经验资产、中国药企适配、开放接口和责任审计形成差异化。

- 项目名称：TrialCompiler
- 参赛场景：AI 先锋未来人才大赛，健康元临床开发效率方向
- 当前版本：v1.0
- 日期：2026-07-17
- 当前验证边界：公开资料桌面研究与完全合成数据评测

## A. 简短版

TrialCompiler 的创新不在于使用了大模型、RAG 或多个 Agent，而在于把临床试验方案从“供模型一次性生成的长文本”转化为“由结构化事实驱动、能够编译、测试、追踪和增量更新的文档工程”。系统先由专业人员确认 Trial Fact Sheet，再通过文档依赖图定位事实在不同章节、表格和关联文件中的传播关系，最后执行确定性检查、语义检查、最小修订和人工质量门。这样，一项设计事实变化后，系统不需要重写全部文件，而能找到所有受影响位置、检查旧值残留并生成可审核的红线稿和影响矩阵。

技术上，TrialCompiler 提出带临床治理元数据的 Clinical Semantic Element，将项目事实、规则约束和专家经验分开管理；采用“精确匹配/混合召回 + 适用范围硬门控 + 语义等价精判”的粗到细检索，避免仅因语义相似就误用过期、越权或不适用知识；采用依赖图引导的增量编译，只重新生成和测试受变更影响的内容；采用 Human-governed Experience Compiler，把人工审核轨迹压缩为具有条件、证据、反例、审批状态和有效期的 Decision Capsule，而不是把原始对话直接写入长期记忆。

流程上，系统使用职责分离的 Agent 完成上下文确认、证据检索、候选生成、独立评审、结果汇总和经验提炼，并设置有限返工、结构化失败和人工升级机制。TrialDocBench 与 Trace Replay 则把事实抽取、缺陷检测、影响分析、修订正确性和经验复用转化为可复现实验，可与普通大模型、长上下文、普通 RAG 和单 Agent 进行基线对比。

模式上，TrialCompiler 采用“AI 提候选、专家作决定、系统留证据”的人机协同方式。飞书 Aily 负责交互、补问、协作和审批，TrialCompiler 负责事实治理、文档编译和质量验证。系统不替代医学、统计、注册和质量人员，而是减少重复搬运、全文检索、交叉核对和修改遗漏，使专业人员把时间集中在真正需要判断的工作上。

> **普通 AI 工具在生成文字；TrialCompiler 在维护一套能够被验证、被安全修改并被持续审计的临床设计事实与文档系统。**

## B. 详细版

## 1. 创新命题

临床试验方案编制的难点不是“缺少一段写得通顺的文字”，而是同一项关键设计事实需要在多个章节、表格和关联文件中反复出现，并在多专业团队、多轮修订和多版本流转中保持一致。

普通大模型擅长生成自然语言，但不能天然保证：

- 文本中的数字和时间点来自已确认事实；
- 同一事实在多处表达完全一致；
- 使用的规则版本、地区和适用范围正确；
- 修改一项事实后所有受影响位置都被找到；
- 历史经验没有越权、过期或错误泛化；
- 每个结论能够回到来源、版本和责任人；
- 输出经过独立测试和专业审核。

因此，TrialCompiler 的核心创新命题是：

```text
临床文档生成
    ↓
由已确认事实驱动的临床文档编译
    + 文档依赖分析
    + 自动化单元测试
    + 受控增量更新
    + 人工质量门
    + 可治理经验复用
```

它不是把更多 Agent 放在一个聊天框后面，而是重新定义临床文档如何被表示、生成、测试、修改和积累。

## 2. 与常规方案的区别

| 常规方案 | 主要能力 | 关键局限 | TrialCompiler 的改进 |
| --- | --- | --- | --- |
| Word 人工编制与交叉审核 | 专业人员直接撰写和批注 | 重复搬运、全文检索慢、旧值和漏改难发现 | 结构化事实、自动定位、红线与影响矩阵 |
| 通用大模型聊天 | 快速生成单段或单章文本 | 事实来源、版本和跨章节一致性不可控 | 只使用已确认事实，输出带来源和审核状态 |
| 长上下文一次性生成 | 读取更多资料后生成长文 | 长度不等于正确，难追踪哪项事实影响哪段文本 | 事实层与依赖图显式建模 |
| 普通 RAG | 检索语义相关文本 | 相似内容可能来自错误地区、旧版本或未经批准经验 | 元数据硬门控与语义等价精判 |
| 单 Agent 写作或审核 | 一个模型承担多个角色 | 容易自我确认，失败原因不结构化 | 生成、评审、报告和经验提炼职责分离 |
| 通用多 Agent | 多角色协作 | 常停留在对话编排，缺少事实状态、依赖和质量门 | Active Compilation 状态机与有限返工 |
| 文档管理系统 | 文件存储、版本和权限 | 通常不理解事实级依赖和语义冲突 | 事实级版本、依赖传播与文档单元测试 |
| 传统规则检查 | 对固定格式和数字做校验 | 难发现跨章节语义冲突 | 确定性检查与语义检查组合 |

## 3. 技术创新一：Clinical Semantic Element

### 3.1 从文本块升级为受治理的语义单元

普通 RAG 把知识存成文本块和向量。TrialCompiler 将每项可复用知识表示为 Clinical Semantic Element，它不仅包含语义键和值，还包含决定其能否被使用的治理元数据：

```text
element_type        project_fact / rule_constraint / decision_capsule
document_type       适用文档类型
section_type        适用章节
jurisdiction        适用地区
therapeutic_area    治疗领域
trial_phase         试验阶段
authority           来源权威类型
source_ids          原始证据
version             来源版本
valid_from/until    生效和失效时间
approval_status     审批状态
access_scope        权限范围
supersedes          替代关系
```

### 3.2 三类知识不混用

1. `project_fact`：本项研究采用什么设计；
2. `rule_constraint`：文档需要满足什么规则；
3. `decision_capsule`：在特定条件下经过批准的审核经验。

例如，“主要终点评估时间为 Week 16”是项目事实；“主要终点在摘要、正文和统计描述中应一致”是规则；“终点时间变化后应检查访视表和统计章节”是程序性经验。三者可以相互引用，但不能因为文字相似而互相替代。

### 3.3 创新价值

这种表示让知识能够被验证、撤销、替代、过期、按权限隔离和按适用范围调用，使知识库从“语义相似文本仓库”变为“可治理临床语义层”。

## 4. 技术创新二：Governance-aware Coarse-to-Fine Retrieval

### 4.1 粗到细检索链路

```text
规范化语义键精确匹配
    ↓ 未命中
BM25 / 向量 ANN 高召回粗筛
    ↓
地区、治疗领域、阶段、文档类型、版本、审批和权限硬门控
    ↓
轻量语义等价精判
    ↓
verified hit / rejected candidate
```

高向量相似度只表示“可能相关”，不能直接表示“可以复用”。精判器回答的不是两个文本是否谈论同一主题，而是旧事实、规则或经验是否能在当前任务中直接、安全地使用。

### 4.2 知识准入函数

```text
Admit(m, q)
= SemanticEquivalent(m, q)
  AND ScopeMatch(m, q)
  AND VersionValid(m)
  AND HumanApproved(m)
  AND AccessAllowed(m, q)
```

任一硬条件不满足，候选知识都不能进入 Agent 上下文。系统同时保存被拒绝候选及原因，便于审计和优化。

### 4.3 与普通 RAG 的本质区别

普通 RAG 优化“能否找到相关内容”；TrialCompiler 进一步优化“相关内容是否有权、有效且适用于当前临床文档任务”。这对存在版本、地区、阶段和角色边界的临床场景尤为关键。

## 5. 技术创新三：Dependency-guided Incremental Compilation

### 5.1 文档依赖图

TrialCompiler 建立 Clinical Document Graph，将 Fact、Rule、Section、Table、Document、Issue、Revision 和 Human Decision 连接起来，使事实与文档表达之间的关系显式可计算。

### 5.2 影响集合

当事实 `f` 发生变化 `delta f` 时，系统先沿图计算候选影响范围：

```text
CandidateImpact(delta f) = Reachable(DocumentGraph, f)
```

随后结合边类型、规则条件、版本和语义复核得到最终影响集合：

```text
ImpactSet(delta f)
= Filter(CandidateImpact, scope, rule, version, semantic relevance)
```

系统只重新生成和测试该集合中的文档单元，未受影响内容保持稳定。

### 5.3 增量编译输出

- 直接与间接受影响位置；
- 旧值及其语义变体残留；
- 候选最小修订；
- 修改前后红线；
- 需要重新运行的单元测试；
- 需要参与审核的专业角色；
- 变更影响矩阵和审计轨迹。

### 5.4 创新价值

普通生成工具倾向于重新写整段或整份文件，容易引入无关变化。依赖图引导的增量编译把修订范围控制在必要位置，兼顾漏改率、过度传播率和人工复核成本。

## 6. 技术创新四：Human-governed Experience Compiler

### 6.1 为什么不能直接保存历史对话

原始对话、Word 批注和 Agent 轨迹包含偶然表达、错误尝试、项目特有信息和未确认判断。直接写入长期记忆会导致错误经验反复传播，也会增加上下文成本。

### 6.2 Decision Capsule

Experience Compiler 将已完成人工审核的轨迹压缩为：

```text
Trigger             触发条件
Conditions          适用条件
Recommended Action  推荐动作
Rationale           理由
Evidence            证据
Counterexample      反例与不适用条件
Approval Status     审批状态
Valid Until         有效期
```

系统不会记忆“把 Week 12 改成 Week 16”这个项目答案，而是提炼程序性经验：“当已批准的核心时间事实变化时，应遍历依赖图、定位受影响章节、生成带来源的最小修改并重跑一致性测试”。

### 6.3 人工治理流程

```text
完整审核轨迹
    ↓
F Agent 提取候选经验
    ↓
去重、补充适用条件和反例
    ↓
授权专家批准、修改或拒绝
    ↓
生成短 Action Card
    ↓
后续任务按条件检索和引用
```

未经批准、已过期、已被替代或权限不匹配的经验不能指导真实任务。该机制既积累组织经验，又避免把单个项目意见错误泛化成通用规则。

## 7. 技术创新五：成本感知的知识与记忆生命周期

知识库不能无限增长。TrialCompiler 根据使用频率、重新获取成本、访问延迟、稳定性、权威性、年龄和存储大小管理 Clinical Semantic Element 的生命周期：

```text
U(m)
= a * log(1 + verified_frequency)
  + b * refetch_cost
  + c * latency
  + d * staticity
  + e * authority
  - f * age
  - g * size
```

只有通过完整准入判断的真实命中才增加使用频率；ANN 粗筛候选不增加热度。监管规则、企业批准规范和专家批准经验受到额外保护，临时模型输出、低频且可低成本重建的中间结果优先淘汰。

这使记忆管理从简单的“保存所有历史”转变为具有价值、权威、成本和失效意识的知识生命周期管理。

## 8. 技术创新六：Evaluator-guided Active Compilation

### 8.1 从被动生成到主动编译

普通模型收到输入后直接输出答案。TrialCompiler 的执行过程是：

```text
A 确认上下文和边界
    ↓
B 检索事实、规则和证据
    ↓
C 生成或修复候选文本
    ↓
D 独立执行质量检查
    ├─ 通过 → E 汇总交付
    ├─ 可修复 → 带结构化问题返回 C
    └─ 证据不足/高风险 → 人工升级
    ↓
F 从人工审核结果提炼候选经验
```

### 8.2 有限返工而非无限自我反思

C-D 循环设置最大轮次和明确退出条件。D Agent 必须返回问题类型、严重度、证据位置、受影响事实、失败测试和修复要求，而不是只给“还不够好”的模糊评价。

### 8.3 混合评价器

质量判断由三类评价器组合：

1. 可验证规则：数字、时间点、编号、旧值残留；
2. 语义评价器：定义冲突、范围错配、含义漂移；
3. 人工评价器：医学合理性、统计科学性、注册策略和正式批准。

这种设计将“生成、测试、修复、再测试”引入临床文档编制，但不让模型替代专业审批。

## 9. 技术创新七：TrialDocBench 与 Trace Replay Harness

### 9.1 为什么评测本身也是核心能力

临床文档系统不能只凭几个展示案例证明有效。TrialDocBench 将系统能力拆成可独立测量的任务：

| 层级 | 任务 | 示例指标 |
| --- | --- | --- |
| L1 | 事实抽取和来源定位 | Fact F1、数值准确率、证据定位准确率 |
| L2 | 知识准入 | 不适用知识拒绝率、过期规则误用率 |
| L3 | 缺陷检测 | Precision、Recall、Macro F1、严重缺陷召回率 |
| L4 | 影响分析 | 影响节点召回率、过度传播率、漏改率 |
| L5 | 最小修订 | 事实保持率、缺陷修复率、无关改动率、人工接受率 |
| L6 | 经验复用 | 正确复用率、错误经验引入率、上下文成本 |
| L7 | 审计与安全 | 来源覆盖率、轨迹完整率、越权知识拦截率 |

### 9.2 Trace Replay

系统保存任务输入、知识候选、拒绝原因、Agent 状态、工具调用、规则结果、人工反馈和最终输出。相同案例能够在不同模型、检索策略、规则版本或 Agent 架构下重复运行，形成可复现对比。

### 9.3 基线与消融

计划比较：

- 普通大模型直接生成；
- 长上下文提示；
- 普通 RAG；
- 单 Agent；
- 多 Agent 但无事实层；
- 无元数据门控；
- 无依赖图；
- 无独立评价器；
- 无经验治理；
- 完整 TrialCompiler。

消融实验可以回答每个模块到底带来了多少正确性、稳定性和审核效率提升，避免把效果笼统归因于“大模型更强”。

## 10. 模式创新：事实、规则和经验三层治理

传统知识库常把项目内容、法规要求和历史意见放在一起检索。TrialCompiler 将三者分层：

- 项目事实由当前项目授权人员确认和版本化；
- 规则约束由权威来源、适用范围和有效期治理；
- 专家经验由具体审核轨迹提炼，并经人工批准后按条件复用。

这形成一种新的临床 AI 知识治理模式：模型可以帮助提取和匹配，但不能自行改变知识的权威状态。

## 11. 模式创新：AI 提候选、专家作决定、系统留证据

TrialCompiler 不追求无人化临床方案生成，而采用明确的人机责任分工：

| AI 负责 | 专业人员负责 |
| --- | --- |
| 候选事实提取 | 研究设计事实确认 |
| 来源定位与依赖分析 | 医学、统计和注册合理性判断 |
| 重复文本辅助编制 | 风险处置和策略决定 |
| 冲突、遗漏和旧值提示 | 候选修订接受、修改或拒绝 |
| 红线稿、影响矩阵和证据表 | 正式文档批准 |
| 候选经验提炼 | 经验适用范围与有效性批准 |

这种模式兼顾自动化效率、专业责任和结果可审计性，比“一键生成并由人兜底”更适合高要求文档场景。

## 12. 流程创新：从全文重审到变更驱动审核

常规流程中，一项事实变化后需要人工重新搜索和通读多份文件。TrialCompiler 将流程改为：

```text
事实变更提议
→ 人工确认候选新版本
→ 依赖图计算影响范围
→ 生成最小修订
→ 自动重跑相关测试
→ 输出红线和影响矩阵
→ 指定专业角色审核
→ 审批后生效并保留历史
```

审核从“重新阅读所有内容”转变为“围绕明确变更、证据和影响范围进行定向复核”，有望降低重复劳动和同步遗漏。

## 13. 流程创新：审核经验成为受控资产

普通流程中的经验散落在邮件、会议纪要和批注中。TrialCompiler 把已经发生的人工决定转化为可审核、可撤销、可过期、带反例的 Decision Capsule，并通过飞书工作流完成确认。后续项目只在条件匹配时调用该经验，使审核结果形成组织能力，而不是停留在个人记忆中。

## 14. 飞书协同创新

飞书 Aily 不只是把网页聊天框搬进企业协作软件，而是承担：

- 用户、项目、文档和任务识别；
- 缺失信息补问；
- 事实确认卡片；
- 医学、统计、注册和质量角色分派；
- 审核状态提醒；
- 红线稿和影响矩阵返回；
- 人工反馈采集；
- 审批轨迹与后台编译任务连接。

TrialCompiler 核心负责事实、图、测试和编译；飞书负责人与流程。两者结合形成面向真实协作场景的产品闭环，而不是一个孤立的演示模型。

## 15. 可检验技术假设

项目的创新需要通过实验被证伪或支持。当前提出以下假设：

1. Clinical Semantic Element 相比普通文本块 RAG，可降低过期、越权和范围错误知识的准入率；
2. 元数据硬门控加语义等价精判，相比仅向量相似度检索，可提高知识使用精度；
3. 文档依赖图相比全文搜索，可提高受影响位置召回率并降低过度修改率；
4. 生成与评审职责分离，相比单 Agent，可提高严重缺陷召回率和修订一次接受率；
5. Decision Capsule 相比原始历史轨迹，可在减少上下文成本的同时降低错误经验引入率；
6. 增量编译相比整篇重写，可减少无关文本变化并缩短人工复核范围；
7. Trace Replay 能够提高不同模型和不同版本结果的可复现性。

## 16. 当前验证策略与诚实边界

当前阶段无法采集真实临床项目、企业内部文档或患者数据，因此验证采用：

1. 网上公开法规、指南、模板、论文和公开产品资料；
2. 完全合成且不对应真实患者或真实项目的协议片段；
3. 程序化和人工设计的跨章节矛盾、旧值残留、范围错配等缺陷；
4. 多模型基线、模块消融和轨迹回放；
5. 由团队成员完成技术性数据清理、结构化和评测实现。

当前能够证明的是原型在合成任务中的事实一致性、缺陷检测、影响分析、修订和审计能力。当前不能声称：已经接入健康元内部流程、已经在真实临床项目中节省具体工时、已经满足正式生产和合规要求，或能够替代医学、统计、注册和质量人员。

## 17. 原创性边界

为了使比赛陈述准确可信，TrialCompiler 明确区分基础技术、外部启发与项目贡献：

- 大语言模型、向量检索、BM25、ANN、RAG、多 Agent 和飞书 Aily 是实现手段，不单独作为原创点；
- Semantic Element、粗到细召回、主动推理与 Agent 评测框架受到现有研究和公开技术分享启发；
- TrialCompiler 的贡献在于将这些方法重新组合并扩展到临床文档工程：加入项目事实/规则/经验类型隔离、版本和权限门控、文档依赖图、增量编译、文档单元测试、人工状态门、受控经验编译和分层评测；
- 项目不宣称发明了临床方案写作、RAG、多 Agent 或文档版本管理，而强调一套针对临床设计事实跨文档一致性问题的系统化技术路线。

清楚说明继承关系不会削弱创新，反而能让评委准确看到我们真正解决了什么新的组合问题。

## 18. 最适合比赛展示的创新主线

比赛展示不应从 Agent 数量或模型参数开始，而应使用一个可观察的事实变更：

1. 导入完全合成的方案正文和访视表；
2. 系统提取并由人确认 Trial Fact Sheet；
3. 展示主要终点评估时间在多个位置被引用；
4. 提议将 Week 12 改为 Week 16；
5. 文档依赖图计算直接和间接影响位置；
6. 系统发现旧值、跨表冲突和统计章节风险；
7. C-D Agent 闭环生成并验证最小修订；
8. E Agent 输出红线稿、影响矩阵和证据表；
9. 人工接受、修改或拒绝候选修订；
10. F Agent 提炼待批准 Decision Capsule；
11. Trace Replay 展示每一步依据；
12. TrialDocBench 对比普通大模型、普通 RAG 和完整系统。

这条主线可以在一次演示中同时呈现产品价值、技术创新、人机协作、安全边界和量化评测。

## 19. 一句话总结

> TrialCompiler 的核心创新，是把受治理的临床事实、文档依赖图、粗到细知识准入、评价器驱动的增量编译、人工批准的经验记忆和可回放评测整合为一套临床文档工程系统，使 AI 从“生成一段看起来合理的文字”升级为“在专业人员控制下维护一套可验证、可追踪、可安全更新的文档体系”。
