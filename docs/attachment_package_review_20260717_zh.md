## 3. 总体架构与受控数据流附件

### 3.1 最有价值的新增内容

该附件把 TrialCompiler 从“AI 文档助手”明确提升为“受控事实驱动的临床文档工程系统”，并补充了四项清晰原则：

1. 来源优先：事实、规则和修订建议能够回到文件及具体位置；
2. 确认优先：候选事实未经授权人员确认不得驱动正式编制；
3. 增量优先：围绕事实变化计算影响范围，不无差别重写全文；
4. 可撤销优先：保留历史版本、适用范围和替代关系。

附件进一步提出“一条受控数据主线、一个飞书入口、三条横向控制线”：

```text
企业受控资料与规则
→ 文档解析与候选事实
→ Trial Fact Sheet
→ 事实—章节—表格—文件依赖图
→ 编译与文档单元测试
→ 受控候选交付物
→ 人工质量门
→ 企业 DMS / 正式质量体系
```

三条横向控制线分别为数据安全、质量控制和责任控制。这种表达比单纯列模块更适合比赛答辩，因为它同时说明了数据怎么走、错误在哪里被拦截、最终责任由谁承担。

### 3.2 与当前仓库的关系

附件与现有 Trial Fact Sheet、Clinical Document Graph、Document Unit Tests、Experience Compiler 和 TrialDocBench 方向一致，不需要推翻现有设计。它主要补强了以下内容：

- 企业 DMS 是正式签核、发布、归档和版本治理的制度性终点；
- 飞书 Aily 是协同入口和任务编排层，不承担最终专业判断；
- 候选事实、已确认事实与正式有效版本必须分开；
- 生成与独立审核应分开执行，避免系统对自己的输出自证；
- 当前比赛原型与生产级身份、租户、签名和长期审计能力必须明确区分。

### 3.3 需要继续落地的地方

- 建立“目标架构模块—代码目录—演示页面—测试证据”的实现对照矩阵；
- 将影响等级固定为 `must_update`、`likely_update`、`expert_judgment`、`history_only`；
- 为候选、待确认、已确认、有效、已替代和冲突状态建立统一状态机；
- 在合成案例中展示 DMS 前的人工批准边界，而不是只输出自动修订文本。

## 4. 法规标准与功能映射附件

### 4.1 最有价值的新增内容

该附件采用：

```text
规范要求 → 功能控制 → 输出证据 → 责任边界
```

这条链路能证明 TrialCompiler 的功能不是从通用大模型能力随意拼接，而是从临床研究质量、协议结构化、数据治理、变更控制和专业责任要求反向推导。附件把 ICH E6(R3)、E8(R1)、M11、CDISC USDM/DDF、中国 GCP、个人信息保护、E9/E9(R1) 和企业 SOP 映射到了具体产品控制。

### 4.2 已完成的官方核验

- ICH M11 最终 Step 4 文件日期为 2025-11-19；
- CDISC USDM v4.0 的公开发布日期为 2025-06-03；
- 中国 2026 修订版《药物临床试验质量管理规范》自 2026-09-01 起施行，2020 版在此之前继续适用；
- ICH E6(R3)、E8(R1)、E9 和 E9(R1) 在本地来源索引中均已有正式来源与快照。

### 4.3 发现的缺口

1. 附件正文末尾使用“官方来源”文字，但没有形成可点击链接、编号和正文回链；
2. 验收标准要求覆盖 SPIRIT，但原映射表未单列 SPIRIT；结构化映射表已补充该行；
3. 个人信息保护法尚未进入当前公开来源索引，需补全国人大或中国政府网权威来源；
4. 企业 SOP 只能作为未来真实试点的内部受控输入，当前不能用公开材料假设其具体要求；
5. 映射只能说明产品设计对齐，不能宣称法律合规、监管认可、CSV 完成或生产级验证。

## 6. 对比赛叙事的直接贡献

三份附件组合后，可以形成一条更完整的答辩叙事：

> 临床试验方案不是普通长文，而是由重复出现、彼此依赖、需要跨角色共同维护的设计事实构成。TrialCompiler 先把授权资料中的候选事实交给专业人员确认，再以受控事实驱动章节、表格和关联文件；依赖图使变更影响可计算，文档单元测试使一致性可验证，人工质量门和 DMS 保留专业责任与正式发布边界。法规与标准被转译成具体控制和验证证据，因此系统不仅会写，也能说明为什么这样写、哪里受影响、谁确认过、是否达到发布条件。
