# 1. 方案整体概述

> 医学资料整合提示：本方案的对象不是单一“方案正文”，而是临床试验方案、SoA、ICF、SAP、CRF/eCRF、IWRS/IRT、药物管理手册、实验室手册、影像手册和注册/申报材料共同构成的临床文档与执行文件体系。TrialCompiler 的核心任务是管理关键设计事实在这些文件和系统之间的一致传播。

- 项目名称：TrialCompiler
- 参赛场景：AI 先锋未来人才大赛，健康元临床开发效率方向
- 当前版本：v1.0
- 日期：2026-07-17
- 当前验证边界：公开资料案头调研与完全合成数据评测


## A. 简短版

临床试验方案及其关联文件编制工作量大、审核周期长。主要终点、评估时间、样本量、入排标准、给药方案、访视安排和统计分析原则等关键设计事实，会重复出现在方案摘要、正文、表格以及统计分析计划等关联文件中。一项事实发生变化后，工作人员往往需要人工全文检索、逐处修改并开展多轮交叉审核，容易出现旧值残留、正文与表格冲突、章节定义不一致、版本混乱和关联文件漏改等问题。与此同时，专家审核经验长期散落在 Word 批注、电子邮件、会议纪要和历史文件中，难以在后续项目中受控复用。

TrialCompiler 将临床试验方案及其关联文件转化为一种**可编译、可测试、可追踪、可增量更新的文档工程体系**。系统先从授权研究资料中提取候选事实，由医学、统计和注册等专业人员确认后形成带来源、版本和审核状态的 Trial Fact Sheet；随后建立关键事实与章节、表格及关联文件之间的依赖图，基于已确认事实辅助分章节编制，并通过确定性检查和语义一致性检查发现冲突、遗漏和旧版本残留。当事实发生变化时，系统沿依赖图计算影响范围，生成影响矩阵、候选修订文本、修改前后红线稿、证据表和审核报告，最后由授权专业人员审核批准。

TrialCompiler 不是替代专家做临床、统计或注册决策，也不是让大模型一次生成数百页文档。它把 AI 放在候选信息提取、依赖分析、重复文本编制、风险识别和修改定位等适合自动化的位置，把研究设计判断、事实确认、风险处置和正式批准保留给专业人员。飞书 Aily 作为交互和任务编排入口，连接资料上传、缺失信息补问、专业确认、Agent 工作流和审核协作。当前阶段只使用网上公开资料和完全合成案例进行验证，不采集真实临床项目、患者数据、企业内部文件或真实访谈数据。

> **TrialCompiler 让临床试验方案像软件代码一样具备编译、测试、追踪和安全更新能力，使专业人员将更多时间投入真正需要医学、统计和注册判断的工作。**

## B. 详细版

## 1. 比赛问题与方案选择

本项目面向 AI 先锋未来人才大赛中健康元提出的临床开发效率问题，选择其中最适合通过 AI 系统形成完整闭环、并能够被量化验证的一项具体任务：

> **临床试验方案及其关联文件编制工作量大、审核周期长，关键设计事实在多章节、多表格和多文件之间重复出现，容易产生跨章节不一致、版本混乱和修改遗漏。**

我们将问题范围聚焦于临床试验方案设计阶段的：

1. 文档编制；
2. 一致性审核；
3. 变更影响分析；
4. 修改建议与红线稿生成；
5. 审核经验的受控沉淀与复用。

当前方案明确不处理：

- 患者病历筛查；
- 入排标准自动匹配患者；
- 临床数据清洗；
- 安全信号自动判定；
- 诊断、治疗或受试者入组决策；
- 未经专业人员批准的正式临床文件发布。

这种聚焦使项目能够围绕一个清晰、真实且高频的临床开发痛点建立产品、技术和评价闭环，而不是把多个临床环节堆叠成缺乏深度的通用 Agent 平台。

## 2. 为什么临床方案编制不是普通写作问题

临床试验方案是研究设计、执行和质量管理的核心文件，并与方案摘要、访视流程表、知情同意书和统计分析计划等关联文件保持紧密关系。同一项研究设计事实可能在多处以不同语言反复出现，例如：

- 主要终点在方案摘要、研究目的、终点表和统计分析部分重复出现；
- 终点评估时间同时影响正文、访视流程表和统计描述；
- 目标样本量需要与随机分组、分析集和统计假设保持一致；
- 入排标准需要与目标人群、筛选流程和安全性要求保持一致；
- 给药方案会影响治疗描述、访视、依从性与安全性评估；
- 一次方案修订可能同时影响多个章节和关联文件。

传统全文搜索只能寻找相同字符，难以稳定识别“表达不同但依赖同一事实”的内容。普通大模型虽然可以快速生成流畅文本，但无法天然保证：

- 输入事实已经被专业人员确认；
- 同一事实在所有位置保持一致；
- 引用的是正确版本和正确适用范围的规则；
- 修改一处后所有受影响位置均已更新；
- 未受影响的内容不会被过度改写；
- 每项结论都能追溯到来源和人工决策；
- 专家的一次项目意见不会被错误推广为企业通用规则。

因此，本项目不把任务定义为“让 AI 写得更快”，而是定义为“让临床文档中的事实、规则、依赖、修改和审核状态能够被计算和验证”。

## 3. TrialCompiler 的核心思想

TrialCompiler 借鉴软件工程中的编译、依赖管理、单元测试、增量构建、版本控制和持续集成思想，把传统静态 Word/PDF 文档重新表示为临床文档工程对象。

软件工程与 TrialCompiler 的对应关系如下：

| 软件工程对象 | TrialCompiler 对象 |
| --- | --- |
| 源代码 | 已确认项目事实、模板、规则和证据 |
| 变量 | 主要终点、时间点、样本量、入排标准等 Trial Facts |
| 抽象语法树/依赖图 | Clinical Document Graph |
| 编译器 | 分章节辅助编制与结构化生成流程 |
| 单元测试 | 数字、术语、时间点、引用和语义一致性检查 |
| 增量构建 | 事实变化后的影响传播与局部重新编制 |
| 编译错误 | 矛盾、遗漏、旧值残留和规则缺失 |
| 构建报告 | 红线稿、影响矩阵、证据表和审核报告 |
| 代码审查 | 医学、统计、注册和质量人员审核 |
| 可复用模块 | 经批准、带适用条件的审核经验 |

系统的核心价值不是生成更多文字，而是使文档生成和修改过程具备清晰的输入、依赖、测试、输出、版本和责任边界。

## 4. 项目事实、规则约束与专家经验

TrialCompiler 将三类容易被普通知识库混在一起的信息严格分开。

### 4.1 项目事实层

项目事实回答“本项研究采用什么设计”，例如：

- 本研究的主要终点是什么；
- 主要终点评估时间是 Week 12 还是 Week 16；
- 目标样本量是多少；
- 研究人群和入排标准是什么；
- 给药方案和访视安排是什么；
- 采用什么统计分析原则。

AI 只能从资料中提取候选事实。候选事实经过医学、统计、注册等授权人员确认后，才进入有效状态并用于后续编制与测试。

### 4.2 规则约束层

规则约束回答“相关内容需要满足什么编制、质量和合规要求”，来源包括：

- 公开法规和指南；
- 企业模板和标准操作程序；
- 文档必备内容；
- 编制格式与质量检查规则。

规则具有来源、版本、地区、适用文档、适用章节和有效状态，不能因为与项目事实文字相似而被误当成项目设计决策。

### 4.3 专家经验层

专家经验回答“在什么条件下，审核人员曾如何识别和处理某类问题”。一条经验需要包含：

- 触发条件；
- 适用范围；
- 推荐动作；
- 修改理由；
- 来源证据；
- 不适用反例；
- 审批状态；
- 有效期或替代关系。

未经人工批准的模型输出、历史对话和单项目意见不能直接进入长期经验库。

## 5. 面向人的完整使用流程

TrialCompiler 从人的工作过程出发，而不是要求用户理解 Agent 内部结构。

```text
创建项目
  -> 上传公开或已授权研究资料、模板和参考文件
  -> 系统提取候选事实与缺失信息
  -> 医学、统计、注册等人员确认关键事实
  -> 形成 Trial Fact Sheet
  -> 系统建立文档依赖图
  -> 用户选择需要编制或审核的章节
  -> 系统生成候选章节或审阅现有内容
  -> 运行文档单元测试和语义一致性检查
  -> 输出缺陷、证据、影响范围和候选修订
  -> 专业人员接受、修改、拒绝或升级问题
  -> 生成红线稿、影响矩阵、证据表和审核报告
  -> 经正式流程批准后形成新版本
  -> 将经授权确认的审核经验沉淀为受控规则
```

用户看到的是项目事实确认、文档任务、风险问题、红线稿和审核状态；多 Agent 只是系统内部完成检索、编制、修复和质量控制的实现方式。

## 6. 典型变更示例

假设医学、统计和注册团队讨论后，拟将主要终点评估时间由 `Week 12` 调整为 `Week 16`。

传统方式通常需要工作人员在多个文件中全文搜索 `Week 12`，但这种方法可能遗漏：

- 使用“第十二周”“第 12 周”或其他表达的位置；
- 没有直接写 Week 12、但依赖该时间点的统计描述；
- 访视流程表中的对应行；
- 下游关联文件中的终点定义；
- 旧版本文本残留。

TrialCompiler 的处理方式是：

```text
创建候选事实新版本
  -> 计算依赖图中的直接和间接影响范围
  -> 定位章节、表格和关联文件
  -> 检查旧值和语义残留
  -> 生成候选修订文本
  -> 生成修改前后红线对照
  -> 输出影响矩阵和证据来源
  -> 提交相应专业角色审核
  -> 审核批准后更新有效事实版本
```

系统不会自行判断 Week 16 在医学或统计上是否合理。它负责确保专业人员做出的设计变化被完整、准确、可追踪地传播到文档体系中。

## 7. 飞书 AI 在方案中的作用

飞书 Aily 不是临床判断模型，而是 TrialCompiler 的协作入口与任务编排层：

1. 接收用户上传的授权资料；
2. 识别项目、文档类型和任务类型；
3. 针对缺失信息向用户补问；
4. 展示待确认 Trial Facts；
5. 触发后端编制、审核或变更影响分析；
6. 返回风险问题、候选红线、证据和审核包；
7. 让不同角色在飞书中完成反馈、确认和状态流转；
8. 记录任务、版本、人工决策和审计信息。

TrialCompiler 后端负责事实治理、依赖计算、知识检索、Agent 协作和文档测试；飞书负责降低使用门槛并融入团队日常协作。

## 8. 系统输出

系统不只返回一段生成文字，而是返回可审核的文档工程产物：

- Trial Fact Sheet；
- 候选章节正文；
- 跨章节问题清单；
- 文档单元测试结果；
- 事实到文档位置的依赖关系；
- 变更影响矩阵；
- 修改前后红线稿；
- 证据引用表；
- 未确认事实和待升级问题；
- Agent 执行轨迹；
- 人工审核决策记录；
- 最终审核报告；
- 待人工批准的经验候选。

## 9. 专业责任与安全边界

TrialCompiler 坚持 human-in-the-loop，而不是 human-out-of-the-loop。

系统可以：

- 提取候选事实；
- 定位引用位置；
- 检测确定性冲突；
- 提示可能的语义冲突；
- 生成候选文本；
- 计算候选影响范围；
- 整理证据和审核结果。

系统不能自行完成：

- 医学合理性判断；
- 统计假设和样本量合理性判断；
- 注册策略判断；
- 伦理、受试者权益和风险收益最终判断；
- 正式版本批准；
- 将候选事实自动变为有效事实；
- 将模型经验自动变为企业规则。

所有高风险输出必须能够追溯到输入、来源、模型版本、提示词版本、规则版本和人工决策。

## 10. 当前数据与验证策略

当前仍处于初步研究阶段，无法采集真实企业案例，也不应在没有授权时开展从业者访谈。因此，现阶段采用：

1. 网上公开法规、指南、论文、模板、行业报告和产品资料；
2. 根据公开规则重新构造的完全合成项目；
3. 明确标记 `synthetic: true` 的跨章节缺陷和事实变更案例；
4. 由医学成员对公开资料理解和合成案例逻辑进行普通表格复核；
5. TrialDocBench 对系统能力进行可重复的离线测试。

现阶段可以证明：

- 系统能否正确提取合成事实；
- 能否发现跨章节矛盾；
- 能否拒绝错误范围、过期或未批准知识；
- 能否覆盖事实变化的金标准影响范围；
- 能否生成最小且证据可追溯的修订；
- 多 Agent、依赖图和受控记忆是否带来可量化提升。

现阶段不能声称：

- 已在健康元内部流程中验证；
- 已证明真实客户节省多少时间；
- 已通过真实临床项目验证；
- 已达到正式临床生产或合规系统要求。

真实访谈、真实工作流测量和企业验证应在获得授权、知情同意和完整数据治理方案后开展。

## 11. 预期价值

### 11.1 对医学、统计和注册人员

- 减少重复搬运和机械全文检索；
- 更快定位跨章节冲突；
- 更清楚地看到一项设计变化影响什么；
- 将精力集中于真正需要专业判断的问题。

### 11.2 对项目管理和质量人员

- 明确事实版本和审核状态；
- 获得可追溯的修改、证据和批准记录；
- 降低旧值残留和关联文件漏改风险；
- 形成可复核的文档质量报告。

### 11.3 对组织

- 将散落审核意见转化为受控知识资产；
- 避免相似缺陷在后续项目中重复发生；
- 建立可度量的文档工程流程；
- 为进一步接入企业文档系统、质量体系和私有化部署奠定基础。

## 12. 比赛展示主线

比赛展示应围绕一个具体变化展开，而不是只展示聊天界面：

1. 导入一组完全合成但结构真实的方案章节和访视表；
2. 展示系统提取的 Trial Fact Sheet；
3. 由专业人员确认主要终点评估时间；
4. 系统发现现有文档中的 Week 12/Week 16 冲突；
5. 展示事实依赖图和受影响位置；
6. 生成最小红线、证据表和影响矩阵；
7. 展示独立质量 Agent 复核与有限返工；
8. 由人类接受、修改或拒绝候选修订；
9. 展示完整审计轨迹和待批准经验；
10. 使用 TrialDocBench 指标对比普通大模型、普通 RAG、单 Agent 和完整系统。

这条主线能够同时说明问题价值、产品可用性、技术深度、安全边界和量化结果。

## 13. 一句话总结

> TrialCompiler 不是一个替专家写临床方案的聊天机器人，而是一套以已确认研究事实为源、以文档依赖图为骨架、以自动测试和人工审核为质量门、以增量编译和受控经验为持续改进机制的临床文档工程系统。
