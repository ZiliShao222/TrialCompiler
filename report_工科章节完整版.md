# 第六部分 技术方案、系统架构与正确性机制

临床试验方案及其关联文件具有多来源、多版本、跨专业和强约束特征。同一项研究设计事实可能同时
出现在 Protocol、SAP、ICF、CRF、注册记录及内部决策材料中；不同文件又可能因人群、时间状态、
文档职责或监管范围不同而保留合法差异。人工逐份核对容易遗漏变更影响，直接使用大模型改写又
难以稳定保证来源、版本、数值边界和专业授权。

TrialCompiler 采用“概率型 AI 推理 + 确定性临床编译 + 人工治理”的技术路线：以版本化事实和
文档依赖图统一跨文件信息，由多智能体发现语义问题并根据不确定性选择取证或转交，再通过规则、
原子补丁和沙箱复检把候选结果编译为可追溯、可验证、可阻断的审核材料。系统辅助专业人员形成和
执行决策，但不替代医学、统计、注册及质量角色的正式授权。

本部分依次说明总体架构、事实与证据底座、多智能体协同、影响传播及数学正确性、不确定性与
可解释性、模型—规则协同、人工发布治理及私有化部署路径。

## 6.1 设计目标与总体分层架构

### 6.1.1 架构目标与原则

TrialCompiler 面向同一临床设计事实跨文档重复、变更传播容易遗漏以及语义判断难以形式化的问题，
需要同时回答五个问题：依据来自哪里、当前事实是什么、哪些内容受其影响、系统对判断有多大不确定
性、谁有权使结果生效。架构据此遵循四项原则：关键结论必须绑定来源和版本；事实先于章节并通过
依赖关系传播；模型生成、机器验证和人工授权相互分离；任何状态变化均可复检、阻断和回滚。

### 6.1.2 总体架构

TrialCompiler 采用“五层主链 + 独立评测面”的总体架构。五层主链依次为协同接入层、证据与事实
层、智能体（Agent）推理层、确定性编译与质量层、人工治理与正式系统层；独立评测面从冻结运行中
读取产物，但不向生成与审核过程回传预先冻结的金标准标签（Gold Standard，以下简称“金标准”）。
各层并非简单的软件组件堆叠，而是对应临床文档从“资料”转变为
“候选结论”、再转变为“可审批修订”的权威升级过程。

```text
医学/统计/注册/运营/质量人员
                │
                ▼
① 协同接入层：任务契约、资料提交、范围确认、人工补问
                │
                ▼
② 证据与事实层：来源冻结 → 文档单元化 → 候选事实 → 版本化事实基线
                │
                ▼
③ 智能体推理层：角色化审核 → 语义候选 → 竞争假设 → 不确定性决策
                │                         │
                │              证据不足时触发受控取证
                │                         └──────→ 返回证据与事实层
                ▼
④ 确定性编译与质量层：规则检查 → 依赖传播 → 原子补丁 → 沙箱复检 → 机器门禁
                │
                ▼
⑤ 人工治理与正式系统层：事实批准 → 问题处置 → 修订批准 → 质量保证复核 → 文档管理系统发布

独立评测面：冻结金标准 / 负对照 / 独立评分程序 / 校准集—测试集划分 / 回归测试
              └──只评价冻结产物，不向①—⑤提供答案
```

第一层是协同接入层。医学、统计、注册、临床运营和质量人员通过飞书 Aily、命令行界面
（Command-Line Interface，CLI）或应用程序接口（Application Programming Interface，API）发起任务，
提交公开或经授权的文件，并说明项目、任务类型、文档范围、研究阶段、允许来源和期望产物。系统先
将自然语言请求转换为结构化任务契约；范围、责任角色或关键资料缺失时先补问，不让模型自行猜测。
该层负责协作和留痕，不承担正式电子签名或文件发布。

第二层是证据与事实层。系统为每份文件和外部记录登记稳定来源标识、版本、适用范围、定位信息和
内容摘要，再将文档拆分为可定位的章节、表格和字段。模型与规则可以从中提出候选事实，但候选须
经过来源核对和专业确认后才能进入版本化临床试验事实表（Trial Fact Sheet）。事实层将终点、人群、
样本量、给药、访视和统计原则从重复文本中提升为共享对象；临床文档依赖图（Clinical Document
Graph）则记录各文档单元对事实的依赖。后续智能体读取的是带来源和状态的事实，而不是每次重新从
全部原文猜测当前设计。

第三层是智能体推理层。不同角色智能体在共享状态上执行证据审查、语义差异发现、候选修订和独立
复核，而不是通过自由聊天形成不可追踪共识。大模型重点处理跨文档语义等价、隐含冲突、适用范围判断、
竞争解释和候选表达。遇到证据不足时，不确定性治理模块比较继续读取资料、调用工具、请求专业确认、
暂缓或拒答的收益与成本；若选择取证，新观测证据（observation）必须重新登记并返回第二层，随后
重新运行相关
审核节点，不能在模型内部直接把新信息写成批准事实。

智能体审核采用 A–H 受控状态机，而不是让多个模型自由讨论后给出一份综合答案。一次既有文档审核
按下列顺序运行：

```text
A Context Lock（锁定上下文）
→ B Evidence Review（证据审查与问题识别）
→ C Repair Proposal（修订提案）
→ D Independent Quality Gate（独立沙箱复检）
→ G Uncertainty Governance（决定提交、取证、讨论或转交）
      ├─ 需要新证据：H Governed Evidence Acquisition
      │                → 新证据登记 → 返回 B/C/D/G
      └─ 证据充分或需要人工：E Review Packet（审核包）
                              → F Experience Candidate（经验候选）
                              → 合格人员决定
```

其中，A 节点把项目、文档类型、允许来源、当前批准事实和任务范围写入上下文锁，防止后续智能体
跨项目或跨版本取证。B 节点同时运行确定性预检和语义审查，输出带来源定位的问题记录（finding），
而不是直接改文档。C 节点只对证据充分且事实状态允许的问题记录生成修订提案（Repair Proposal）；
存在多种合理解释时改为专业决策请求（Decision Request）。D 节点在独立沙箱副本中应用候选补丁，
检查目标问题是否关闭、是否改变了非目标范围、来源链是否保留以及是否产生新问题，避免生成智能体
自己评价自己。

G 节点汇总 B 的证据、C 的候选和 D 的复检结果，维护竞争假设及剩余不确定性。它可以选择提交
候选（`commit`）、获取新证据（`acquire`）、增加受控讨论（`deliberate`）或转交人工（`defer`）。
选择获取新证据时，H 节点只能获取允许列表中、范围匹配且摘要可验证的证据，并把观测证据写回共享
状态；随后必须重新经过 B、C、D 和 G，不能因取证后置信度升高而直接提交。选择转交人工时，系统
生成面向指定医学、统计或注册角色的专业决策请求，并暂停相关状态迁移。

机器阶段结束后，E 节点将事实、问题记录、影响范围、候选补丁、复检结果、剩余不确定性和所需决定
汇总为审核包（Review Packet）。F 节点只提取可能复用的经验候选（Experience Candidate），不自动
写入组织记忆；经验必须经过专业
人员批准、限定适用范围并去除项目敏感信息后才可在后续任务中检索。整个流程共享类型化状态和审计
事件，每个智能体只能读写其职责内的字段，因此“多智能体”表示职责分离和受控交接，而不是多个
聊天机器人轮流生成文本。

对于全新方案生成，系统不复用“先发现缺陷再补丁”的审核链，而采用分阶段生成链。第一阶段中，
`G1 Evidence Planner`（证据规划智能体）整理证据、关键问题和候选事实表，`G2 Section Writer`
（章节撰写智能体）在同一事实约束下生成各章节，`G3 Quality Judge`（质量评审智能体）检查来源边界、
未支持声明和跨章节一致性；第一阶段冻结后，第二阶段才允许接收新增申办方、监管或中心反馈，由
`G4 Fact Reconciler`（事实调和智能体）调和当前有效事实，`G5 Incremental Reviser`（增量修订智能体）
修订受影响章节，`G6 Artifact Compiler`（关联产物编译智能体）同步访视活动计划表（Schedule of
Activities，SoA）、估计目标（estimand）、统计和实施类关联产物，`G7 Quality Judge`（第二阶段质量
评审智能体）执行复核。隐藏参考资料仅供最终独立评估器使用，不得回流 G1—G7。因此，既有文档审核
与全新方案生成共享来源、事实、门禁和审计底座，但使用不同的智能体状态机，
避免把生成任务和整改任务混成同一种流程。

第四层是确定性编译与质量层。智能体产生的问题记录和修订提案进入类型、来源、状态、
标识符、数字、日期、严格边界和旧值残留检查。事实发生变化时，系统沿依赖图计算受影响章节，只对
事实明确、范围唯一的位置生成原子补丁；补丁先应用于沙箱副本，再重新执行事实抽取、规则检查、
负对照保护和新增问题扫描。该层输出通过或阻断的机器结论以及可复核证据，不因大模型语言评分较高
而放宽硬约束。

第五层是人工治理与正式系统层。机器通过只表示候选结果具备进入专业审批的条件，不表示内容已经
获得临床或组织授权。系统分别收集事实批准、问题处置、修订批准和发布授权；需要医学、统计、注册
或质量保证（Quality Assurance，QA）判断的事项以专业决策请求进入相应角色。只有阻断问题关闭、
补丁复检通过、签字对象与当前版本摘要一致并完成独立质量保证复核后，候选文件才进入企业文档管理
系统（Document Management System，DMS）的受控签核、发布和归档流程。

独立评测面与上述业务主链隔离。它保存冻结金标准、负对照、实验划分和独立评分程序，用于评价文档
角色识别、受控缺陷检测、自然候选路由、修订局部性和不确定性策略。评测面可以读取冻结输入、预测、
补丁和审计记录，但金标准与隐藏参考不得进入智能体上下文。由此，业务流程回答“结果能否进入审批”，
评测流程回答“在预先定义的任务上表现如何”，二者不能相互代替。

### 6.1.3 关键控制关系

系统将数据、控制和评测三类状态分离。来源、事实、文档单元和补丁属于数据状态；角色权限、人工
决定和发布门禁属于控制状态；金标准、隐藏参考和实验划分属于评测状态。模型输出不能改变批准状态，
无范围的人工“同意”不能替代对象级授权，评测答案也不能进入生成上下文。

各部分通过包含项目、版本、对象标识、来源摘要和状态的结构化接口连接。依赖缺失、摘要变化、权限
不足、输出结构无效或机器门失败时，系统保留既有权威状态并失败关闭；可恢复技术错误允许有限重试，
专业证据或授权缺失则转为补问、专业决策请求或人工处理。后续各节分别展开事实、推理、传播、
治理和正确性控制的具体实现。

## 6.2 可信计算底座：来源、知识库与版本化事实

### 6.2.1 证据来源与知识库构成

TrialCompiler 将“项目事实依据”和“通用参考知识”分开管理。项目事实依据用于判断当前研究中
什么内容有效，包括经授权的方案、统计分析计划、知情同意书、病例报告表、注册记录、申办方决定
和专业人员确认；通用参考知识用于提供法规原则、文档结构、方法和风险线索，不能直接替代项目
决定。通用文献和模板的检索结果仅用于形成候选问题，不直接改变当前研究事实。

当前知识库按用途分为五类：

| 来源类别 | 主要内容 | 在系统中的作用 | 不得越过的边界 |
|---|---|---|---|
| 项目授权材料 | 当前项目文件、正式决定、批准版本和专业回复 | 建立和更新项目事实 | 未批准草稿不得冒充当前事实 |
| 公共注册与监管来源 | 美国 ClinicalTrials.gov 临床试验注册平台快照，国际人用药品注册技术协调会、美国食品药品监督管理局、欧洲药品管理局等公开资料 | 核对公开字段、法规原则和来源范围 | 通用指南不能替代项目医学或统计决定 |
| 标准、模板与关联文档 | 干预性试验方案标准条目（SPIRIT）、TransCelerate 及机构公开的方案、统计分析计划和知情同意模板 | 提供章节结构、字段覆盖和检查线索 | 模板示例值不能写入项目事实 |
| 科学方法与技术文献 | 多智能体、不确定性量化、可解释性、临床文档自动化等论文 | 支持算法设计、指标选择和研究假设 | 论文结论不等于本系统已经验证有效 |
| 经批准的组织经验 | 已确认的合法差异、规则适用条件、误报模式和整改经验 | 为后续任务提供受范围约束的经验检索 | 单次运行经验未经审核不得跨项目复用 |

知识库通过统一来源目录管理资料身份、版本、原始链接、内容摘要、使用目的和适用边界，并按监管
伦理、标准模板、数据规范、人工智能方法及项目材料等主题组织。进入验证流程的公开研究资料另行
冻结注册快照、原始文件和案例范围。检索结果必须能够返回来源登记信息和原文位置；主题分类、文本
相似度或模型摘要本身不构成事实依据。

### 6.2.2 来源登记、权威与适用范围

来源登记用于建立资料身份、原文定位、版本状态、适用范围和访问权限之间的稳定关联。所有进入事实
提取、语义审核或评测流程的资料均须先形成来源记录；仅有文件名、文本片段或模型摘要而缺少来源
定位的内容，不得用于支撑正式事实。

来源记录的主要登记要素如下：

| 登记维度 | 主要内容 | 控制目的 |
|---|---|---|
| 资料身份 | 来源编号、标题、文件类型、提供方 | 区分同名文件及不同来源 |
| 原文定位 | 网页地址、本地文件、页码、章节、表格或单元格 | 支持原文核查和证据追溯 |
| 版本信息 | 版本号、生效时间、获取时间、内容摘要 | 识别历史版本和内容变化 |
| 适用范围 | 研究、地区、人群、阶段、时间状态和文档用途 | 限定来源能够支持的结论 |
| 访问范围 | 项目权限、跨项目复用范围、评测专用标记 | 防止越权检索和评测信息泄漏 |

系统使用 256 位安全散列算法（SHA-256）生成文件内容摘要，并将摘要与来源编号和版本绑定。相同
下载地址或文件名对应的内容发生变化时，系统建立新版本，不覆盖已经参与事实批准、人工裁决或金
标准评价的历史文件。内容摘要用于验证评审对象是否发生变化，不用于判断文件内容是否正确。

来源权威采用“来源性质与适用范围联合判定”，不设置脱离用途的全局优先级。监管指南用于约束通用
原则，企业模板用于约束文档结构，项目批准文件用于确定当前设计，公开注册记录用于核对公开字段和
实际状态，科学文献用于支持方法选择和风险识别。通用指南、模板示例和技术论文均不能直接替代项目
医学、统计或注册决定。

来源差异须在研究、版本、人群、时间状态和文档用途一致的条件下进行比较。例如，批准方案中的计划
入组人数 220 与公开注册结果中的实际入组人数 198 分别描述计划状态和实际状态，不构成需要自动
改写的事实冲突；统计模板中的示例样本量 200 仅提供格式参考，不具备项目事实效力。只有适用范围
相同且规范化内容不兼容时，系统才生成冲突候选并进入专业调和。

知识库检索沿用上述来源控制。智能体只能访问任务契约允许的来源集合，检索结果必须返回来源编号、
版本及原文位置。仅有摘要、原文不可定位或版本无法确认的内容，只能作为补充取证线索。独立评测
使用的金标准和隐藏参考与业务知识库隔离，不向生成或审核智能体开放。

### 6.2.3 版本化临床事实模型

系统将跨文档重复出现的设计信息提升为版本化临床事实，而不是仅保存章节摘要。每条事实记录
（Fact Record）包括事实编号、名称、规范值、单位、适用人群、时间框架、状态、来源、责任角色、
版本、更新时间和前一值。例如数字“220”只有与“计划总随机样本量、双臂合计、包含 15% 脱落”
共同记录时才可比较；“第 24 周”也必须说明它表示主要终点评估、筛选窗口还是随访结束。

事实状态包括草稿、候选变更、需要人工复核、已批准、已拒绝和已被新版本替代。模型抽取的内容
首先进入草稿或需要人工复核状态；只有具备相应职责的人员确认规范值、单位、范围、版本和来源后，
事实才可成为指定范围内的已批准事实。新版本正式批准后，旧版本转为历史状态并继续保留，以支持
历史文件解释、审计和回滚。模型置信度、补丁执行成功或多数来源投票均不能直接改变批准状态。

同一事实可以由多个来源共同支持，来源之间可以是一致支持、补充说明、版本替代、合法范围差异或
待调和冲突。发生冲突时，系统保留各候选值及其证据，生成专业决策请求，说明冲突位置、可选处理、
影响文件和责任角色，而不由模型静默选择一个值。

### 6.2.4 文档单元与证据追溯链

系统将临床试验方案（Protocol）、访视活动计划表、统计分析计划、知情同意书、病例报告表和注册
记录拆分为可独立
定位的文档单元。单元可以是章节、段落、表格行、单元格或注册字段，并记录文档类型、文本、来源和
引用的事实。单元过粗会扩大修改范围，过细则增加维护成本；当前原型以章节和页级定位为主，企业
试点可进一步扩展至表格单元格和字段级。

完整证据链为：

```text
原始文件或注册快照
→ 来源记录及原文定位
→ 版本化事实及批准状态
→ 引用该事实的文档单元
→ 问题记录
→ 修订提案或专业决策请求
→ 沙箱复检
→ 最终人工决定
```

审核人员可以从问题向上追溯原始依据，也可以从事实向下查看受影响文件。任何必要环节缺失时，
系统仍可保存候选，但不得将其表述为来源完整或可自动修订的结论。

### 6.2.5 事实调和与完整性控制

系统比较不同文档前，先统一单位、日期、周数、大小写、范围边界和试验标识符，再判断是否属于同一
研究、同一人群、同一版本、同一时间状态、同一估计目标和同一文档用途。差异被划分为完全一致、
语义等价、合法范围差异、版本替代以及证据不足或真实冲突。只有适用范围相同且规范化结果不兼容，
才进入冲突处置。

例如，方案中的计划入组人数与注册结果中的实际入组人数可以同时成立；临床试验方案的连续研究日与
病例报告表按周期重新编号的“第 1 天”可以由时间映射解释；同一版本中主要终点分别写为第 24 周和
第 28 周，才构成需要进一步裁决的候选冲突。无法分类时，系统转交相应专业角色，而不是默认选择
最新值。

当前 50 个公开案例包含 65 份官方文件和 50 份注册快照。系统保存原始链接、字节数、页数、获取
时间和 SHA-256 摘要；自然候选进一步保存文件名、页码、短原文、候选编号和预测摘要。一个正文
不可提取的文件在构建时被拒绝并由后续案例替补，说明“下载成功”不等于“证据可用”。

该底座维持三项基本不变量：已批准事实必须具有可定位的有效来源；正式修订提案必须追溯到问题记录
和事实；需要人工复核的状态不能由模型置信度直接升级为已批准。来源完整、定位完整和语义正确分别
评价：文件摘要一致只能证明对象未改变，页码齐全只能证明内容可查找，二者都不能代替专业正确性
判断。
## 6.3 多智能体组织层：职责、状态与执行控制

多智能体组织层通过职责分离、结构化状态交换和条件路由协调各项任务，并通过并行汇合、独立复核、
人工授权和异常恢复机制，控制并行修改、模型自评及异常重试可能引发的执行风险。

### 6.3.1 职责分离

多智能体设计用于分离证据整理、内容生成、问题识别、修订建议、质量复核和治理处置。不同智能体
代表不同的任务契约、可见上下文、工具权限和输出类型，不代表其具有相应组织岗位的批准权。同一
底层模型可以在原型阶段承担多个节点的语言推理，但各节点仍须保持独立输入、结构化输出和状态权限。

| 职责类型 | 主要任务 | 允许形成的结果 | 不得执行的动作 |
|---|---|---|---|
| 上下文与证据 | 锁定任务范围、登记来源、提取候选事实 | 任务契约、来源记录、候选事实 | 猜测缺失事实或批准事实 |
| 生成与修订 | 依据已确认事实形成章节或最小修订 | 候选章节、修订提案 | 覆盖正式文件 |
| 专业审核 | 从医学、统计、注册、运营和质量视角识别问题 | 带证据的问题记录 | 自行批准其发现的问题 |
| 独立质量复核 | 在沙箱中检查来源、范围、回归和负对照 | 复检结果、阻断状态 | 用语言质量分覆盖硬失败 |
| 不确定性与治理 | 选择取证、提交或转交，整理专业决策事项 | 取证请求、审核包、专业决策请求 | 冒充专业批准人 |
| 经验管理 | 整理经确认的适用条件、误报和整改经验 | 经验候选 | 未经批准写入组织知识 |

医学审核智能体可以发现终点或安全性表述风险，统计审核智能体可以发现样本量、估计目标或分析集
问题，注册审核智能体可以发现公开字段和监管表述问题；这些输出均为机器候选。正式专业结论仍由
具备相应职责的人员作出。

### 6.3.2 共享状态与条件路由

各智能体通过统一工作流状态交换信息，不传递无约束的完整对话。共享状态包含任务范围、来源、
事实、文档单元、问题记录、修订提案、专业决策请求、质量结果、审计事件和当前流程状态。每个对象
具有稳定编号、版本和状态；每个节点只能更新其职责允许的字段。

例如，专业审核节点提交的问题记录必须包含问题类型、严重程度、文档位置、关联事实和来源；修订
节点必须引用已有问题和事实；质量节点必须分别报告目标问题是否关闭、是否出现新问题以及负对照
是否保持。结构化交接使节点能够被独立测试、重放和替换，也使人工审核能够区分机器发现、机器建议
和人工决定。

流程根据状态条件路由，而不是机械执行所有节点。任务范围不完整时返回补问；来源无效时阻断并要求
重新定位；事实未确认时进入专业事实审核；问题能够由已批准事实唯一修复时进入修订与沙箱复检；
涉及医学、统计或注册选择时生成专业决策请求；复检产生新问题时返回审核与修订；全部机器条件满足
后才形成候选审核包。没有发现问题时仍须检查资料覆盖范围，不能将空结果直接判为通过。

### 6.3.3 并行汇合、人工中断与独立复核

在来源和事实基线冻结后，医学、统计、注册、运营和质量视角可以并行检查。并行节点只允许新增问题
记录或提出候选建议，不得同时改写正式文本。汇合时按事实、文档位置和语义范围合并重复问题；证据
或结论相反的记录保留为冲突，不通过多数投票自动消除。

多个修订提案在应用前接受区间和事实冲突检查。非重叠且前置文本一致的补丁可以合并；修改范围重叠、
同一事实存在不同候选值或不同专业角色提出相反处置时，系统形成冲突包，列出各方证据、影响范围和
可选方案，并转交有权角色决定。该机制避免并行执行产生“最后写入者覆盖前者”的结果。

涉及专业判断时，工作流进入可恢复的人工中断状态。专业决策请求记录待决问题、证据、选项、影响
文件、所需角色和剩余不确定性。恢复执行前，系统核对决定者角色、决定对象版本和输入摘要；等待
期间来源或文件发生变化时，原决定须重新确认适用范围。

独立质量复核不读取生成节点的自我评价，而以实际候选文件、批准事实、确定性规则、负对照和未决
人工状态为依据。质量门检查来源完整性、事实状态、阻断问题、补丁局部性、摘要一致性、新增问题、
人工授权和回滚信息。语言质量评分仅作为辅助观察，不参与硬门禁补偿。

### 6.3.4 异常恢复与经验准入

网络超时、临时模型故障和可重试解析错误可以在限定次数内重试。系统通过任务编号、节点编号和输入
摘要识别重复调用，防止重复创建事实或重复应用补丁；节点结果写入检查点，恢复时从最后一个完整
状态继续。结构化输出修复失败、摘要不一致或权限校验失败时采用失败关闭。

专业证据不足、角色意见冲突和未知依赖不属于技术错误，不通过重复调用模型解决。此类情况应补充
来源、请求人工判断或保持转交状态。若连续迭代未关闭问题、未降低不确定性或反复读取同一证据，
系统终止自主循环并升级人工。

任务结束后，可将已经确认的“问题—证据—决定—修订—结果”整理为经验候选，并记录治疗领域、
试验阶段、文档类型、适用条件、来源、版本和批准者。经验只有在来源有效、适用范围匹配且经过专业
批准后才可供后续任务检索；法规、企业程序或来源发生变化时，原经验应失效或缩小适用范围。语义
相似仅用于排序，不构成经验适用性的判定。
## 6.4 核心计算一：依赖图、变更传播与数学正确性证明

本节解决 TrialCompiler 最核心的计算问题：**如何把临床文档中的自然语言事实转换为可量化、
可比较和可验证的表示，并在事实发生变化时，保证系统找到应处理的跨文档影响，只修改获得授权的
范围，在证据或条件不足时停止释放？**

这一问题的困难不只是抽取数字。相同数字可能属于不同实体、单位、人群和时间坐标；不同措辞可能
表达同一事实；同一句话也可能存在多个合理的医学或统计解释。若直接把整段文本压缩为向量相似度
或模型置信度，系统无法据此判断某项关系能否驱动正式修订，也无法证明修改是否完整、是否越界。

TrialCompiler 因此先将原文拆分为带来源定位的文档单元，再把其中可确定的内容转换为类型化事实、
数值、集合和逻辑约束，把无法唯一解释的内容保留为带证据状态的竞争候选；经来源核验和必要裁决后，
确认关系进入依赖图，并用于影响传播、原子补丁和重新编译。数学部分最终证明同一条主线：在自然
语言表示、依赖覆盖、补丁前置条件、复检器和人工授权等假设成立时，系统能够携证释放；任一证明
义务无法完成时，系统必须阻断或转交人工处理。

### 6.4.1 自然语言事实与依赖图的计算表示

设来源、规范事实、文档单元、finding、修订提案和人工决定的集合分别为
$\mathcal{S},\mathcal{F},\mathcal{U},\mathcal{I},\mathcal{R},\mathcal{H}$，完整图为

$$
G=(V,E),\qquad
V=\mathcal{S}\cup\mathcal{F}\cup\mathcal{U}\cup\mathcal{I}\cup\mathcal{R}\cup\mathcal{H}.
$$

文档单元不是整份文件，而是可定位、可单独比较和可局部修订的最小业务单元，例如 Protocol
中的入排标准条款、SoA 的一个访视行、SAP 的分析集定义、ICF 的风险段落、CRF 的字段约束或
注册页面中的 endpoint 字段。边至少包括 `source_of`、`depends_on`、`identifies`、`repairs`
和 `authorized_by`。每条边保存

$$
e=(from,to,type,status,evidence,created\_by,version),
$$

其中 `status` 分为 `explicit`、`candidate`、`approved` 和 `rejected`。明确字段引用、结构化
映射或人工确认的关系可作为有效传播边；仅由向量相似度或大模型推断得到的关系保持 candidate，
不能直接驱动正式改文。这样可避免把“语义上看起来有关”误当成已获授权的事实依赖。

**依赖图的构建与维护。**

依赖图按四层证据逐步建立。第一层是结构化显式边，例如事实 `primary_endpoint.timepoint` 被
Protocol endpoint 段、SoA 第 24 周行、SAP 主分析时间点和注册结果指标共同引用。第二层是
可确定抽取边，通过 NCT 标识符、日期、周数、样本量、剂量、访视编号以及表格字段映射建立。
第三层是自然语义候选边：模型从自由文本中发现可能涉及 estimand、分析人群、缺失数据、终末
事件或多重性的段落，但只记录候选、页码、原句和理由。第四层是专家裁决与人工补边；被确认的
候选升级为 approved，被否决的候选保留为 rejected 负样本，供后续回归测试使用。

图随事实版本增量维护，而不是每次重新扫描并覆盖历史。新增事实产生新节点；事实替代使用
`supersedes` 关系连接旧、新版本；文档换版时，单元通过稳定章节标识、标题路径和内容摘要进行
重定位。若定位失败，系统将相关边标记为 `orphaned` 并阻断自动补丁，而不是把旧偏移量套到
新文件。由此可以区分“事实已经改变但文档尚未同步”和“文档结构改变导致原依赖位置失效”。

正式传播仅使用显式依赖边 $E_{explicit}$ 和已经批准的语义边 $E_{approved}$；尚未裁决的语义边
$E_{candidate}$ 只形成复核任务。事实 $f$ 的有效影响集和待复核影响集分别定义为

$$
I^*(f)=Reach(f;E_{explicit}\cup E_{approved}),
\qquad
I_{review}(f)=Reach(f;E_{candidate})\setminus I^*(f).
$$

$I^*(f)$ 可进入确定性检查和补丁规划，$I_{review}(f)$ 不能驱动正式修改。该划分使语义模型可以
扩大候选召回范围，但不能仅凭相似度把未经确认的关系升级为正式依赖。

**自然语言到可计算表示。**

临床文档中的事实通常不是以数据库字段出现，而是分散在正文、表格、脚注和跨文件引用中。系统
首先按标题路径、段落、表格行列和字段边界切分文档，并为每个单元绑定文件版本、页码或结构位置、
原文摘要和内容摘要。在此基础上，规则抽取器负责识别日期、周数、剂量、样本量、试验标识符、
比较符和枚举值；语言模型负责提出实体归属、语义角色、跨句指代和潜在依赖候选。两类结果统一
写入类型化中间表示，而不是让自然语言回答直接驱动修改。

对可规范化事实，系统使用以下元组：

$$
z=(e,a,v,u,c,t,p,s,l,r),
$$

其中 $e$ 为实体，$a$ 为属性，$v$ 为规范值，$u$ 为单位，$c$ 为比较关系，$t$ 为时间锚点，
$p$ 为适用人群，$s$ 为作用域，$l$ 为来源定位，$r$ 为证据状态。例如“主要终点在第 24 周评估”
被表示为“主要终点—评估时间—24—周—等于—随机化后—目标人群—当前方案版本—来源位置—候选
或已批准”。数字“24”只有与上述字段共同出现时才具有可计算意义。

规范化过程根据内容类型生成不同的数学对象：日期和访视时间被转换为有序时间点或区间；剂量和
样本量被转换为带单位的数值；“大于”“不小于”等表达被转换为严格或非严格不等式；分析人群被
转换为集合谓词；枚举策略被转换为有限状态；无法唯一归一化的语义表述则保留为多个竞争解释。
例如“第 1、4、7、10 天给药”可表示为

$$
D=(1,4,7,10),\qquad \nabla D=(3,3,3),
$$

从而与“间隔大于 3 天”所对应的约束 $\min(\nabla D)>3$ 进行确定性比较。又如“全部随机化
受试者”被表示为集合谓词 $P_R=\{x:randomized(x)=1\}$；若另一处分析集还要求至少用药一次，
则系统能够计算集合包含关系，但不能据此自行决定应修改哪一种统计定义。

自然语言映射不被强制压缩为单一概率。对文档单元 $u_j$ 与事实 $f_i$ 的关系，系统记录三值状态

$$
R_{ij}\in\{confirmed,candidate,rejected\},
$$

并同时保存支持原句、来源位置、抽取方式、竞争解释和待确认责任角色。结构化字段匹配、明确引用
或人工确认可形成 `confirmed`；仅由模型语义判断得到的关系保持 `candidate`，只能触发复核和取证；
已确认的合法差异进入 `rejected`，作为防止误传播的负对照。由此，自然语言处理的输出先成为可
审计候选，再经过来源核验和状态升级进入依赖计算。

若一个单元存在多种合理解释，系统建立有限假设集合 $H_u=\{h_1,\ldots,h_m\}$。只有在先验和
观测更新规则均已明确配置时，才附加满足 $\sum_h b_u(h)=1$ 的信念分布 $b_u$；否则只记录候选
排序和“概率未校准”状态，不把模型生成的小数解释为真实正确率。后续新增来源、规则结果或人工
回答作为观测更新 $b_u$，并决定关系进入 `confirmed`、继续取证还是转交专业人员。这样，确定性
事实进入约束计算，歧义语义进入有限假设和不确定性计算，两者通过同一来源与状态字段衔接。

这一转换形成从文本到计算的完整链路：

```text
原始文档
→ 带来源定位的文档单元
→ 规则抽取与语义候选
→ 类型化事实元组和约束
→ 来源核验与候选裁决
→ 有效依赖边及影响矩阵
→ 传播、补丁和重新编译
```

因此，后续矩阵和谓词计算的可靠性取决于表示阶段是否完整保留实体、单位、时间、人群、作用域和
来源。无法唯一映射的语义不会被伪装成确定数值，而是保留为候选解释并进入不确定性取证或人工裁决。

### 6.4.2 变更影响、差异分类与专业边界

一次事实变化不是简单的字符串替换，而表示为

$$
\Delta f=(f,v_0\rightarrow v_1,reason,source,actor,time),
$$

其中旧值、候选新值、变更原因、授权来源、发起角色和时间戳均不可缺失。系统首先计算
$I^*(f)$，再对每个受影响单元计算旧值残留、规范值缺失、边界冲突、来源失配和范围失配等
谓词。以旧值残留为例：

$$
Stale(u,f)=\mathbf{1}[u\in I^*(f)]\cdot
\mathbf{1}[ContainsSameFact(u,v_0)].
$$

这里必须使用 `ContainsSameFact`，而不是无上下文的字符串 `Contains`。例如数字“24”可能
同时表示第 24 周、24 名受试者或表格序号；只有实体、单位、字段和语境均指向同一事实时才能
判为旧值残留。对 NCT ID 也必须区分当前研究身份与相关研究引用，避免把合法引用错误改写为
主试验编号。

每个受影响单元被分到四类处置：`required_update` 表示新值已批准、依赖明确且文本可唯一修改；
`possible_update` 表示存在语义关联但范围或表达尚不唯一；`expert_decision` 表示必须由医学、统计
或注册角色选择策略；`history_only` 表示该单元属于冻结历史版本，只记录差异而不回写。分类同时
保存触发谓词和反证条件，审核人员可以看到它为什么被归类，以及满足什么条件后可以改变处置。

**确定性计算与专业判断的边界。**

数字、日期、周数、试验标识符、严格不等式、枚举值和同一事实的旧值残留适合确定性计算。
例如给药发生在第 1、4、7、10 天，相邻间隔均为 3 天，因此“给药间隔大于 3 天”与计划表
冲突，而“间隔不小于 3 天”不冲突。此处必须显式区分 $>3$ 与 $\ge3$，不能交给模糊语义
相似度处理。

相反，estimand、分析人群、缺失数据、多重性、终末事件和注册策略通常不存在唯一的字符串
修复。例如“死亡后缺失值采用多重插补”可能涉及 composite、while-on-treatment 或
hypothetical 等不同 intercurrent-event 策略。系统可以定位矛盾链、列出受影响章节并生成
Decision Request，但不能自动选择统计策略。由此，影响计算负责完整暴露决策面，专业人员负责
确定业务语义；二者不能相互替代。

### 6.4.3 原子修订与重新编译

对 `required_update` 单元，系统生成 Repair Proposal，而不是直接生成一份无法审计的“新文档”。
每个原子编辑表示为

$$
o=(section,start,end,before,after,fact\_ids,source\_ids,precondition),
$$

并记录触发 finding、候选文本、修改理由和目标版本。`precondition` 至少要求目标区间仍等于
`before`、事实仍处于批准状态、来源版本未变化。只要任一前置条件失效，补丁就必须重新计算。
这能阻止多人并行编辑后把旧补丁错误应用到新文档。

补丁遵循最小范围原则：只修改能够由批准事实唯一确定的表达，不顺带润色邻近段落。多个补丁
区间不重叠且前置文本一致时，可以按偏移量从后向前合并；区间重叠、同一位置存在不同候选值、
目标范围不清或存在多种合理解释时，系统产生冲突对象并停止自动应用。补丁先作用于沙箱副本，
原始文档及其摘要保持不变。

**重新编译与闭环判定。**

“补丁成功执行”不等于“问题已经解决”。沙箱文档必须重新进入抽取、规范化、依赖计算和全部
确定性检查，相当于一次局部重新编译。对补丁 $p$ 定义六项闭环条件：规范新值已出现
$C_1$；需要移除的位置不再残留旧值 $C_2$；实际修改范围等于获批目标范围 $C_3$；事实—来源—
修订链完整 $C_4$；原 finding 已关闭 $C_5$；没有产生新的阻断 finding $C_6$。只有

$$
Closed(p)=\bigwedge_{r=1}^{6}C_r=1
$$

时，修订才可进入独立质量审核。若新值出现但旧值仍在另一受影响章节，状态是
`partially_repaired`；若 finding 关闭但产生新冲突，则状态是 `regression_detected`；二者均不能
被“补丁已应用”这一技术状态掩盖。

### 6.4.4 端到端示例与适用边界

假设经授权来源确认主要终点时间点由 Week 24 改为 Week 28。系统建立事实变化
`primary_endpoint.timepoint: 24→28`，直接影响集包含 Protocol endpoint 段、SoA 访视行、SAP
主分析规则和注册结果指标；ICF 中若只描述研究总时长且未引用主要终点时间点，则不应因关键词
“week”被纳入。Protocol、SoA 与注册字段中的同一事实旧值可形成 `required_update`；SAP 若还
涉及分析窗、缺失处理或 estimand，则进入 `expert_decision`。获批补丁只改三个确定位置，随后
重新运行 Week 值一致性、访视窗、来源链和旧值残留检查。只有四个依赖单元全部获得明确处置、
三个补丁通过六项闭环条件、SAP 决策由合格统计角色完成，整项变更才能结束。

**适用边界与覆盖扩展。**

依赖图的完备性是条件化的：只有已登记的事实和有效边才能保证传播。遗漏事实、扫描质量差、
章节重定位失败、隐含语义未被发现或候选边未获裁决，都可能造成漏传；过宽的语义边则会造成
误传。因此报告不声称对任意临床文档的隐含依赖具有理论上的完全召回。

工程上通过来源完整性检查、结构化字段映射、自然候选挖掘、人工补边、rejected 负样本、图覆盖
统计和回归测试持续扩大边集合。每次人工确认或否决都进入可追溯样本库，用于检验后续版本是否
仍能找到同一依赖、是否错误扩散到负对照。未知依赖被显式保留为风险和待审任务，而不是由一个
高置信度数字隐藏。

依赖图解决了“已知一项变化后应检查哪里”，但当多个事实解释竞争、候选边尚未确认或证据不足时，
仅靠图可达性不能决定下一步。不确定性计算因此将这些未决状态表示为信念状态（belief state），并使其触发取证、
拒答或人工升级。

### 6.4.5 变更编译的条件安全性

**形式化目标与证明义务。**

给定当前项目状态 $X$ 和事实变更 $\Delta f=(f,v_0\rightarrow v_1)$，一次变更编译表示为

$$
\mathcal{C}(X,\Delta f)=(X',q,\Pi),
$$

其中 $X'$ 为候选新状态，$q\in\{release,block,defer\}$ 分别表示释放、阻断和转交，
$\Pi$ 为随结果生成的证明记录：

$$
\Pi=(\pi_s,\pi_z,\pi_i,\pi_p,\pi_v,\pi_h).
$$

六个分量分别记录来源、类型化表示、影响范围、原子补丁、重新编译结果和人工授权。数学部分不试图
证明自然语言理解或医学判断在所有情况下正确，而是约束发布行为：系统只有在六类证明义务全部完成
时才能输出 $release$；自然语言无法唯一表示、依赖关系尚未确认或专业判断尚未完成时，输出必须
保持为 $block$ 或 $defer$。

自然语言到类型化表示的准确性不由定义本身保证。规则抽取部分由确定性测试验证，语义映射部分由
真实文档标签、合法差异负对照和人工裁决评估；未获确认的关系保持候选状态，不进入正式传播边集合。
因此，后续证明从“已登记并获确认的计算对象”开始，其结论不外推至尚未识别的隐含语义。

**三项可证明性质。**

**性质一：已登记依赖范围内的传播覆盖性。** 设有效边集合为

$$
E^*=E_{explicit}\cup E_{approved},
\qquad
I^*(f)=Reach(f;E^*).
$$

若文档单元 $u$ 对事实 $f$ 的依赖已经登记在 $E^*$ 中，则按照可达集定义必有
$u\in I^*(f)$。因此，系统能够保证已登记依赖不会在传播计算中被遗漏；这一结论不等于所有自然
语言隐含依赖均已被发现。候选语义边进入待复核集合，只有确认后才获得该性质。

**性质二：原子补丁的局部性。** 设批准修改范围为 $T$，补丁由若干原子编辑组成：

$$
P=\{o_1,\ldots,o_r\},
\qquad
o_j=(section,start,end,before,after,precondition).
$$

若每个编辑区间均包含于 $T$，各区间两两不重叠，且应用时目标版本、修改前文本和事实状态仍满足
前置条件，则

$$
Changed(X,X')\subseteq T.
$$

证明如下：每项编辑只作用于自身目标区间，所有目标区间均为 $T$ 的子集，因此 $T$ 外内容不受
修改；按偏移量从高到低应用时，不相交编辑不会改变尚未执行编辑的坐标。若区间重叠或前置条件
失效，系统不再具有该局部性保证，必须停止自动应用。

**性质三：发布门禁的非补偿性。** 设软质量分为 $S(X')$，必要硬约束为
$g_1(X'),\ldots,g_m(X')\in\{0,1\}$，人工授权为 $H(X')$，则发布条件为

$$
Release(X')=H(X')\land\bigwedge_{j=1}^{m}g_j(X').
$$

任一硬约束失败或人工授权缺失时，$Release(X')=0$；由于 $S(X')$ 不属于该合取式，提高语言
流畅度、模型置信度或综合评分不能抵消来源缺失、修改越界、复检失败或授权缺失。

重新编译按照六项条件判定整改是否完成：规范新值出现、应删除位置不再残留旧值、实际修改范围
等于批准范围、事实—来源—修订链完整、原问题关闭且没有新增阻断问题。上述条件是发布前的复检
规则，而不是对医学语义正确性的独立证明。

**条件安全释放命题。** 若系统输出

$$
\mathcal{C}(X,\Delta f)=(X',release,\Pi),
$$

则必须同时满足

$$
Traceable(X')=1,
\qquad
RegisteredImpactDisposed(X')=1,
$$

$$
Changed(X,X')\subseteq T,
\qquad
Revalidated(X')=1,
\qquad
HumanAuthorized(X')=1.
$$

证明来自发布条件的构造：$\pi_s$ 和 $\pi_z$ 提供来源及表示记录，$\pi_i$ 证明所有已登记影响
均有明确处置，$\pi_p$ 及原子补丁局部性保证修改不超出 $T$，$\pi_v$ 保存重新编译结果，
$\pi_h$ 保存有效人工授权。发布门禁逐项检查这些记录；任一记录缺失或检查失败，合取条件为假，
$q$ 不能取 $release$。因此，该命题保证系统不会在已声明的证明义务未完成时静默释放结果。

自动整改只在未关闭阻断问题持续减少且没有新增问题时继续。若新证据扩大影响范围、产生新的阻断
问题或引发专业意见冲突，系统立即停止自动循环并转入人工决策，不对开放式临床推理作自动终止保证。

**证明边界与验证责任。**

上述命题证明的是发布机制的条件安全性，而不是整个临床内容的无条件正确性。它能够支持的结论是：
对已经识别、登记并纳入检查的对象，系统可以验证来源、处置范围、补丁局部性、重新编译状态和人工
授权，并在证明义务不完整时阻止释放。

该命题不能证明任意临床文档中的事实均能被完整抽取，也不能证明所有隐含依赖、语义冲突或专业
策略均能被机器正确识别。自然语言表示准确性、候选依赖发现率和不确定性排序能力由第七部分实验
评估；样本量、估计目标、缺失数据和监管路径的最终选择由相应专业角色裁决；精确率、召回率、
F1 值和 Wilson 区间属于实验结果的统计描述，不属于数学正确性证明。

因此，本节的数学保证可以概括为：**对已确认计算对象实施可验证传播和局部修订，在证明记录完整时
携证释放，在表示、依赖、复检或授权不足时安全阻断。** 这一保证既覆盖系统能够实际执行和审计的
工程性质，也避免把条件性结果扩大为对临床总体准确性或专业正确性的承诺。

## 6.5 核心计算二：不确定性驱动取证与解释忠实性

TrialCompiler 的 AI 研究核心不是让模型为一段答案附加一个 confidence，而是研究概率型 Agent
如何在多文档、证据不完整且来源可能冲突的环境中形成竞争假设，主动选择下一项证据，并在无法
安全收敛时拒绝提交。系统因此把不确定性视为能够改变行动和发布状态的计算对象；可解释性则不是
自动生成一段理由，而是要求 finding、动作和候选补丁携带可重放、可干预和可证伪的证据。

### 6.5.1 先区分诊断信号、不确定性与授权状态

来源冲突、证据缺失、模型分歧、影响范围未知和权限不足在界面上都可能表现为“无法确定”，但
它们并不是同一种量。来源冲突和证据缺失是观测到的诊断信号；权限不足是治理状态；多次采样
不一致是估计信号；只有针对明确事件、明确层级定义的分布或风险，才是不确定性估计。否则一个
没有语义对象的“87% confidence”既不能校准，也不能决定该读取哪份文件。

本项目从两个正交维度描述不确定性。第一维是性质：epistemic uncertainty 来自知识、来源或
模型能力不足，原则上可以通过取证、验证或专家判断降低；aleatoric uncertainty 来自环境或观测
过程本身的随机性和不可约歧义，不能承诺通过更多推理消除。第二维是位置：不确定性可以位于
observation、belief state、next action 或 trajectory outcome。例如 OCR 对一个数值的识别风险
属于 observation 层；两个规范事实解释的竞争属于 belief 层；下一步应读 SAP 还是注册记录属于
action 层；整条轨迹最终能否生成通过独立复检的补丁属于 outcome 层。

任何数值输出必须同时声明 target event。可接受的事件包括“该候选事实经专业裁决为真”“该
finding 与冻结 Gold 匹配”“该补丁通过独立复检”或“该轨迹无错误提交”；不可接受的写法是脱离
事件只报告“系统置信度”。只有估计器在独立 calibration split 上拟合或选阈值，并在未参与调参
的 test split 上评价后，报告才允许使用 calibrated probability。模型自报概率、reviewer rule
score 和演示数据中的手工 posterior 均不能据此冒充真实校准结果。

### 6.5.2 跨文档事实调和的有限假设状态

对一个候选差异，系统首先建立有限且互斥程度足够的假设集合
$\mathcal{H}=\{h_1,\ldots,h_q\}$。例如 NCT ID 不一致可以对应：$h_1$ 当前文件身份错误，
$h_2$ 文中是合法的相关研究引用，$h_3$ 来源范围配置错误，$h_4$ 证据不足。主要终点时间点
不同则可能对应真实冲突、历史版本、亚组与总人群差异或计划值与实际值差异。假设必须附有支持
证据、反对证据、可证伪条件和允许的下一动作，而不是由模型自由扩写成无限故事。

在时刻 $t$，Agent 维护 belief state

$$
b_t(h)=P(H=h\mid o_{1:t},a_{1:t-1}),\qquad
\sum_{h\in\mathcal{H}}b_t(h)=1,
$$

其熵为

$$
H(b_t)=-\sum_{h\in\mathcal{H}}b_t(h)\log b_t(h).
$$

观察 $o_{t+1}$ 到达后，系统依据显式 observation model 更新 posterior：

$$
b_{t+1}(h)=
\frac{P(o_{t+1}\mid h,a_t)b_t(h)}
{\sum_{h'}P(o_{t+1}\mid h',a_t)b_t(h')}.
$$

该公式要求实验明确给出 prior 和 likelihood，不能让大模型临场编造似是而非的小数。当前原型
使用有限假设和显式 observation-to-posterior 配置验证状态更新、治理和报告链路；这证明算法
管道可执行，不代表 posterior 已由真实临床数据拟合。

### 6.5.3 不确定性估计信号及其适用边界

系统可以组合多种估计信号，但不把它们互相等同。token likelihood 或 NLL 成本低，却会受到
文本长度和措辞影响；verbalized confidence 对黑盒模型易取得，但通常过度自信；多次采样的
self-consistency 可以发现答案不稳定，却无法发现“稳定地答错”；semantic entropy 在语义等价类
上度量采样分散，能减少字面差异的干扰，但仍依赖多次采样和可靠的语义聚类；来源或 verifier
分歧贴近本任务，却必须考虑多个 verifier 的相关错误，不能把多数票直接当概率。

若未来训练 trajectory calibrator，可把早期风险、跨步不确定性梯度、工具失败和证据稳定性作为
特征预测最终补丁是否正确，但必须防止数据泄漏和分布漂移。conformal 方法可以在明确的校准总体、
exchangeability 假设、nonconformity score 和 coverage event 下提供覆盖或风险保证；如果这些
条件没有写清，报告不能只因为使用了 conformal 一词就声称具有分布无关安全保证。

对于长轨迹，还必须说明怎样聚合步骤风险。设第 $t$ 步错误概率估计为 $r_t$，可以报告终态风险
$r_T$、平均风险 $T^{-1}\sum_t r_t$，或最弱步骤风险 $\max_t r_t$。TrialCompiler 的发布门禁优先
使用最弱步骤和关键动作风险，因为平均值可能掩盖一次错误的来源选择、事实覆盖或高风险补丁。

### 6.5.4 从被动置信度显示到主动取证

Agentic UQ 的关键是让不确定性改变下一步动作。系统的动作空间包括读取指定来源、请求补充附件、
增加语义采样或 verifier 预算、调用确定性工具、询问医学/统计/注册专家、生成候选补丁、降低主张
具体度、defer 和 abstain。对候选取证动作 $a$，预期信息增益定义为

$$
EIG(a)=H(b_t)-\mathbb{E}_{o\sim P(o\mid a,b_t)}[H(b_{t+1}\mid a,o)].
$$

若动作成本为 $c(a)$，风险权重为 $\lambda$，则在允许动作集合 $\mathcal{A}_{safe}$ 中选择

$$
a^*=\arg\max_{a\in\mathcal{A}_{safe}}
\{EIG(a)-\lambda c(a)\}.
$$

成本不仅是模型 token，还包括 PDF 页数、外部查询、人工分钟数、等待时间和敏感数据暴露面。
高 EIG 但越权的动作不属于 $\mathcal{A}_{safe}$；低成本但不能区分竞争假设的重复读取也不应被
选择。每个动作记录选择前 belief、预期收益、实际 observation、更新后 belief、实际熵下降、来源
版本和内容摘要，使评估者能判断 Agent 是否真的因不确定性而取证。

### 6.5.5 提交、继续取证、暂缓与人工转交

设最高后验假设为 $h^*$，候选补丁通过独立验证的估计风险为 $R_t$，剩余预算为 $B_t$。停止策略
不是单一 confidence 阈值，而是受硬门禁约束的选择性决策：

$$
Decision_t=
\begin{cases}
commit, & b_t(h^*)\ge\tau_h,\ R_t\le\tau_r,\ HardGate=1;\\
acquire, & \max_a(EIG(a)-\lambda c(a))>0,\ B_t>0;\\
defer, & QualifiedHumanAction\ available;\\
abstain, & \text{otherwise}.
\end{cases}
$$

即使 belief 越过阈值，候选结果仍必须重新进入第 6.4 节的来源、影响范围、原子补丁和沙箱复检，
不能绕过确定性质量门。若连续若干轮没有关闭 finding、没有带来实际熵下降，或重复访问同一来源，
系统应停止自循环并转交人工。这样“不知道”会产生明确的安全动作，而不是变成更长的自然语言
推理。

### 6.5.6 可解释性的五个层次与携证输出

本项目区分五种经常被混用的概念：provenance 说明用了哪些来源；observability 记录实际执行了
哪些动作和工具；plausibility 表示解释在人看来是否合理；faithfulness 要求解释确实反映影响决策
的因素；causal/mechanistic explanation 则声称内部机制或结构因果关系。前两者是工程可审计性，
并不自动证明后两者。大模型生成的流畅 rationale 或 Chain-of-Thought 也不能因为“听起来合理”
就被视为忠实解释。

TrialCompiler 因而输出 Proof-Carrying Clinical Compilation（携证临床文档编译）对象，而不是
只输出修改后的文字。一个携证结果至少包含：可证伪主张、规范事实及版本、引用来源及定位、竞争
假设和剩余不确定性、实际 action/tool trace、依赖影响集、候选原子补丁、补丁前置条件、沙箱复检
结果、负对照结果、人工授权状态和未清偿的专业决策债务。该对象既可以证明“机器结果可进入合格
人员审批”，也可以证明“当前证据不足，系统正确阻断”。

解释面向角色分层呈现。医学审核者需要看到临床语义和风险证据；统计人员需要看到 estimand、
分析集和缺失处理的逻辑；QA 需要看到来源、版本、改动范围和复检结果；系统维护者需要看到工具
轨迹和失败代码。分层只改变呈现，不改变底层证据对象，防止为不同用户生成彼此不一致的理由。

### 6.5.7 用反事实重放检验解释忠实性

对黑盒模型，当前可以验证的是 behavioral counterfactual faithfulness，而不是模型内部神经机制。
若解释声称证据 $e$ 对决定 $d$ 是必要的，则在固定模型、提示、工具和随机配置下移除 $e$ 并重放；
决定应发生预期变化。若把 $e$ 替换为支持竞争假设 $h'$ 的对立证据，系统的 belief、动作、finding
或补丁应向 $h'$ 方向移动。若输出不变，则该证据可能只是被展示出来，却没有真正影响决策。

设 $N$ 个案例均声明某证据必要，necessity flip rate 定义为

$$
NFR=\frac{1}{N}\sum_{i=1}^{N}
\mathbf{1}[d_i^{-e}\neq d_i].
$$

设对立证据干预后的期望方向为 $q_i$，contrastive sensitivity 定义为

$$
CS=\frac{1}{N}\sum_{i=1}^{N}
\mathbf{1}[Direction(d_i^{e\rightarrow e'},d_i)=q_i].
$$

还应报告只保留解释证据时的 sufficiency、移除解释证据时的 comprehensiveness、随机重放下干预
效应的 replay variance，以及解释中无法在实际 trace 或来源找到支撑的 unsupported-rationale
rate。上述指标只能支持行为层的忠实性结论；没有结构因果模型与受控干预设计时，不声称证明了
模型内部因果机制。

### 6.5.8 校准、选择性预测与实验指标

若 $p_i$ 表示目标事件成功的预测概率、$y_i\in\{0,1\}$ 为独立标签，则 Brier score 为

$$
Brier=\frac{1}{N}\sum_{i=1}^{N}(p_i-y_i)^2.
$$

ECE 将样本按置信度分箱并比较每箱平均预测与经验准确率；样本有限或分箱不稳定时，同时报告
binning 方案、样本数和置信区间。若系统只提供风险排序而没有可靠概率，应报告 pairwise rank
accuracy 或 rank-calibration，而不是强行计算“校准概率”。

对拒答或人工升级策略，在阈值 $\tau$ 下定义 coverage 为系统自动提交的样本比例，selective risk
为这些已提交样本中的错误率。risk–coverage 曲线展示自动化覆盖扩大时风险如何变化，AURC 汇总
整条曲线。一个有用的不确定性机制应在降低 coverage 时优先排除高风险样本，而不是随机拒答。
此外还需报告错误自动提交率、defer recall、每个正确闭环的证据成本、单位成本实际熵下降、取证后
成功增益、负对照保护率和 trajectory weakest-link 校准。

实验应比较 deterministic rules only、single-shot LLM、fixed RAG + LLM、fixed RAG + passive
UQ/abstention、uncertainty-guided acquisition，以及 acquisition + counterfactual explanation gate。
只有这种消融才能回答提升来自普通检索、被动拒答、主动取证还是解释门禁，而不能把整个系统的
改进笼统归因于“使用了 AI”。

### 6.5.9 当前实现范围与声明边界

当前仓库已经具备有限假设 belief state、显式 observation-to-posterior 更新、成本感知 EIG 选证、
commit/acquire/defer 决策、受控取证预算、轨迹记录，以及 Brier、ECE、risk–coverage/AURC、排序
准确率、necessity flip rate 和 contrastive sensitivity 的可复现实验底座。未知 observation、
重复访问、越权动作和预算耗尽采用失败关闭；混用 calibration/test split 时禁止产生校准声明。

但当前尚未完成真实多采样模型 UQ、冻结且由专业人员裁决的独立 calibration/test 数据集、learned
trajectory calibrator、来自真实外部工具或专家的 observation model、多轮真实来源主动取证对照
实验，以及具备统计功效的解释忠实性 benchmark。因此现阶段可以声称“已经实现并测试 UQ/XAI
实验与治理机制”，不能声称“真实临床文档上的概率已经校准”或“主动取证策略已经证明优于所有
基线”。这一边界使研究亮点建立在可复现机制和明确待验证假设上，而不是把演示分数包装成临床
有效性结论。

6.5 和 6.6 分别给出了影响推理与不确定性行动，但二者仍需要一个执行契约：开放语义任务由模型
完成到什么程度，哪些结果必须交给规则验证，冲突时谁具有阻断权。6.7 将前述算法放回同一双平面
系统，完成从“核心计算”到“可核验机器输出”的合流。

## 6.6 双平面合流：大模型推理与确定性临床编译

TrialCompiler 不把大模型当作一台能够直接接收文件并交付最终答案的黑盒，也不把规则系统包装成
AI。整体采用“概率型 AI 推理平面 + 确定性临床编译平面”的双平面结构：前者负责规则难以枚举的
语义发现、竞争假设、证据规划和候选表达；后者负责类型、来源、状态、依赖、边界、修改范围和
发布门禁。二者通过类型化中间对象交换结果，而不是通过自然语言互相覆盖。

### 6.6.1 两类能力的互补边界

确定性规则适合处理存在明确语法和判定边界的问题，包括 NCT 标识符、日期、Week/Day/Hour、
剂量与单位、严格不等式、批准状态、来源定位、旧值残留、补丁区间和文档摘要。例如访视发生在
第 1、4、7、10 天，相邻间隔为 3 天；“间隔大于 3 天”必然为假，而“间隔不小于 3 天”为真。
这类问题若交给语言模型凭语感判断，反而降低稳定性。

大模型适合处理表达空间开放的问题，包括不同措辞是否表达同一临床概念、一个段落究竟描述总
人群还是亚组、Protocol 与 SAP 的统计范围是否相容、某个 NCT ID 是当前研究身份还是相关研究
引用、缺失信息应从哪类证据补充，以及如何把已批准事实改写成与原章节风格相容的候选文本。
这些任务的输出天然带不确定性，不能未经验证直接转成正式事实。

两类能力的关系不是“规则先做简单题、模型做复杂题”这么简单。规则还承担模型输出后的反证与
约束检查；模型也可为规则发现新的覆盖盲区。例如模型提出 Week 24→28 的候选语义变更后，规则
检查所有已登记依赖单元是否仍残留 24，并验证补丁是否越界；若大量人工复核反复确认某类固定
模式，团队可以把它提升为新的确定性规则。系统由此形成“语义发现—人工裁决—规则固化—回归
验证”的知识工程循环。

### 6.6.2 类型化中间表示与信任升级

模型不能把自由文本直接写入正式文档。它必须输出符合模式约束的对象，例如 `FactCandidate`、
`SemanticFindingCandidate`、`EvidenceRequest`、`RepairProposal` 或 `DecisionRequest`。一个语义
候选至少包含：主张、适用范围、引用原句、来源定位、竞争解释、不确定性对象、受影响事实和建议
责任角色。解析失败、字段缺失、引用无法定位或对象越出项目范围时，结果失败关闭。

对象在流水线中按以下信任等级升级：

```text
model suggestion
→ schema-valid candidate
→ source-grounded candidate
→ deterministic checks passed
→ qualified-human approved fact/decision
→ sandbox-validated repair
→ QA-authorized release
```

任何阶段都不能跳级。模型多次给出相同答案只增加一种稳定性信号，不能把 candidate 自动变成
approved；规则通过说明已编码谓词未发现问题，也不能自动等于医学或统计判断正确；人工批准事实
也不自动批准具体补丁或最终发布。这一信任升级链把语言流畅度、机器一致性和组织授权严格分离。

### 6.6.3 两个平面在 A–H 状态机中的交接

第 6.1.2 节已经定义 A–H 节点及取证回环，本节只说明两个平面在哪里交接。A 的范围锁
和 B 的确定性预检属于编译平面；B 中的语义审查和 C 的候选表达主要由 AI 推理平面完成；D 把候选
重新降回可计算对象，在沙箱中执行编译检查；G 读取 B/C/D 的真实结果形成 belief 与下一动作；H
获取新 observation 后回到 B，而不是直接进入 E。E 汇总的是已经通过交接约束的 Review Packet，
F 产生的经验也仍只是 candidate。

```text
编译平面锁定输入与硬约束
→ AI 平面发现语义问题、竞争假设与候选修订
→ 编译平面验证来源、影响范围、补丁与回归
→ 不确定性治理决定提交、继续取证或转交
→ 人工治理决定事实、修订和发布是否生效
```

每次跨平面交接都必须携带类型化对象、输入摘要、证据定位和状态，不能只传递一段自然语言。
候选生成和独立质量判断使用不同节点与产物；D 检查实际沙箱结果，而不是让 C 自己声明修改成功。
若存在 evaluator-only 材料，它们与生成上下文保持输入目录和摘要隔离，避免评估答案回流。这样，
A–H 的职责与状态控制见第 6.3 节，本节说明该组织结构如何执行 AI—规则协同。

### 6.6.4 规则、模型与人工的冲突裁决

当模型与确定性规则意见不一致时，不能通过平均分数解决。若规则对应类型、数值、来源、权限或
发布硬约束，则规则失败产生 `blocked_machine_gate`，无论语言质量分多高都不得释放。模型可以
质疑规则适用范围并生成复核请求，但不能覆盖门禁。若模型发现的是规则没有编码的语义风险，则
保留为 candidate 并请求相应专业角色裁决；规则没有报警不能被解释为“语义上正确”。

同样，确定性 finding 也不自动等于临床结论。NCT ID 字符串不同是可重复发现的工程事实，但它
可能是文件身份错误，也可能是合法相关研究引用；死亡、MAR 和 multiple imputation 同时出现可
触发统计风险规则，但最终 intercurrent-event 策略需要统计与医学判断。系统分别保存“检测事实”
和“专业处置”，避免把稳定检测器包装成自动临床裁决器。

合法差异作为负对照进入规则与模型两侧。例如 Protocol 使用连续 Day -1 至 Day 11，而 CRF/SAP
按四个 Period 分别记录每期 Day 1，可能是合法时间轴映射。只比较数字会产生误报；只有在来源、
scope 和映射关系均不成立时才升级为冲突。负对照保护率因此与问题召回率同等重要。

### 6.6.5 模型调用的证据与安全约束

进入模型上下文的材料必须经过项目、文档类型、版本、范围和权限过滤。提示词要求模型引用输入
中的证据标识和定位；不存在于允许上下文的来源不得被补造。外部文档中的指令性文字按不可信数据
处理，不能改变系统角色、工具权限、输出路径或发布状态。模型输出经过结构校验、引用存在性检查、
范围检查和敏感字段检查后才可进入下一节点。

为了可复现，运行记录至少包括模型标识、提示模板版本、温度与采样参数、输入摘要、工具调用、
结构化输出摘要和失败代码。若模型或提示发生变化，旧 checkpoint 不能无条件复用。涉及长文档时，
分章节执行和 checkpoint 可以提高容错性，但每个断点都要绑定输入摘要，防止在材料已经换版后继续
使用旧中间结果。

### 6.6.6 协同输出与可核验完成条件

最终报告不是把规则列表与模型回答并排放置，而是把二者编译为统一 finding：规则或模型类型、
触发事实、原始证据、目标章节、依赖范围、严重程度、竞争解释、不确定性、责任角色、允许动作和
关闭条件。对确定性问题，关闭条件可以是新值出现、旧值移除和回归检查通过；对语义问题，关闭
条件还包括专业决定及其适用范围。

一次协同运行只有在以下条件满足时才算机器阶段完成：所有模型输出均已模式化；所有正式主张均有
可定位来源；候选与批准状态没有混淆；硬门禁全部通过；受影响单元均获得明确处置；补丁在沙箱中
复检；剩余不确定性和专业决策债务显式列出。这个完成定义使大模型提供开放语义能力，同时把可
重复性、边界精度和状态安全交给确定性编译平面。

机器阶段最多能够产生来源充分、范围明确、复检通过并携带剩余不确定性的候选结果，但不能自行
赋予医学、统计、注册或发布效力；相应效力须由具备职责权限的专业人员和质量管理角色授予。

## 6.7 治理出口：人工授权、整改状态与发布边界

第 6.4 节说明怎样计算和修订，第 6.5—6.6 节说明 AI 怎样形成候选并接受机器约束，本节回答
最后一个不能由算法代替的问题：谁有权让哪一种状态发生变化。人工在这里不是模型置信度不足时
临时加入的兜底，而是事实生效、专业风险接受和正式发布的授权主体。系统自动化的是证据整理、
影响计算、候选修订和复检，不是专业责任本身。

### 6.7.1 四类批准不可相互代替

TrialCompiler 把四种容易混淆的批准分开。`fact approval` 确认某个值、定义或策略在指定范围内可
进入规范事实层；`finding disposition` 决定问题成立、误报、接受风险、需要补证或超出范围；
`repair approval` 允许特定补丁作用于特定文档版本和区间；`release authorization` 才允许修订后
材料进入下一正式阶段。

四者形成有向前置关系，而不是一个布尔 `approved=true`：

$$
FactApproved\not\Rightarrow RepairApproved,
\qquad
RepairApproved\not\Rightarrow ReleaseAuthorized.
$$

例如批准“主要终点时间点为 Week 28”，并不意味着批准系统生成的所有 Week 24→28 替换；SAP
中的分析窗可能需要另行统计判断。补丁在沙箱中通过也不等于可以发布，因为独立 QA、剩余 finding
和文档版本仍需核对。这一分层防止一次宽泛点击被解释成无限授权。

### 6.7.2 授权对象、作用域与有效期

每项授权表示为

$$
Auth=(subject,role,object,action,scope,validity,evidence,signature),
$$

其中 `object` 可以是事实、finding、Decision Request、补丁或发布包；`scope` 限定项目、国家/
地区、文档类型、章节、版本和适用人群；`validity` 规定授权时点和失效条件。每次决定保存决定人、
资格角色、输入证据、理由、被授权对象摘要及电子签名或等效审计标识。

授权绑定内容摘要而不仅是文件名。对象文本、来源版本、事实版本、影响范围或候选补丁发生实质变化
后，旧签字状态转为 `stale_authorization`，必须重新确认，不能静默继承。撤销批准时不删除旧记录，
而是新增 `revoked` 事件并记录原因和替代决定，以保留完整时间序列。

### 6.7.3 专业角色与责任分离

医学负责人负责疾病定义、临床终点含义、安全风险和医学可接受性；统计负责人负责 estimand、分析
集、样本量、缺失数据、多重性和敏感性分析；注册负责人负责注册字段、地区路径和公开披露范围；
临床运营角色负责访视可执行性、中心流程和招募约束；数据管理角色负责 CRF 字段、编码和数据流；
QA 负责版本完整性、审计链、程序遵循和发布前独立复核。

系统管理员可以配置账号、运行环境和技术工作流，但不能借管理员权限代替医学、统计、注册或 QA
签字。候选补丁的提出者原则上不能独自完成同一补丁的最终 QA 放行；高风险事实的批准者与发布者
应保留可识别的职责分离。若小团队必须兼任角色，系统应显式记录角色兼任和补偿性复核，而不是
伪造独立性。

### 6.7.4 Decision Request：把不确定性转成可回答问题

当证据不足、多个解释均合理或修改方式不唯一时，系统生成 Decision Request。请求至少包括：
待决问题、竞争事实或假设、支持与反对证据、来源权威和版本、受影响文档单元、各选项的下游影响、
模型与规则分歧、剩余不确定性、建议责任角色、答复期限和不作决定的风险。选项必须允许“需要更多
证据”“转交其他角色”和“均不接受”，避免用预设选项迫使专家认可模型建议。

专家答复不能只有一段自由文本。结构化决定包括所选处置、理由、适用范围、生效事实、是否允许
机器补丁、必须人工编辑的部分、复检要求和失效条件。例如 NCT ID 不一致可能源于文件身份错误、
相关研究合法引用或来源范围错误。即使专家确认“需要处理”，仍可返回
`approved_but_manual_edit_required`；这表示问题处置已获批准，但机器尚不能唯一决定应改编号、
改文件分类还是改来源范围。

### 6.7.5 整改工单状态机与退出条件

主整改路径为：

```text
open
→ triaged
→ decision_pending
→ approved_for_repair
→ repaired_in_sandbox
→ revalidated
→ qa_approved
→ released
```

`rejected`、`false_positive`、`accepted_risk`、`needs_evidence`、`manual_edit_required`、
`partially_repaired`、`regression_detected`、`stale_authorization` 和 `withdrawn` 是独立分支，不能
压缩成一个“已处理”。每次状态转移具有明确前置条件、允许角色、必需产物和退出证据；非法跳转
被拒绝并写入审计日志。

`accepted_risk` 也不是关闭问题的快捷键。它必须包含风险说明、影响范围、批准角色、补偿措施、
复审日期和是否阻断发布。`false_positive` 必须保存负对照或范围解释，后续规则回归时用于防止同类
误报。`needs_evidence` 必须产生具体取证任务和预算，而不是让 finding 无限停留在开放状态。

### 6.7.6 发布门禁与非补偿原则

设阻断 finding 数为 $B(X)$，高风险决定是否齐备为 $H(X)$，沙箱复检是否通过为 $V(X)$，独立
QA 是否完成为 $Q(X)$，审核对象摘要是否仍一致为 $D(X)$，则

$$
Release(X)=
\mathbf{1}[B(X)=0]\cdot H(X)\cdot V(X)\cdot Q(X)\cdot D(X).
$$

任一项为 0，发布结果均为 0。语言质量分、模型置信度、修订数量和截止日期压力都不出现在补偿项
中，因此不能用“整体看起来很好”抵消一个未解决的硬门禁。紧急情况下可以缩小发布范围、发布经
批准的部分版本或走有记录的例外流程，但不能把失败状态改名为通过。

发布包至少冻结：输入文件与摘要、批准事实快照、finding disposition、Decision Request 决定、
实际补丁、沙箱复检结果、剩余非阻断风险、角色签字、软件/规则/模型版本和最终输出摘要。任何正式
输出都可由这些对象重放到相同的治理结论；无法重放的结果不能作为受控发布证据。

### 6.7.7 审计、撤销、回滚与事后学习

审计日志采用追加式事件记录，至少包含 actor、role、timestamp、before/after state、object digest、
reason、evidence 和 correlation ID。查看、取证、提出修改、批准、应用、撤销和发布分别记录，不能
只保存最终文档。敏感内容按最小必要原则展示，日志访问本身也应被审计。

若发布后发现来源错误、专业决定撤销或补丁产生延迟回归，系统根据事实版本、影响集和发布摘要定位
受影响产物，创建 recall/rollback 工单，并恢复到最近一个已批准状态；不能通过删除事件伪造从未
发生。事后确认的合法差异、误报模式和人工补边可以形成 Experience Candidate，但只有经过专业
复核、去除项目敏感内容并限定适用范围后，才可成为后续运行可检索的组织经验。

### 6.7.8 一个完整授权示例

以 enrollment 从 120 调整为 150 为例：来源更新首先产生候选事实，临床运营或相应责任角色在指定
国家、版本和招募范围内完成 fact approval；系统依据第 6.4 节计算 Protocol、注册记录、预算假设和
中心计划的影响集。纯数字旧值可形成原子补丁，但若样本量 120 同时来自统计功效计算，系统不得仅
做全局替换，而需统计角色判断这是计划招募数变化还是统计假设变化。

随后 finding disposition 分别确认各处是旧值、合法历史还是需专家决定；repair approval 只覆盖
已确定的 Protocol 与注册字段；补丁进入沙箱并复检旧值残留、统计一致性和负对照。QA 核对实际
修改范围、来源链和未决统计问题。只有统计问题已关闭或被明确排除在本次发布范围外、所有摘要仍与
签字对象一致，release authorization 才能成立。该例说明人工授权并不降低自动化价值，而是让
Agent 的高效传播与专业责任形成可审计接口。

## 6.8 工程落地：私有化部署及系统对接路径

当前代码提供 CLI 和 FastAPI，可与飞书 Aily 及企业系统逐步对接。生产部署仍需完成项目与
角色双层最小权限、传输和存储加密、密钥外置、访问日志、数据保留期限、异常访问检测、备份
恢复、模型供应链和回滚演练。API 密钥通过环境变量提供，不进入仓库或运行报告。

建议先在公开、合成或历史非生产项目开展 POC，冻结输入、模型和指标，对比纯人工流程与
TrialCompiler 辅助流程的发现率、错误修订率和审核时间。达到预注册质量、安全和流程门槛后
再扩展文档类型与团队。当前项目没有声称已经完成企业生产部署或真实患者数据合规验证。

# 第七部分 原型验证、实验结果与可复现性

## 7.1 当前原型完成情况

仓库已经实现事实和来源契约、依赖图、多角色审核、分阶段方案生成、确定性检查、语义候选、
原子修订、沙箱复检、Decision Request、审计记录、不确定性实验、独立 scorer、CLI 和 API。
本报告冻结版本在本地完整运行得到 130 passed；GitHub Actions 对此前冻结提交运行成功，当前
提交仍以仓库 CI 页面记录为准。验证材料只使用公开研究层资料或完全合成材料，不包含患者级数据。
这里的“完成”表示研究原型与可复现实验链已经打通，不表示企业生产部署、法规认可或临床有效性
验证已经完成。

## 7.2 原型验证总体设计

验证遵循“来源先冻结、预测先于 Gold、案例级隔离、正负对照并存、生成与评分分离”的原则。
完整性测试回答资料是否真实、可读和可复现；角色 Gold 回答文档角色识别；受控缺陷 Gold
回答两个可确定任务；自然候选回答系统能否在真实 PDF 中以召回优先方式形成可裁决队列并正确
保持不确定性；
人工审批实验回答批准后的补丁是否真正正确。不同证据层不能相互冒充。

实验单元按案例划分，而不是把同一研究的不同文档或标签随机拆到不同集合。50 个案例固定分为
30 train、10 calibration 和 10 held-out test；同一 NCT 下的注册快照、Protocol、SAP、case
contract、缺陷标签和修订记录保持同组。calibration 用于阈值或策略选择，test 只用于冻结后的
最终评价。自然候选尚无外部 Gold，因而即使预测文件已经先行冻结，也不能提前把 test 名称当作
准确率证据。

| 证据层 | 数据来源 | 回答的问题 | 可以报告 | 不能报告 |
|---|---|---|---|---|
| 完整性层 | 50 例注册快照与 65 份 PDF | 资料是否真实、可读、可冻结 | 数量、页数、摘要、覆盖 | 模型准确率 |
| 角色层 | 官方 hasProtocol/hasSap | 文档角色是否识别正确 | 角色分类指标 | 临床缺陷能力 |
| 受控缺陷层 | 两类注入缺陷与负对照 | 确定性检查是否稳定 | 混淆矩阵、Wilson 区间 | 自然语义总体性能 |
| 自然候选层 | 真实 PDF 页级线索 | 是否发现并保留不确定性 | 覆盖、路由、定位完整率 | 未裁决前准确率 |
| 修订闭环层 | 模拟批准与沙箱补丁 | 批准后是否改对且不越界 | 正确性、局部性、回归 | 真实专业授权有效性 |

汇总值不是手工填入报告的唯一证据。语料规模来自 `build_report.json` 与 `corpus_profile.json`；
角色结果来自 `role_baseline_results.json`；受控缺陷结果来自 `seeded_defect_results.json`；自然候选
冻结摘要来自 `natural_candidate_evaluation_pre_gold.json`；修订闭环来自
`simulated_human_repair_report.json`。每个汇总文件均可下钻到 case contract、逐条标签、预测或
修订记录，第二遍精修将逐项复算正文表格与这些机器产物的一致性。

## 7.3 NCT04683926 一致性审核案例

公开案例 NCT04683926 / OMNI-PAIN-103 包含 Protocol、SAP、ICF 和注册资料。项目建立 27 条
人工事实基线、正例、合法差异负对照和独立 Gold Scorer。初始规则在严格数值边界、跨来源
时长和合法双时间轴方面存在漏报与误报；增加相应规则后，冻结结果为 TP=8、FP=1、FN=0、
TN=2，Precision=0.8889、Recall=1.0000、F1=0.9412、负对照准确率=1.0000。

额外 FP 来自一条未匹配 Gold 的可报告语义提醒，而不是把两个负对照判错。该结果证明当前
案例上的工程改进，不代表跨疾病、跨文档类型的临床总体准确率。

独立 scorer 的计分单位是预先冻结的 Gold test，而不是模型自行生成的 finding 名称。一个
阳性 Gold 只有在事实、章节、语义谓词和必要证据都匹配时才计为 TP；负对照若被错误报告则
计为 FP；未匹配任何正例的额外可报告 finding 也计为 FP。这样可以防止系统通过改写 finding
名称或输出大量模糊提醒获得虚高 Recall。初始版与整改版对比如下：

| 指标 | 初始版 | 整改版 |
|---|---:|---:|
| TP / FP / FN / TN | 4 / 2 / 4 / 1 | 8 / 1 / 0 / 2 |
| Precision | 0.6667 | 0.8889 |
| Recall | 0.5000 | 1.0000 |
| F1 | 0.5714 | 0.9412 |
| 负对照准确率 | 0.5000 | 1.0000 |

整改新增严格给药间隔、跨来源住院时长、筛选窗口包含边界和来源链完整性规则，并为合法
Protocol/CRF 双时间轴增加 scope-aware 负对照。由此，已知漏报被补齐，同时已知误报被
消除；这比单纯扩大报警数量更能说明规则改进的有效性。

## 7.4 Metformin-PAD 受控生成案例

该工作流以完全合成或历史重建测试包验证分阶段可见性。Phase 1 仅读取初始资料并生成候选
事实、Synopsis 和章节；Phase 2 在冻结前期结果后加入新证据并增量修订；隐藏参考资料只在
生成结束后交给独立评估器。每次运行保存输入与提示摘要、模型、来源边界和 checkpoint。

首次软评分表面较高，但模拟医学、统计和注册审核发现样本量无法复现、estimand 与分析集
不一致、终末事件被不当纳入 MAR 插补、模拟监管材料被权威化等问题。新增机器门后，运行被
标记为 `blocked_machine_gate` 和 `blocked_missing_evidence`。这一失败整改说明系统亮点不在
“生成得像”，而在发现高分不可信后能够阻断并形成角色化整改工单。

机器门将问题编译为结构化整改工单，而不是只输出一段批评。每条工单记录责任角色、受影响
事实和产物、必须执行的修订动作、所需证据及可机器检查的退出条件。例如样本量工单要求统计
人员提供可复现公式或模拟输出，并同步 Synopsis、统计章节和相关登记字段；estimand 工单
要求明确人群、变量、治疗条件、伴发事件策略和汇总量；模拟监管材料工单要求删除“监管已
认可”类无依据表达并恢复资料的合成属性。只有退出条件全部为真，工单才可关闭。

这一案例也验证了分阶段资料隔离：Phase 1 运行不能看到 Phase 2 新证据，Phase 2 只能在
前期摘要冻结后增量处理，Evaluator-only 资料不能回流生成上下文。每次运行保存 checkpoint
digest，因此可以判断后续结果是合法增量还是历史输入被静默改变。

## 7.5 50 例真实公开语料与文档角色 Gold

项目通过 ClinicalTrials.gov API v2 冻结 50 个真实研究、50 份注册快照和 65 份官方 PDF，
共 3,025 页。48 个为干预性研究、2 个为观察性研究；35 个案例使用合并 Protocol/SAP，
15 个使用分立文件。入选文件均通过 PDF 头、非空页数和正文可提取检查，URL、页数、字节数
和 SHA-256 均可复核。

官方 `hasProtocol/hasSap` 元数据形成 130 个二元角色标签，50 个案例按 30/10/10 分为
train、calibration 和 held-out test。测试集规范文件名基线在 28 个标签上得到 100% 点估计，
Wilson 95% 区间为 87.94%—100%。但正文关键词基线在同一测试集上仅得到 TP=19、FP=4、FN=1、
TN=4，Accuracy=82.14%、F1=88.37%；把两者做简单 OR 的 hybrid 基线 Accuracy=85.71%、
F1=90.91%。因此 100% 结果说明官方文件命名和元数据映射在这批冻结资料上稳定，不说明自由文本
语义分类已经解决，更不是临床缺陷检测性能。

语料构建不是下载前 50 个搜索结果。构建器逐项核对官方文档元数据、文件头、页数、正文

Protocol/SAP 角色，导致合并文档假阴性；整改后改为官方元数据定义角色、正文关键词仅作

## 7.6 200 个受控缺陷 Gold 与检测结果

在 50 个真实案例上派生两类可确定缺陷：canonical NCT identifier 冲突和 enrollment 变更
未传播。每例每类各有一个正例和匹配负对照，共 200 个标签，正负各 100。标识符负对照包含
正确 NCT 和作为相关研究引用的另一 NCT；enrollment 负对照包含已经正确传播的新值，因此
不是容易通过的空文本。

Held-out test 的 10 个案例、40 个标签上得到 TP=20、FP=0、FN=0、TN=20，Precision、
Recall、F1 和 Accuracy 点估计均为 100%，Accuracy Wilson 95% 区间为 91.24%—100%。全量
200 标签判断正确，区间为 98.12%—100%。结论严格限定为两类受控确定性任务。

Wilson 区间用于避免在分母较小时把 100% 点估计解释成确定规律。对成功数 $k$、样本数 $n$
和正态分位数 $z$，区间中心与半宽为

$$
center=\frac{\hat p+z^2/(2n)}{1+z^2/n},
$$

$$
half=\frac{z}{1+z^2/n}
\sqrt{\frac{\hat p(1-\hat p)}{n}+\frac{z^2}{4n^2}},
\qquad \hat p=k/n.
$$

因此 40/40 的下界仍只有约 91.24%，200/200 的下界约 98.12%。报告区间的目的不是削弱
结果，而是明确现有样本对总体结论提供了多强的证据。

这一层的“真实”仅指底层 case contract 和来源来自真实公开研究；缺陷本身是按预定义规则受控
注入的，标签由注入操作确定，而不是从自然临床文本中由专家发现。两类任务的判定边界与检测器
高度一致，适合验证标识符 scope、事实传播、负对照保护和回归稳定性，不适合证明大模型语义推理
能力。把这一限制与 7.7 的未裁决自然候选并列报告，正是为了避免用容易复现的确定性任务代替困难
的自然语义评价。

## 7.7 真实自然候选与不确定性处理

系统从 65 份真实 PDF 中按召回优先策略挖掘 1,393 条自然差异候选，再按案例与类别分层选择
上下文质量较高的候选，形成 197 条、
48 个案例、8 个类别的队列，覆盖分析人群、缺失数据、多重性、主要终点时间、终末事件、
试验标识符和入组范围等。每条具有文件名、页码、短原文和所需角色。

数字差异可能表示计划/实际、筛选/随机化、总样本/亚组；另一 NCT 可能只是相关研究引用。
因此系统没有自行制造 Gold：197 条均保持 `candidate_not_gold`，0 条自动确认，197 条全部
路由专业复核。裁决前预测摘要已冻结。由于当前没有时间进行高成本外部裁决，本次不计算这
部分的 Precision、Recall、F1 或 Accuracy；在没有对漏掉的真实问题进行 Gold 标注前，也不能
计算或声称候选挖掘 Recall。当前可报告的只有队列规模、类别覆盖、路由行为和来源定位完整性，
并把该队列作为后续 POC 资产。

候选分布为：analysis population 30 条、enrollment scope/state 43 条、estimand/intercurrent
event 2 条、missing data 27 条、multiplicity 10 条、primary endpoint/time 39 条、terminal
event 41 条、trial identifier scope 5 条。系统冻结 disposition 中，43 条属于入组范围或
状态不明确，69 条需要统计语义裁决，80 条需要医学与统计上下文，5 条需要 NCT 范围复核。
所有 197 条都有文件名和页码，来源定位完整率为 100%。

这组数据的价值不是提供一个新高分，而是暴露真实文本中的困难分布：同一数字可能代表不同
人群和时间状态，“death”可能是 SAE 定义而非 estimand 缺陷，另一个 NCT 可能是相关研究，
“intention-to-treat”出现也不自动证明分析集矛盾。系统选择 defer 而不是 auto-confirm，体现
了不确定性治理在自然数据上的实际作用。

## 7.8 模拟人工批准后的修订闭环

项目在 50 例上构造 200 个“发现—决定—修订—复检”场景，模拟统计或数据治理角色批准
100 个缺陷、驳回 100 个负对照。在 100 个模拟批准缺陷中，50 个 enrollment 问题具备安全
确定性补丁，另外 50 个 NCT ID 问题需要人工编辑。因此闭环成功率的预定义分母是 50 个
`auto_repair_available` 场景，而不是全部 100 个已批准缺陷。对这 50 个场景，50/50 新值
写入、旧值移除、仅改变目标章节、来源链保持、finding 关闭，新增 finding 为 0；100 个负
对照均未变化。成功率点估计为 100%，Wilson 95% 区间为 92.87%—100%。

另外 50 个 NCT ID 冲突虽获“需要处理”的模拟批准，系统仍拒绝未经证明的字符串替换并转
人工编辑。它们不是修订失败样本，而是按预定义安全策略正确进入 `manual_edit_required`；若把
它们排除在分母之外却不说明原因，就会形成选择性报告，因此本节同时报告 50 个可自动修订和
50 个必须人工编辑的处置。该结果验证授权与执行能力是两个不同门槛。模拟角色只验证流程机制，
不构成真实医学、统计或注册批准。

每条闭环记录同时保存 before/after digest、finding ID、提案数、改变章节、规范值是否出现、
旧值是否消失、来源链是否保持、复检后剩余 finding 和新增 finding。验收不是只比较最终
字符串，也不是只检查原报警消失。50 个可修订场景的 `semantic_patch_correct`、
`minimal_scope_patch`、`source_trace_preserved` 和 `finding_closed_after_patch` 均为真；
`new_finding_after_patch_count`、`negative_control_changed_count` 和 `unexpected_change_count`
均为 0。held-out test 的 10 个可修订问题同样全部闭合。

## 7.9 失败案例与机器门禁整改

验证过程中修复了权威状态与摘要漂移、0 finding 的 vacuous pass、软评分掩盖硬失败、固定
关键词误作角色真值、过度并发取证和只检查报警消失而不检查补丁语义等问题。当前摘要由权威
状态重新物化；未闭合 finding 进入决策队列；角色 Gold 使用官方元数据；补丁复检同时检查
新旧值、范围、来源链和新增问题。报告保留这些失败，是为了证明系统能够从暴露的问题形成
可执行整改，而非只展示成功案例。

| 暴露问题 | 风险 | 整改 | 回归证据 |
|---|---|---|---|
| state 与 summary 漂移 | 旧摘要误导为可发布 | 摘要由权威状态重新物化 | assurance 一致性测试 |
