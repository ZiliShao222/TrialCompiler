## 第六部分 技术方案、系统架构与正确性机制

### 6.1 总体分层技术架构

TrialCompiler 采用“飞书 Aily 协同入口—TrialCompiler 智能处理后端—企业正式质量体系”
的三层架构。飞书 Aily 负责资料接入、任务发起、缺失信息补问、事实确认卡展示、任务分派、
状态跟踪和审批留痕；TrialCompiler 后端负责来源登记、候选事实提取、文档生成、依赖计算、
一致性检查、候选修订、沙箱复检和审核材料生成；企业既有 DMS 或质量体系继续承担电子签名、
正式发布、归档和受控版本管理。三层职责彼此分离，智能处理模块不能绕过正式质量体系直接
发布临床文件。

后端以版本化事实和文档依赖图为核心，而不是把方案仅视为一份静态 Word。系统中的主要
对象包括来源 `SourceRef`、事实 `FactRecord`、文档单元 `DocumentSection`、审查发现
`ReviewFinding`、修订提案 `RepairProposal`、人工决定请求 `DecisionRequest` 和审计记录。
多角色 Agent 围绕这些共享对象协作，避免不同 Agent 各自维护不可核对的自然语言状态。

### 6.2 受控数据流

一次任务的数据流依次经过：任务边界确认、来源登记、候选事实提取、专业人员确认、事实
基线建立、文档生成或审核、依赖影响计算、候选修订、独立质量检查、人工决定和正式系统
衔接。原始资料、候选事实、已批准事实和评测金标准采用不同可见性边界。Gold labels 和
隐藏参考资料仅供独立评测，不作为待评模型生成答案时的输入，从而降低答案泄漏和循环评价。

所有 AI 输出默认保持候选状态。来源冲突、范围不清、证据不足或专业含义无法唯一确定的
内容进入待复核队列，不以模型置信度自动补全。候选修订首先应用于沙箱副本，只有在目标
问题关闭、未引入新问题、来源链保持完整且相应人工审核完成后，才具备进入后续受控流程的
条件。

### 6.3 来源登记与证据追溯

每份输入资料记录来源标识、标题、定位地址、版本、权威等级、适用范围和有效日期。事实不
只保存一个文本值，还保存来源列表、状态、责任角色、版本、更新时间和前一版本值。文档章节
同时记录所引用的事实和证据来源，使审核人员能够从发现回到章节、从章节回到事实、再从事实
回到原始证据。

系统将“能够给出答案”与“能够证明答案来自哪里”作为两个不同条件。缺少来源定位的候选
内容不能因语言流畅或模型高分而自动成为已批准事实。当前公开语料已经保存官方 URL、文件
字节数、页数和 SHA-256；自然候选还保存真实 PDF 文件名、页码和短原文。后续企业 POC
需要继续将定位精度扩展至章节、表格和单元格级。

### 6.4 Trial Fact Sheet 事实基线

Trial Fact Sheet 用于统一管理研究目标、终点、样本量、研究人群、入排标准、给药方案、
访视安排、评估时间和统计原则等关键设计事实。事实状态至少区分草稿、候选变更、需要人工
复核、已批准、已拒绝和已替代。状态迁移由证据和人工职责共同约束，模型概率不能直接把
`requires_human_review` 改为 `approved`。

当事实发生变化时，系统保留旧值、新值、版本和变更理由，而不是静默覆盖原值。所有依赖
该事实的章节和关联文件通过事实标识连接到同一基线，为跨文件一致性检查和增量传播提供
可复现输入。

### 6.5 临床文档依赖图与影响计算

临床文档依赖图可表示为二部图：一侧为事实节点，另一侧为方案章节、SoA、SAP、ICF、CRF
和注册记录等文档单元；边表示引用或语义依赖关系。当事实 `f` 发生变化时，影响集合
`I(f)` 由与该事实相连的文档单元构成。系统先计算影响集合，再对局部内容生成候选修订，
避免对整份文档进行无差别重写。

影响清单可区分直接更新、可能更新、需要专家判断和仅保留历史四种处置。对于样本量、数值
边界、时间点、试验标识符和旧值残留等可形式化关系，系统运行确定性检查；对于 estimand、
缺失数据、多重性、医学安全性和注册策略等语义关系，系统输出证据和不确定性并路由给相应
专业人员，而不自行宣布专业结论。

### 6.6 大模型、Agent 与确定性规则协同

大模型适合资料理解、自然语言归纳、候选文本生成和复杂语义线索发现，但存在幻觉、自洽性
偏差和置信度失真。确定性规则适合标识符、数字、日期、严格边界、来源完整性、旧值传播和
状态约束等可形式化问题。TrialCompiler 将二者组合：大模型扩大语义覆盖，规则提供稳定、
可重复的硬检查，独立 scorer 和人工复核对输出进行外部约束。

多 Agent 的作用不是简单增加模型调用次数，而是分离证据整理、候选生成、缺陷扫描、修订
建议、独立复核和决策请求等职责。生成 Agent 不能同时担任最终评分者或批准者；语言模型
给出的高分也不能抵消机器硬门发现的阻断问题。

### 6.7 不确定性表示与主动取证

对于存在多个可能解释的候选问题，系统维护有限假设集合上的 belief state，并将不确定性
显式表示为后验分布或熵。可选动作不只包括“回答”，还包括获取特定资料、请求特定角色
判断、暂缓处理或在证据充分时提交。系统以预期信息增益与获取成本共同评价取证动作，从而
优先寻找最可能减少决策不确定性的证据。

相关评估同时记录 Brier score、ECE、risk–coverage/AURC、成对排序准确率，以及移除或
替换关键证据后的 necessity flip rate 和 contrastive sensitivity。校准集和测试集必须按
案例隔离；如果 split 泄漏、未知 observation 或冻结摘要漂移，评估器失败关闭，不允许
输出校准声明。当前仓库已经打通这一实验链路，但尚无足够的真实专家轨迹支持对主动取证
策略作总体性能声明。

### 6.8 原子修订、人工授权与整改闭环

候选修订遵循最小修改原则。一个修订提案必须关联触发 finding、目标章节、原文、候选文本、
事实标识、证据来源和修订理由。多个提案发生重叠或冲突时，系统不能静默合并，而应转入
人工判断。所有补丁先在沙箱中执行，并复查目标 finding 是否关闭、是否出现新 finding、
负对照是否受影响、来源引用是否保持以及修改范围是否超出提案。

人工批准是进入整改的必要条件，但不代表系统可以执行任意自动修改。对于能够由规范事实
唯一决定的旧值传播，系统可在批准后生成最小补丁；对于试验身份、医学含义或范围归属仍
存在多种解释的问题，即使批准“需要处理”，系统仍返回人工编辑要求。固定闭环为：合格人员
决定—事实调和—关联内容传播—确定性复检—独立质量审核。

### 6.9 独立质量门与数学正确性说明

系统采用硬门禁对软评分的非补偿性约束。设软质量分为 `S(x)`，阻断性约束集合为
`G={g1,...,gm}`；只有当所有 `gi(x)=1`、人工授权状态满足要求且审计信息完整时，候选项
才可能进入下一阶段。无论 `S(x)` 多高，只要任一硬约束失败，发布授权均为假。因此语言
模型评分不能覆盖来源缺失、事实冲突、未决专业判断或回归失败。

数学技术证明覆盖来源可追溯、状态单调性、影响传播完备性的条件化表述、原子补丁局部性、
硬门禁安全性、belief 更新和选择性风险等工程性质。证明结论仅在已编码规则、已登记依赖和
明确假设范围内成立，不声称形式化覆盖全部医学、统计和监管判断。

### 6.10 私有化部署及系统对接路径

当前原型提供命令行和 FastAPI 接口，可与飞书 Aily、企业文档系统和质量流程逐步对接。
正式部署需要配置项目与角色双层最小权限、传输和存储加密、密钥外置、访问日志、数据保留
期限、异常访问检测、备份恢复和模型调用边界。API 密钥不得写入仓库、报告或运行产物。

建议先在非生产或历史项目中开展 POC，使用脱敏或公开资料验证准确性、审核时间和流程适配，
再根据预设退出条件逐步扩围。当前原型不承诺已经完成企业级私有化、安全合规或真实患者
数据处理验证。

## 第七部分 原型验证、实验结果与可复现性

### 7.1 当前原型完成情况

仓库已经实现来源与事实数据契约、Trial Fact Sheet、文档依赖图、A–H 多角色审核流程、
受控方案生成、确定性检查、语义审核、原子修订、沙箱复检、人工决策请求、审计记录、
不确定性实验器、独立 benchmark scorer、命令行工具和 FastAPI 服务。当前自动化测试结果
为 `130 passed`，GitHub Actions 对最终提交验证成功。

### 7.2 原型验证总体设计

验证采用公开资料或完全合成材料，不使用患者级数据。评价遵循四项原则：测试输入与 Gold
隔离；按研究案例划分 train、calibration 和 held-out test，避免同一研究跨 split；同时
设置正例、负对照和边界例；生成模块不能自行宣布正确，结果由独立 scorer 或确定性断言
计算。报告同时披露分母、置信区间和适用范围，避免用单个高分替代整体证据。

### 7.3 NCT04683926 一致性审核案例

第一条工作流使用公开研究 NCT04683926 / OMNI-PAIN-103 的 Protocol、SAP、ICF 和注册资料，
建立 27 条人工整理的事实基线、跨文件一致性 Gold tests、合法差异负对照和独立评分器。
系统识别数值边界、研究范围、时间点、执行要求、来源链和事实变更传播问题，并生成影响矩阵、
候选修订、复检结果和人工审核包。

最新冻结结果为 TP=8、FP=1、FN=0、TN=2，Precision=0.8889、Recall=1.0000、F1=0.9412，
负对照准确率=1.0000。该结果优于历史基线，但只代表当前案例和 Gold/scorer 版本，不能外推
为临床生产准确率。

### 7.4 Metformin-PAD 受控生成案例

第二条工作流使用完全合成或历史重建的 Metformin-PAD 测试包。Phase 1 仅读取初始可见资料，
Phase 2 在冻结前期结果后加入新证据并执行增量修订，隐藏参考资料只在生成冻结后交给评估器。
该流程能够生成 Synopsis、九个方案章节和关联产物，并记录输入、提示、模型、来源边界和
checkpoint。

首次语言模型盲评曾给出较高分，但模拟医学、统计和注册质量复核发现样本量、estimand、
终末事件、证据依据和模拟监管资料权威化等严重问题。新增确定性机器门后，运行被正确标记为
`blocked_machine_gate` 和 `blocked_missing_evidence`，说明系统不会用语言流畅度或软评分
掩盖基础矛盾。模拟审核不构成真实专业批准。

### 7.5 50 例真实公开 Protocol+SAP 语料

为降低单案例偏差，项目通过 ClinicalTrials.gov API v2 冻结 50 个真实公开研究、50 份注册
快照和 65 份官方上传 PDF，共 3,025 页。35 个案例使用合并 Protocol/SAP，15 个使用分立
文档；48 个为干预性研究，2 个为观察性研究。所有入选 PDF 均通过文件头、非空页数和正文
可提取性检查，文件 URL、字节数、页数和 SHA-256 均已保存。

文档角色 Gold 直接来自官方 `hasProtocol/hasSap` 元数据，共 130 个二元标签。50 个案例按
30/10/10 分为 train、calibration 和 test，不发生案例泄漏。Held-out test 上，规范文件名
基线在当前 28 个角色标签中得到 100% 点估计，Accuracy Wilson 95% 区间为 87.94%–100%；
该指标只表示公开文档角色识别，不是临床缺陷检测准确率。

### 7.6 200 个受控缺陷 Gold 与核心检测结果

在 50 个真实案例上派生两类可确定裁决缺陷：canonical NCT identifier 冲突和 enrollment
候选变更未传播。每例每类各设置一个正例与一个匹配负对照，共 200 个标签，其中正例 100、
负对照 100。Held-out test 包含 10 个案例和 40 个标签，得到 TP=20、FP=0、FN=0、TN=20，
Precision、Recall、F1 和 Accuracy 点估计均为 100%，Accuracy Wilson 95% 区间为
91.24%–100%。全量 200 标签全部判断正确，Accuracy Wilson 95% 区间为 98.12%–100%。

这一结果支持“系统在两类受控确定性缺陷上表现稳定”，但不能外推到 estimand、缺失数据、
多重性、医学安全性和注册策略等自然语义缺陷。

### 7.7 真实自然候选与裁决前冻结

系统进一步解析 65 份真实 PDF，高召回挖掘 1,393 条自然差异候选，覆盖 analysis population、
missing data、multiplicity、primary endpoint/time、terminal event、trial identifier scope
和 enrollment scope/state 等类别。经案例与类别平衡后形成 197 条、48 个案例、8 个类别的
多元裁决集。

大量数字差异可能分别表示计划值与实际值、筛选人数与随机人数、总样本与亚组，其他 NCT ID
也可能只是相关研究引用。因此这 197 条记录保持 `candidate_not_gold`，不能由系统自标为
缺陷。裁决前预测已经冻结：197/197 条均具有来源文件和页码，0 条被自动确认，197 条全部
进入合格人员复核。预测摘要 SHA-256 为
`59E16BEAEB7B6C07ABD9F0601300CC96EF4F41D3AB890F41DE94F1DC7F239FA2`。

### 7.8 模拟人工批准后的修订闭环

为验证系统能否在发现缺陷后真正完成正确修订，项目在 50 例上执行 200 个“发现—人工决定—
沙箱修订—重新检查”场景。模拟统计或数据治理角色批准 100 个缺陷、驳回 100 个负对照。
其中 50 个 enrollment 旧值问题具备确定性补丁，50/50 个补丁均正确写入新值、移除旧值、
只改变目标章节、保持事实与来源链、关闭原 finding，且没有引入新 finding。修订成功率点估计
为 100%，Wilson 95% 区间为 92.87%–100%；100 个负对照均未改变。

另外 50 个 NCT identifier 冲突虽然获得“需要处理”的模拟批准，但修订器没有执行未经证明
的字符串替换，而是返回 `approved_but_manual_edit_required`。因为该冲突可能表示文件身份
错误、相关研究引用或来源范围错误，必须由人工决定具体编辑方式。这证明人工批准是进入整改
的必要条件，但不能绕过自动修订的安全白名单。上述人工角色为受控模拟，不构成真实医学、
统计、注册或质量授权。

### 7.9 失败案例、机器门禁整改与当前局限

原型验证暴露并修复了摘要状态漂移、软评分掩盖硬失败、固定关键词误作角色真值、候选下载
过度并发和修订后验证不足等问题。当前摘要由权威状态重新物化；未闭合 finding 必须进入
决策队列；官方元数据承担文档角色 Gold，正文关键词仅用于审计；构建器按批取证并在达到
目标后停止；补丁验证同时检查新值、旧值、修改范围、来源链和新增 finding。

仍未完成的工作包括：197 条自然候选的外部合格专家裁决；跨治疗领域和跨语言验证；真实
Agent 轨迹的校准；企业模板、SOP 和权限适配；生产性能、安全和合规验证；真实业务工时与
ROI 对照。因此不得声称系统已经通过临床验证、能够替代专业人员或能够自动发布正式方案。

扩围的最低条件应包括：阻断性问题零漏放行；正式文件零静默覆盖；来源链和审计记录完整；
未批准事实不进入正式流程；错误修订率和人工审核时间达到预注册阈值；数据权限、安全和回滚
演练通过。任何一项硬条件未满足，系统保持研究或受控试点状态，不进入生产发布。

#### TrialCompiler 技术分析与原理研究笔记

##### 核心研究主题：临床文档推理中的不确定性与可解释性

原型验证表明，如果只强调文档图、规则、状态机、审计日志和多代理编排，TrialCompiler 容易被理解为一个软件工程系统，AI 只是可替换的文本接口。因此，本项目将核心问题重新明确为：**概率型模型如何在多文档、证据不完整且可能相互冲突的临床语境中主动形成语义假设，并把不确定性解释到足以决定下一步动作。**

文献复核后，本项目不再把“证据缺失、来源冲突、语义歧义、模型分歧、影响未知、权限不足”直接称为六类 Agent 不确定性。这些更准确地说是可观察的诊断信号或治理状态。Agent UQ 的基本分解仍需区分可通过知识与推理约减的 epistemic uncertainty 和环境本身随机性的 aleatoric uncertainty；同时还要说明不确定性位于外部 observation、内部 belief state、下一步 action，还是整条 trajectory 的最终成功事件。

因此，研究对象被收窄为“跨文档事实调和 Agent 中，不确定性驱动的证据获取与忠实补丁解释”。潜在状态是当前应当生效的规范事实，Protocol、SAP、ICF、注册记录、工具结果与专业人员反馈是具有不同权威、时效和噪声的 observation。系统维护对候选事实解释的有限 belief distribution，以期望熵下降减去取证成本来比较下一项证据动作；如果选择性风险不能降到预设门槛，则拒绝自动提交补丁。

这里也严格区分 provenance、observability 和 explainability。来源引用只能证明模型“看到了什么”，轨迹日志只能证明“发生了什么”，都不能自动证明模型给出的理由忠实。候选解释需要接受证据移除或替换后的重放测试：如果模型声称某来源决定了 defer/patch 选择，移除或用冲突来源替换它应当产生可预测的行为变化。该测试最多支持 behavioral counterfactual faithfulness，不声称解释了模型内部机制。

本项目由此提出 **Proof-Carrying Clinical Compilation（携证临床文档编译）**。AI 推理平面负责规则难以覆盖的跨文档语义等价、隐含冲突、主动取证、多解释比较和不确定性估计；确定性编译平面检查来源权威、事实溯源、影响闭包、最小编辑、沙箱反证、负对照与人工授权。模型不能只交付一段像答案的文字，而必须交付可证伪主张、证据、候选补丁及剩余不确定性。

编译器最终生成 `trialcompiler.assurance/v1` 携证证明。证明可以给出“机器结果可进入合格人员审批”，也可以给出“安全阻断并等待专业处置”。无法自证的推理不会被隐藏，而会成为显式的专业决策债务。因此，多智能体、RAG、规则和审计是安全底座；真正的研究亮点是**把概率型 AI 的不确定推理编译成可解释、可核验、可阻断和可回滚的临床变更证明**。

##### 摘要

TrialCompiler 是一个面向临床试验方案及其关联文件的受控文档工程原型。它试图解决的核心问题并不是“让大模型自动写一份 Protocol”，而是：同一个试验设计事实会重复出现在 Protocol、Schedule of Activities（SoA）、Statistical Analysis Plan（SAP）、Informed Consent Form（ICF）、Case Report Form（CRF）以及注册记录中；当一个关键事实被修改时，人工很难完整识别所有受影响位置，也很难证明每一次修改均有来源、有版本、有责任人并经过适当审核。

系统因此将临床文档处理重构为“事实层—依赖图—编译检查—候选修订—质量门—人工批准”的工程流程。大语言模型负责语义理解和候选文本生成，确定性规则负责可复现检查，专业人员保留医学、统计、注册和质量决策权。当前仓库已实现既有文档审核与证据驱动方案生成两条工作流，并在公开或完全合成案例上完成端到端验证；它仍是研究原型，不用于真实患者数据或临床生产决策。

##### 零、方法来源与问题演化：从目标对齐到临床文档编译

TrialCompiler 并不是从“生成一份临床试验方案”这个单点功能开始的。项目早期关注的是一个更一般的协作问题：当团队使用 AI 辅助完成复杂任务时，模型往往能够流畅完成局部文本，却容易逐步偏离项目真正需要服务的全局目标。也就是说，AI 可能会把一个活动写得很完整、把一段文本改得很漂亮，但这些局部产物未必真的支持原始目标、利益相关方需求、证据边界或后续执行。

因此，早期方法验证首先把任务抽象为一个“目标约束下的状态转换”问题：

```text
粗糙任务描述
        ↓
确认全局目标与边界
        ↓
检查当前方案是否偏离目标、证据或约束
        ↓
提出最小必要修改
        ↓
由独立质量门判断修改是否真正解决问题
        ↓
将可复用经验写入受治理记忆
```

这一阶段的重点不是某一种具体场景，而是验证一种通用控制思想：AI 不应只输出“看起来合理”的文本，而应始终围绕被确认的全局目标、证据来源、约束条件和人工审核结果进行工作。这个思想后来被保留下来，并迁移到临床文档场景中，形成 TrialCompiler 当前的核心结构。

早期目标对齐实验为当前系统留下了七个关键设计遗产：

1. **全局目标记忆**：先确认任务真正服务的目标和边界，再处理具体文本。
2. **角色分离的代理结构**：目标确认、缺陷扫描、候选修订、质量判断和报告生成由不同角色承担，避免单个模型同时“写、改、评自己”。
3. **缺陷类型体系**：把问题拆成可复查的类型，而不是只给笼统评价。
4. **修复循环**：当质量门认为候选修改不足时，系统返回修订步骤继续迭代。
5. **独立质量门**：最终产物必须经过外部规则或独立评审，而不是依赖生成模型自信程度。
6. **受治理记忆**：有价值的经验只能作为候选知识沉淀，并保留来源、适用范围和人工确认状态。
7. **人工决策边界**：当证据不足、目标冲突或风险较高时，系统应提出决策请求，而不是擅自确定答案。

进入临床试验方案场景后，这些抽象对象被重新映射为更严格的文档工程对象：

| 通用目标对齐概念 | TrialCompiler 中的临床文档对象 |
|---|---|
| 全局目标与边界 | 研究设计目标、适应证、文档类型、监管路径和项目约束 |
| 局部方案或活动草案 | Protocol 章节、SoA 表格、SAP 表述、ICF 文本、CRF 字段和注册记录 |
| 目标偏离或缺陷 | 跨章节不一致、无来源事实、旧值残留、统计/监管风险、执行边界不清 |
| 候选修复 | 带事实锚点、来源证据和影响范围的候选红线或修订建议 |
| 独立质量门 | 确定性规则、语义审核、benchmark scorer、硬失败规则和人工批准 |
| 经验记忆 | 按企业通用、治疗领域、试验阶段、文档类型和项目专属范围治理的审核规则 |

因此，TrialCompiler 可以被理解为同一套方法论在高风险临床文档中的工程化落地：早期解决的是“AI 如何不偏离团队目标”，当前解决的是“临床试验文档如何不偏离经确认的研究事实、监管约束和版本依赖”。两者在数学上都是受约束的状态更新问题，只是当前场景要求更强的来源追踪、版本治理、可复现检查和人工质量门。

这个演化过程也解释了为什么 TrialCompiler 不是普通的 AI 写作助手。普通写作助手的核心动作是生成文本；TrialCompiler 的核心动作是维护一个可审计的项目状态，并在每次生成、审核和修订时检查状态是否仍与事实、规则、依赖和人工确认保持一致。

##### 一、问题定义：为什么临床文档需要“编译”

###### 1.1 临床方案不是一篇孤立文章

一个临床试验的关键设计事实通常会跨文件传播。例如，主要终点的定义和访视时间不仅出现在 Protocol 的目标与终点章节，还可能出现在 Synopsis、SoA、SAP、CRF 字段和注册页面。样本量、入排标准、给药日、PK 采样时间、安全性规则等也具有相同特征。

传统文档流程通常以文件和章节为中心：不同专业人员分别编辑各自负责的 Word、Excel 或系统页面，依赖关系主要保存在人的经验、邮件和会议记录中。由此产生四类技术风险：

1. **事实重复**：同一事实以不同措辞和格式重复出现，难以建立唯一来源。
2. **依赖隐形**：修改一个事实时，无法自动得知它影响哪些章节、表格和关联文件。
3. **规则分散**：医学、统计、注册和质量规则分布在指南、模板、SOP 与专家经验中。
4. **版本并行**：多个文件同步修订，容易保留旧值或形成相互冲突的版本。

###### 1.2 “编译器”类比的含义

TrialCompiler 借鉴编译器思想，但并不声称临床文档与程序代码完全相同。其映射关系为：

| 编译器概念 | TrialCompiler 中的对应对象 |
|---|---|
| 源代码 | Protocol、SoA、SAP、ICF、CRF、注册信息及来源材料 |
| 中间表示 | 版本化 Trial Fact Sheet 与 Clinical Document Graph |
| 类型/语法检查 | 字段完整性、标识符、单位、数值范围和引用完整性检查 |
| 语义检查 | 跨章节、跨文件的定义、范围、时间轴和人群一致性审核 |
| 依赖分析 | 事实到章节、表格、字段和文件的影响传播 |
| 编译错误/警告 | 冲突、漏改、无来源声明、边界差异和人工决策请求 |
| 补丁 | 带事实锚点和来源证据的最小候选红线 |
| 测试与发布门 | benchmark、回归检查、机器门禁与专业人员批准 |

这里最重要的不是比喻本身，而是工作方式的变化：系统先把重复文本还原为可治理事实和依赖，再进行检查与修改，而不是直接让模型对整份文件进行黑箱式重写。

##### 二、总体技术架构

###### 2.1 分层架构

当前原型可分为六层：

1. **交互与接入层**：CLI、FastAPI 和飞书 Aily intake contract，负责接收项目、文档、事实变更和审核请求。
2. **来源与知识层**：登记公开资料、项目资料、法规索引、事实来源和版本信息；不同可见性阶段的数据被隔离。
3. **结构化表示层**：将来源、事实、章节和依赖关系表示为类型化数据对象。
4. **编译与代理层**：执行确定性检查、语义审核、影响分析、候选修订和质量复核。
5. **治理与审计层**：保存输入摘要、模型、提示词摘要、版本、发现、修订、人工决策请求与运行轨迹。
6. **评测层**：使用公开或合成 benchmark、人工金标准、负对照、失败案例和独立 scorer 评价系统，而不是让生成模型自评。

###### 2.2 核心数据对象

当前代码用显式数据契约描述以下对象：

- `SourceRef`：来源编号、标题、定位地址、权威级别、版本和使用范围。
- `FactRecord`：事实编号、名称、值、单位、状态、来源、责任角色和版本。
- `DocumentSection`：章节编号、文档类型、正文、事实引用和来源引用。
- `ReviewFinding`：问题类型、严重度、影响章节、事实锚点、证据来源和是否需要人工审核。
- `RepairProposal`：原文、候选文本、修订理由、事实与证据链，以及当前治理状态。
- `DecisionRequest`：当证据不能唯一确定安全修改时，向合格专业人员提出的结构化决策请求。

这些对象构成系统的“中间表示”。大模型的输出必须经过字段和引用治理后才能进入共享状态，不能直接覆盖正式文档。

###### 2.3 Clinical Document Graph

Clinical Document Graph 以事实为中心建立显式依赖关系：

```text
来源文件/位置
    ↓ 支持
Trial Fact F001...Fn
    ↓ 被引用
Protocol章节 / SoA表格 / SAP规则 / ICF表述 / CRF字段 / 注册记录
    ↓ 发生变更时
影响集合 + 漏改检查 + 候选红线 + 人工决策请求
```

对事实 `f`，系统维护其影响集合：

```text
Impact(f) = {所有引用 f 的章节、表格、字段和文件}
```

当事实从旧值 `v_old` 变为候选值 `v_new` 时，系统扫描 `Impact(f)` 内是否仍出现旧值，生成 `proposed_fact_change_not_propagated` finding。旧值不会被静默覆盖，而是作为 `previous_value` 保留；候选事实必须处于 `proposed_change` 或人工审核状态。

##### 三、既有文档审核工作流

###### 3.1 A–H 角色与治理节点分离

当前审核工作流将专业角色、不确定性治理与受控取证分离为八个节点：

```text
A Context Lock
→ B Evidence Review
→ C Repair Proposal
→ D Independent Quality Gate
→ G Uncertainty Governance
→ H Governed Evidence Acquisition（仅在 G 选择取证且预算允许时）
→ B Evidence Review（重新进入 B/C/D/G）
→ E Review Packet
→ F Experience Candidate
→ qualified human approval
```

- **A：上下文锁定**。锁定项目、文档类型、适用范围、已批准事实和待决事实，默认阻止正式发布。
- **B：证据与问题识别**。合并确定性检查与受治理的语义审核结果，并只检索范围匹配、状态允许的经验。
- **C：候选修订**。围绕已批准事实或受控候选变更生成最小修订；无事实锚点或依赖未批准事实的语义修订会被阻断。
- **D：独立质量门**。在沙箱副本中应用候选补丁，重新运行确定性检查，识别未关闭问题和新回归。
- **G：不确定性治理**。把 B/C/D 的诊断信号、证据来源、质量结果和决策债务编译为标准轨迹产物，选择 commit/acquire/deliberate/defer 下一动作；没有拟合校准器时数值概率保持为空，并禁止校准声明。
- **H：受控取证**。只消费预先批准、项目与文档范围匹配、类型允许且 SHA-256 摘要一致的 observation；在硬预算内成功获取后重新执行 B/C/D/G。篡改、跨范围、未批准或不支持的 observation 记录拒绝原因并失败关闭，不允许模型自行创造证据。
- **E：审核包**。输出发现、影响范围、候选修订、来源、未决事项和责任边界。
- **F：经验候选**。运行经验不会自动进入组织记忆，只有专业人员批准后才可复用。

###### 3.2 确定性规则与语义模型协作

系统采用“规则优先、模型补充、人工裁决”的混合方法。

确定性规则适合处理：

- NCT 标识符不一致；
- Week、day、hour 等明确时间值冲突；
- `>3 days` 与恰好 `3 days` 等严格数值边界；
- 已提出变更但关联章节仍保留旧值；
- 事实、章节和来源引用缺失；
- 来源定位和事实状态完整性。

大模型适合处理：

- 相同概念使用不同自然语言表述；
- 人群定义、统计范围和文档职责之间的语义差异；
- 单凭正则表达式无法可靠判断的跨文件矛盾；
- 需要专业人员进一步确认的问题及候选解释。

为了防止模型误报，系统还设置负对照。例如 Protocol 使用连续 Day -1 至 Day 11，而 CRF/SAP 按四个 Period 分别记录 Day 1，是一种可解释的合法时间轴映射；当两个时间轴事实已经分别批准时，系统不会仅因表示方式不同就自动报告冲突。

###### 3.3 原子化修订与冲突合并

系统不允许多个代理直接覆盖同一段文本。候选修改先被转换为带起止位置、原文、替换文本、事实和证据的原子操作。若多个操作修改不同区间，可以合并；若操作区间重叠且证据无法确定唯一选择，则不强行选一个，而是转化为 `DecisionRequest`，由合格专业人员决定。

这种设计避免了两个常见问题：一是后运行的代理覆盖前一个代理的修订；二是模型在证据不足时用“看起来合理”的措辞自行填补专业决策。

##### 四、证据驱动的方案生成工作流

###### 4.1 为什么生成与审核必须分开

既有文档审核的输入是已经存在的 Protocol、SAP、ICF 等文件；全新方案生成的输入则是疾病背景、药物证据、申办方约束和模拟沟通材料。两者的状态、风险和验收方式不同，因此不能使用同一条代理链简单处理。

生成工作流被分为三个可见性阶段：

1. **Phase 1**：只读取历史截断前可公开获得的证据，生成关键问题、证据矩阵、候选事实、Synopsis 和九个章节草案。
2. **Phase 2**：在 Phase 1 产物基础上增加模拟申办方、监管和中心反馈，执行事实调和、章节增量修订和关联产物同步。
3. **Evaluator-only**：只有生成结果冻结后，独立评估器才能读取隐藏参考设计和评分材料。

系统通过目录边界、输入摘要、提示词摘要和 checkpoint digest 防止隐藏资料进入生成上下文。输入、模型或提示词改变时，旧 checkpoint 会被拒绝复用。

###### 4.2 Phase 1：证据规划与草案生成

Phase 1 使用三个角色：

- `G1 Evidence Planner`：形成证据矩阵、阻断问题、候选 Fact Sheet、假设和 Synopsis。
- `G2 Section Writer`：在相同事实约束下逐章生成九个方案模块。
- `G3 Quality Judge`：检查来源边界、未支持声明、TBD 管理和跨章节一致性。

逐章 checkpoint 使长时间模型调用可以断点续跑，不需要因一次中断而重新生成全部章节。

###### 4.3 Phase 2：增量修订与关联产物编译

Phase 2 使用四个角色：

- `G4 Fact Reconciler`：吸收新增反馈，建立活动事实表、决策日志、变更计划、废止值和未决问题。
- `G5 Incremental Reviser`：对九个章节分别进行增量修订，并记录变更摘要和旧值残留扫描。
- `G6 Artifact Compiler`：同步生成 Trial Fact Sheet、SoA、estimand、样本量与统计、FDA/NMPA 路径、中心与招募计划、变更影响矩阵和问题登记册。
- `G7 Quality Judge`：对 Phase 2 可见资料执行质量审核。

语言模型质量门之后还有不可被其覆盖的确定性机器门。即使模型给出高分，只要确定性门禁发现关键矛盾，状态仍会被强制设置为 `blocked_machine_gate`。

##### 五、关键技术检查原理

###### 5.1 样本量可复现性

对双臂、连续型结局的简化正态近似，系统依据模型产物中的效应差、标准差、双侧显著性水平、目标 power 和脱落率重新计算所需样本量。简化形式为：

```text
n_evaluable_per_arm = 2 × ((z_(1-α/2) + z_power) × SD / effect)^2
n_total = ceil(2 × n_evaluable_per_arm / (1 - dropout_rate))
```

该公式不是对所有 MMRM 或复杂设计的最终统计结论，而是最低限度的可复现性检查。如果生成方案声称使用 MMRM、模拟或其他推导得到不同样本量，则必须提供可执行假设和输出，而不能只给一个无法复算的总人数。

Metformin-PAD 冻结产物写入 `n=220`，同时给出效应差 20 m、SD 65 m、85% power 和 15% 脱落。按上述近似计算，所需总样本量约为 447–448，`n=220` 的近似 achieved power 约为 0.557，因此机器门命中 `DET-STAT-001`。

###### 5.2 Estimand 与分析集一致性

如果 primary estimand 的人群定义为“所有随机化受试者”，而 FAS 又要求“至少接受一次治疗”或“至少有一次基线后测量”，FAS 可能排除随机化后没有用药或没有观察值的受试者，从而不能实现原 estimand 的人群定义。系统将这种矛盾标记为 `DET-STAT-002`，要求统计人员统一 estimand、人群、intercurrent event、缺失数据和敏感性分析。

###### 5.3 终末事件与缺失数据

死亡后功能结局并不是普通意义上的“随机缺失”。如果文本同时包含 death、MAR 和 multiple imputation，系统会触发 `DET-STAT-004`，要求明确死亡、截肢、血运重建等终末或重大临床事件的 estimand 策略，并给出临床可解释的复合策略、while-on-treatment、hypothetical 或其他适当处理及敏感性分析。

###### 5.4 合成监管资料的权威性控制

测试包可以包含模拟 FDA/CDE 会议纪要以测试系统响应，但模拟材料不能被下游文本表述为真实监管接受、要求或背书。系统同时检测“synthetic/模拟”来源线索与“FDA/CDE accepts/requires/endorses”等权威性措辞，命中时触发 `DET-REG-001`。整改要求将其改为非约束性的申办方候选方案，并用公开指南或明确 TBD 支撑区域路径。

###### 5.5 机器门禁整改闭环

每条阻断 finding 会被编译为结构化整改工单，包含：

- 责任角色；
- 受影响事实、章节和关联产物；
- 必须执行的修订动作；
- 可重检的退出条件；
- 是否允许自动应用。

固定闭环为：

```text
qualified decision
→ fact reconciliation update
→ cross-artifact propagation
→ deterministic revalidation
→ independent quality review
```

样本量、estimand、终末事件和监管路径属于高风险设计决策，系统明确设置 `automatic_application_allowed = false`。这不是功能缺失，而是责任控制：程序负责发现、组织、传播和重检，合格人员负责选择和批准。

##### 六、Benchmark 与验证方法

###### 6.1 为什么不能让模型给自己打分

早期 Metformin-PAD 盲评中，语言模型给出 85 分，而按其分项重新计算一度得到 98.18 分；独立模拟医学、统计和注册质量审核随后发现样本量无法复现、estimand/FAS 冲突、终末事件处理不当和模拟监管资料权威化等问题。这说明语言模型 judge 可能出现宽松评分和自洽性偏差。

因此当前评测采用独立金标准和确定性 scorer，报告 TP、FP、FN、TN、Precision、Recall、F1 与负对照准确率，并保留每一项匹配或拒绝的语义条件。

###### 6.2 NCT04683926 公开案例

该案例使用公开 Protocol、SAP、ICF、注册信息以及完全合成 CRF，建立 27 条 Trial Fact Sheet 人工金标准，并测试：

- 饮水要求的逻辑冲突；
- `>3 days` 与实际 3 天给药间隔的数值边界；
- 11 天与约 12 天的定义/计数差异；
- 30 天与 31 天的筛选窗口边界；
- Protocol 与 CRF/SAP 双时间轴合法映射；
- 人群与 SAP 范围问题；
- 完整来源追踪；
- 32 h → 36 h 变更传播；
- 人工治理门。

初始独立评分为：

| 指标 | 初始结果 |
|---|---:|
| Precision | 0.6667 |
| Recall | 0.5000 |
| F1 | 0.5714 |
| 负对照准确率 | 0.5000 |

补充严格数值边界、跨来源时长/窗口、来源链完整性和合法时间轴负控后，在复用同一冻结语义审核结果的情况下，重评分为：

| 指标 | 改进后结果 |
|---|---:|
| Precision | 0.8889 |
| Recall | 1.0000 |
| F1 | 0.9412 |
| 负对照准确率 | 1.0000 |

10 项 gold tests 的期望判断均正确，但仍有一条未匹配模型提醒被保守计为 false positive。该结果证明规则层在本案例上带来真实改进，但案例规模仍小，不能外推为普适临床审核准确率。

###### 6.3 补充公开案例与测试状态

仓库还包含 NCT03232983 和 NCT03117738 两套公开案例，用于测试试验编号、PK 窗口、访视时间、受试者数量范围和计划值/实际值负对照。当前代码已通过 65 项自动化测试，包括数据契约、文档图、修订合并、项目工作区、API、生成工作流、盲评器和 Phase 2 确定性门禁。

自动化测试通过说明工程行为与既定断言一致，不等于临床有效性已经得到验证。专业可靠性仍需更多案例、盲法人工复核和外部专家校准。

##### 七、可复现性、审计与数据边界

###### 7.1 可复现运行

每次运行保存：

- 输入与来源清单；
- 模型名称；
- 提示词文件摘要；
- 输入 payload 摘要；
- checkpoint context；
- 事实和章节版本；
- findings、proposals、decision requests；
- 质量门与最终状态。

生成型工作流对输入、模型和提示词计算摘要。如果上下文变化，含旧 digest 的 checkpoint 不能直接复用，从而降低“输入已经改变但系统沿用旧输出”的风险。

###### 7.2 数据边界

当前原型只使用公开文件、仓库内合成 fixture 和明确标记的模拟材料。系统不处理真实患者级数据，不应输入企业机密，不连接生产 DMS，也不替代真实监管沟通。运行产物默认保存在 Git 忽略的 `outputs/`，API 密钥从环境变量或仓库外 `.env` 读取，不写入运行记录。

###### 7.3 责任边界

系统可以：

- 提取和组织候选事实；
- 给出来源定位；
- 检测明确冲突和旧值残留；
- 生成候选红线和影响矩阵；
- 形成结构化人工决策请求；
- 执行可复现 benchmark 与回归检查。

系统不可以：

- 宣称自动完成医学、统计或注册批准；
- 将模拟监管意见表述为真实监管决定；
- 在证据不足时自动选择高风险设计参数；
- 将模型自评分当成临床质量证明；
- 在当前阶段处理真实患者数据或直接发布正式临床文件。

###### 12.9 真实公开 50 例从“案例数量”转化为不确定性实验契约

2026-07-19 冻结的 50 个 ClinicalTrials.gov 案例不是把单例演示机械复制 50 次，而是把
真实 Protocol/SAP 的 observation space 编译为主动取证契约。每个 action 的成本由所选
PDF 页数给出，65 份文档的成本中位数为 33 页、P90 为 99 页、最大为 241 页；因而策略
可以比较单位页数带来的后验熵下降，而不再默认所有证据同价。35 个合并文档案例与 15 个
分立文档案例还形成天然结构分层，可检验“联合观察”与“分步取证”的策略差异。

构建时，一个正文不可提取的候选被拒绝并由后续候选替补；固定角色关键词导致的假阴性
也被修复为“官方元数据定角色、正文信号做审计”。这一区分本身体现了可解释性原则：
解释信号必须说明其证据层级和失效模式，不能把一个易受版式影响的 proxy 升格为真值。
当前所有 case 均为 `gold_status=not_annotated`，因此它们能够证明来源完整性、覆盖范围和
实验可执行性，却仍不能证明模型准确率、概率校准或主动取证优于基线。后续独立盲标必须
冻结 split 后再建立 finding 与 patch-validity 标签，六臂消融器才允许输出效果声明。

###### 8.3 尚未完成

- 大规模、多治疗领域、跨语言 benchmark；
- 页码、表格和单元格级的通用来源定位；
- 经真实医学、统计、注册和质量专家形成的盲法评分集；
- 企业内部流程、工时和 ROI 的真实基线；
- 生产级身份权限、电子签名、DMS/EDC 集成与系统验证；
- 模型漂移监测、长期运行稳定性和正式合规验证。
- 冻结的真实 Agent 轨迹 calibration/test 数据、learned trajectory calibrator，以及由真实取证工具或专家标注支持的 observation model。

###### 8.4 核心研究结论

当前实验支持的结论不是“大模型可以自动完成临床试验设计”，而是：

> 将临床文档先转换为版本化事实和显式依赖，再组合确定性规则、受治理语义模型、独立 benchmark 和人工质量门，可以把不可控的文本生成转化为可追踪、可测试、可阻断、可继续改进的临床文档工程流程。

TrialCompiler 当前最有价值的技术贡献，是建立了一个诚实的控制框架：模型可以提出候选方案，但来源、依赖、修改、评测和批准均被显式记录；当生成结果存在统计或监管硬伤时，系统能够拒绝用流畅文本和高自评分掩盖问题，并将错误转换成可执行的专业整改闭环。

从项目演化角度看，当前临床文档原型继承并强化了早期目标对齐阶段的核心思想：先确认全局目标和约束，再让多角色代理围绕具体产物进行检查、修复、复核和报告。不同之处在于，临床试验方案把“目标偏离”具体化为事实、来源、章节、表格、版本和监管规则之间的可审计关系，使原本抽象的 alignment 问题落到可以测试、可以记录、可以阻断的文档工程流程中。

#### TrialCompiler 数学技术证明（v2）

##### 0. 文档定位

本文给出 TrialCompiler 当前原型的形式化描述、关键正确性命题与可复现实验对应。与旧版数学分析相比，本版做三项修正：

1. 严格区分当前代码已经实现的性质与未来可能训练的目标函数；
2. 将抽象公式落实到 Trial Fact Sheet、Clinical Document Graph、确定性规则、质量门和 benchmark scorer；
3. 使用 NCT04683926 与 Metformin-PAD 的实际运行结果验证推导，不把语言模型自评分视为证明。

本文证明的是工程控制性质：在明确假设下，系统能够保持来源、状态和变更传播的可审计性，能够阻断一组已编码的严重矛盾。本文不证明生成内容具有普适临床正确性，也不证明系统可以替代合格专业人员。

##### 1. 形式化对象

###### 1.1 来源、事实与文档单元

设授权来源集合为

$$
\mathcal{S}=\{s_1,\ldots,s_m\},
$$

每个来源表示为

$$
s_i=(id_i,locator_i,version_i,authority_i,scope_i).
$$

设事实集合为

$$
\mathcal{F}=\{f_1,\ldots,f_n\},
$$

其中

$$
f_i=(id_i,key_i,value_i,status_i,source_i,owner_i,version_i,previous_i).
$$

`status` 属于有限状态集合：

$$
\Sigma=\{draft,proposed\_change,requires\_human\_review,approved,rejected,superseded\}.
$$

文档单元集合为

$$
\mathcal{U}=\{u_1,\ldots,u_k\},
$$

一个文档单元可以是 Protocol 章节、SoA 表格、SAP 规则、ICF 表述、CRF 字段或注册记录。

###### 1.2 依赖矩阵与文档图

定义事实到文档单元的依赖矩阵

$$
A\in\{0,1\}^{n\times k},\qquad
A_{ij}=1 \iff u_j\text{ 引用或依赖 }f_i.
$$

Clinical Document Graph 可写为

$$
G=(V,E),\qquad V=\mathcal{S}\cup\mathcal{F}\cup\mathcal{U}\cup\mathcal{I}\cup\mathcal{R}\cup\mathcal{H},
$$

其中 $\mathcal{I}$ 为 findings，$\mathcal{R}$ 为 repair proposals，$\mathcal{H}$ 为人工决定。边至少包括：

$$
source\rightarrow fact,\quad fact\rightarrow unit,\quad finding\rightarrow fact,\quad repair\rightarrow finding.
$$

对事实 $f_i$，其影响集合为

$$
Impact(f_i)=\{u_j\in\mathcal{U}:A_{ij}=1\}.
$$

这一定义直接对应代码中的 `fact_to_sections` 与 `impact_set`。

##### 2. 三个系统不变量

###### 不变量 I：来源可追溯性

对任何进入正式审核状态的事实 $f_i$，必须满足

$$
source_i\neq\varnothing
$$

且

$$
\forall s\in source_i,\quad s\in\mathcal{S}\land locator(s)\neq\varnothing.
$$

这意味着事实不能只有模型生成的文字而没有可定位来源。

###### 不变量 II：状态不可由置信度替代

设模型对事实正确性的置信度为 $p_i$。事实可驱动正式修改的必要条件不是 $p_i>\tau$，而是

$$
Effective(f_i)=
\mathbf{1}[status_i=approved]
\cdot\mathbf{1}[source_i\neq\varnothing]
\cdot\mathbf{1}[QualifiedOwner(owner_i)].
$$

即使 $p_i=0.99$，只要未被合格角色批准，$Effective(f_i)=0$。

###### 不变量 III：硬门禁不可由软分数抵消

设语言模型质量分为 $Q_{LLM}\in[0,100]$，确定性阻断 finding 数为 $B\ge0$。系统的释放函数定义为

$$
Release(X)=\mathbf{1}[B(X)=0]\cdot\mathbf{1}[HumanGate(X)=1].
$$

因此：

$$
B(X)>0\Longrightarrow Release(X)=0,
$$

与 $Q_{LLM}$ 的数值无关。

**命题 1（非补偿性）**：不存在任何有限的语言质量增量 $\Delta Q$ 能使一个含阻断 finding 的状态自动释放。

**证明**：若 $B(X)>0$，则第一指示函数为 0，故 $Release(X)=0$。$Q_{LLM}$ 不出现在释放函数的乘积项中，因此任意 $\Delta Q$ 均不改变结果。证毕。

该命题对应 Phase 2 中 `blocked_machine_gate` 对 LLM judge 的强制覆盖。

##### 3. 变更传播的正确性

###### 3.1 候选变更

设事实 $f$ 从旧值 $v_0$ 变为候选值 $v_1$：

$$
\Delta f=(f,v_0\rightarrow v_1).
$$

系统不直接删除 $v_0$，而是保存

$$
previous(f)=v_0,\quad value(f)=v_1,\quad status(f)=proposed\_change.
$$

对每个 $u\in Impact(f)$，定义旧值残留谓词：

$$
Stale(u,f)=\mathbf{1}[Contains(u,v_0)].
$$

系统生成 finding 集合

$$
F_{stale}(f)=\{u\in Impact(f):Stale(u,f)=1\}.
$$

###### 3.2 完备性命题

**假设 A1**：依赖矩阵 $A$ 完整，即所有真实依赖单元均被登记。

**假设 A2**：`Contains` 对待检测的原子旧值无漏检。

**命题 2（旧值残留检查的完备性）**：在 A1、A2 下，所有依赖于 $f$ 且仍包含 $v_0$ 的单元都会进入 $F_{stale}(f)$。

**证明**：任取真实依赖且包含旧值的单元 $u_j$。由 A1，$A_{fj}=1$，故 $u_j\in Impact(f)$；由 A2，$Stale(u_j,f)=1$。根据集合定义，$u_j\in F_{stale}(f)$。证毕。

###### 3.3 健全性命题

**假设 A3**：`Contains(u,v_0)=1` 仅当 $v_0$ 在 $u$ 中代表该事实，而非无关同值。

**命题 3（旧值残留检查的健全性）**：在 A1、A3 下，$F_{stale}(f)$ 中不存在与 $f$ 无关的单元。

**证明**：任取 $u\in F_{stale}(f)$。集合定义要求 $u\in Impact(f)$ 且 `Contains` 为真。前者说明存在依赖，A3 排除同值但不同语义的情况，因此 $u$ 是真实残留位置。证毕。

实际系统不能保证 A1 和 A3 对所有自由文本恒成立，因此 benchmark 同时报告 Impact Recall 与负对照准确率，而不是宣称规则绝对正确。

##### 4. 最小修订与原子补丁

设原文档为 $Y$，目标 finding 集合为 $I$，候选文档为 $Y'$。理想修订满足：

$$
\min_{Y'} D_{edit}(Y,Y')
$$

约束为

$$
Unresolved(Y',I)=0,
$$

$$
NewRegression(Y',Y)=0,
$$

$$
Traceable(Y')=1.
$$

系统把每个修订表示为原子操作

$$
o=(section,start,end,before,after,fact\_ids,source\_ids).
$$

若两个操作 $o_a,o_b$ 的区间不重叠，则可交换：

$$
[start_a,end_a)\cap[start_b,end_b)=\varnothing
\Longrightarrow o_a\circ o_b=o_b\circ o_a.
$$

**命题 4（非重叠补丁可合并）**：若所有原子操作区间两两不重叠，且每个操作的 `before` 与原文一致，则按位置有序应用不会互相覆盖。

**证明说明**：不相交区间修改不同字符集合；从高偏移到低偏移应用可避免前一操作改变后一操作的坐标，因此所有替换均保留。若区间重叠，则交换律不成立，系统不自动选择，而生成 `DecisionRequest`。

这解释了为何 TrialCompiler 采用最小红线和冲突合并，而不是让多个 Agent 顺序重写整章。

##### 5. 数值边界检查

###### 5.1 严格不等式与实际日程

在 NCT04683926 案例中，一处文本声明给药间隔

$$
\Delta d>3\text{ days},
$$

而给药日为

$$
D=(1,4,7,10).
$$

相邻差分为

$$
\nabla D=(4-1,7-4,10-7)=(3,3,3).
$$

因此

$$
\min(\nabla D)=3\not>3.
$$

**命题 5**：给药日 1、4、7、10 与“greater than 3 days”不能同时为真。

该问题不依赖语言模型推理，可由确定性规则稳定发现。这里必须区分 $>3$ 与 $\ge3$，否则会出现边界漏检。

###### 5.2 合法双时间轴负对照

Protocol 使用连续时间轴

$$
T_P=\{-1,1,4,7,10,11\},
$$

CRF/SAP 使用 period-specific 表示，每个 Period 内给药日记为 Day 1。设映射

$$
\phi(period_i,day\ 1)=D_i,
$$

其中 $D_i\in\{1,4,7,10\}$。只要 $\phi$ 被明确记录且两个时间轴事实均获批准，两种表示是坐标变换而非矛盾。

因此系统使用负对照约束：

$$
Approved(F022)\land Approved(F023)\land MappingExists
\Longrightarrow Conflict=0.
$$

该约束使 NCT 案例的双时间轴误报被消除。

##### 6. 样本量可复现性证明

###### 6.1 正态近似

对等比例双臂连续型结局，设两组均值差为 $\delta$，共同标准差为 $\sigma$，双侧显著性水平为 $\alpha$，目标 power 为 $1-\beta$。每组可评价样本量近似为

$$
n_{arm}=2\times((z_{1-\alpha/2}+z_{1-\beta})\sigma/\delta)^2.
$$

若脱落率为 $r$，总随机样本量为

$$
n_{total}=ceil(2n_{arm}/(1-r)).
$$

###### 6.2 Metformin-PAD 代入计算

冻结产物给出：

$$
\delta=20\text{ m},\quad \sigma=65\text{ m},\quad \alpha=0.05,
$$

$$
1-\beta=0.85,\quad r=0.15.
$$

查标准正态分布：

$$
z_{0.975}\approx1.960,\qquad z_{0.85}\approx1.036.
$$

于是

$$
n_{arm}\approx2\times(((1.960+1.036)\times65)/20)^2\approx189.6,
$$

$$
n_{total}\approx ceil((2\times189.6)/0.85)\approx447.
$$

因此生成文本中的 $n=220$ 无法由其自身给出的参数复现。

若仍使用 $n=220$，近似 achieved power 为

$$
Power\approx\Phi(\delta\sqrt{n(1-r)/(4\sigma^2)}-z_{1-\alpha/2})\approx0.557.
$$

**命题 6**：在上述简化双臂正态近似假设下，$n=220$ 不足以达到 85% power。

该证明只用于检查“文本中的数字能否相互复现”。若专业统计方案采用 MMRM、协变量调整或模拟得到其他样本量，必须提供可执行假设和输出，系统不能用此近似公式替代正式统计设计。

##### 7. Estimand 与分析集的一致性

设 estimand 的目标人群为所有随机化受试者：

$$
P_E=\{x:x\text{ 被随机化}\}.
$$

若 FAS 要求至少用药一次且至少有一次基线后测量：

$$
P_{FAS}=\{x:x\text{ 随机化}\land dose(x)\ge1\land postbaseline(x)\ge1\}.
$$

显然

$$
P_{FAS}\subseteq P_E.
$$

只要存在随机化后未用药或无基线后测量的受试者，便有

$$
P_{FAS}\subsetneq P_E.
$$

**命题 7**：若 estimand 声称针对所有随机化受试者，而实现它的分析集排除部分随机化受试者，则分析集不能完整实现该 estimand 人群定义。

系统因此触发 `DET-STAT-002`，但不会自动决定应修改 estimand 还是 FAS；该选择属于合格统计人员的责任。

##### 8. 终末事件不是普通 MAR 缺失

令 $Y_t$ 为时间 $t$ 的功能结局，$D_t=1$ 表示受试者在 $t$ 前死亡。死亡后 $Y_t$ 不是“虽然存在但未观测”的普通随机变量，而可能在临床定义上不存在。若直接设

$$
Y_t\mid D_t=1\sim MAR
$$

并执行普通多重插补，就隐含假设死亡后的功能结局可由存活者观测分布合理生成，这通常缺乏临床可解释性。

因此系统采用保守规则：文本同时出现 death、MAR 和 multiple imputation 时触发 `DET-STAT-004`，要求明确 composite、while-on-treatment、hypothetical 或其他适当 intercurrent-event 策略及敏感性分析。

这是一条风险检测规则，不是对所有 estimand 策略的数学裁决。

##### 9. Benchmark 评分与改进证明

###### 9.1 基本指标

设 TP、FP、FN、TN 分别为真阳性、假阳性、假阴性和真阴性：

$$
Precision=TP/(TP+FP),\qquad Recall=TP/(TP+FN),
$$

$$
F1=2PR/(P+R).
$$

NCT04683926 初始评分：

$$
TP=4,\quad FP=2,\quad FN=4,\quad TN=1,
$$

故

$$
Precision=4/6=0.6667,
$$

$$
Recall=4/8=0.5000,
$$

$$
F1=0.5714.
$$

加入数值边界、跨来源时长/筛选窗口、来源链完整性及合法时间轴负控后：

$$
TP=8,\quad FP=1,\quad FN=0,\quad TN=2,
$$

因此

$$
Precision=8/9=0.8889,
$$

$$
Recall=8/8=1.0000,
$$

$$
F1=(2\times0.8889\times1)/(0.8889+1)=0.9412.
$$

###### 9.2 改进幅度

绝对 F1 增量为

$$
\Delta F1=0.9412-0.5714=0.3698.
$$

相对增幅约为

$$
0.3698/0.5714\approx64.7\%.
$$

同时负对照准确率从 0.5 提升到 1.0，说明提升并非简单通过“报告更多问题”换取 Recall，而是同时消除了已知时间轴误报。

不过该 benchmark 只有 10 个 gold tests。样本规模过小，不应报告窄置信区间或将结果外推到其他治疗领域。正确结论是：在同一冻结案例和同一 scorer 下，新增规则修复了全部已知 FN，并消除了已知负控误报。

##### 10. 整改闭环的终止条件

设阻断 finding 集为 $B_t$，每条 finding 对应一个整改工单 $w_i$，其退出条件谓词为 $Exit(w_i)$。系统允许进入独立质量审核的条件为

$$
Ready_t=
\mathbf{1}[B_t=\varnothing]
\prod_i\mathbf{1}[Exit(w_i)=1].
$$

整改序列为

$$
qualified\ decision
\rightarrow fact\ update
\rightarrow propagation
\rightarrow revalidation
\rightarrow independent\ review.
$$

高风险工单设置

$$
AutomaticApply(w_i)=0,
$$

因此系统可以自动生成整改要求和重检结果，但不能自动批准样本量、estimand 或监管路径。

若每轮至少关闭一个 finding，且不产生新 finding，则 $|B_t|$ 严格递减，最多经过 $|B_0|$ 轮达到空集；若产生新 finding、证据不足或专业意见冲突，则系统转入人工决策而不是无限自循环。

##### 11. 当前证明覆盖与未证明事项

###### 11.1 当前可由代码与测试支持

- 类型化来源、事实、章节、finding、repair 和 decision request；
- 事实到章节的显式依赖与影响集合；
- NCT 标识符、Week、严格数值边界和旧值残留等确定性检查；
- 原子补丁合并和重叠冲突升级；
- Phase 1/Phase 2/Evaluator-only 可见性隔离与 checkpoint digest；
- 样本量、合成监管权威化、estimand/FAS、多重性和终末事件门禁；
- 独立 gold scorer 与负对照；
- 当前 65 项自动化测试通过。

###### 11.2 尚不能证明

- 对任意临床文档均能完整抽取所有事实；
- 对任意跨文件语义冲突均有高 Recall 与高 Precision；
- 当前规则覆盖所有统计或监管风险；
- 语言模型判断具有稳定跨领域校准；
- 系统能够替代医学、统计、注册或质量人员；
- 小规模公开/合成 benchmark 的结果可以代表生产性能。

##### 12. 结论

TrialCompiler 当前最可靠的数学描述不是“一个自动写 Protocol 的生成模型”，而是一个带硬约束的文档状态转换系统：

$$
X_{t+1}=T(X_t,a_t;G,S,H),
$$

其中 $G$ 是显式文档依赖图，$S$ 是带来源和状态的事实集合，$H$ 是人工治理状态。系统优先保持三个不变量：来源可追溯、状态不可由模型置信度替代、硬门禁不可由软分数抵消。

在依赖登记和原子匹配正确的假设下，旧值传播检查具有可证明的完备性与健全性；在不重叠区间假设下，原子补丁可以安全合并；样本量、严格时间边界和集合包含关系可以通过确定性计算复核；独立 benchmark 证明新增规则在 NCT04683926 小型测试集上将 F1 从 0.5714 提升到 0.9412，并同时提高负对照准确率。

因此，本项目的技术贡献不在于宣称 AI 已经获得临床专业正确性，而在于把模型输出置于一个可表示、可检查、可阻断、可回放并最终由专业人员批准的工程体系中。

## 第九部分 原型验证的完整证据与失败整改

### 9.1 原型验证报告全文

以下内容整合自冻结的原型验证报告，覆盖 NCT04683926、Metformin-PAD、50 例真实公开
Protocol/SAP、130 个官方角色标签、200 个受控缺陷标签、197 条自然候选的裁决前冻结，
以及模拟人工批准后的最小修订闭环。报告同时保留失败、整改和不可外推范围。

#### TrialCompiler 双工作流阶段性验收报告

##### 0. 原型暴露的问题、修复与研究转向

复核发现，同一次 NCT04683926 运行曾出现 `workflow_state.json` 已判定机器修复未完成，而 `run_summary.json` 仍保留旧的通过状态。这证明单个 F1 或语言模型评分不能代表系统可信；状态、摘要、决策队列和报告一旦漂移，高分就可能被误解释成可发布。

本轮完成三项工程修复：摘要改为由权威状态完整重新物化；所有机器无法闭合的 finding 必须进入合格人员决策队列；新增 `trialcompiler.assurance/v1`，机器检查跨产物一致性、补丁来源、未决事项处置、沙箱回归、负对照、指标与门禁分离、人工授权和回滚摘要。修复后的公开案例证明结果为 `safely_blocked_pending_qualified_resolution`：检测 F1=0.9412、负对照准确率=1.0，但仍明确 `release_authorized=false`。证明见 `evidence/NCT04683926携证编译证明.json`。

更重要的研究转向是“不确定性驱动的证据获取与忠实补丁解释”。规则擅长已知格式冲突，却不能充分处理语义等价、隐含矛盾和多种合理事实解释。TrialCompiler 把文档审核形式化为 action–observation trajectory：不确定性估计必须注明目标事件以及它位于 observation、belief、action 或 trajectory outcome 的哪一层；来源冲突和模型分歧只作为估计信号，未经独立校准集验证的分数不得称为 calibrated confidence。

解释评价也不再以“是否写出合理理由”为标准。来源追踪、轨迹可观察性和解释忠实性分别评价；对解释忠实性的最低实验要求是 evidence-removal/replacement replay，检验被声称关键的证据发生干预时，finding、取证动作或 patch 选择是否按解释发生变化。该结果属于行为层反事实证据，不宣称揭示模型内部因果机制。

后续消融实验分为纯规则、普通 LLM/RAG、带被动 UQ 的 LLM、以及不确定性驱动主动取证 Agent。除 Precision/Recall/F1 外，评价 ECE/Brier/rank-calibration、risk–coverage/AURC、轨迹最弱步骤校准、单位证据成本带来的熵下降、取证后任务成功增益、拒答质量、负对照保护率，以及证据移除/替换后的解释忠实性。当前公开案例只验证了确定性检测、来源追踪、重放底座与携证阻断；尚未完成真实模型校准和真实主动取证实验，因此不能把现有 reviewer confidence 或审计日志当成 UQ/XAI 结果。

2026-07-18 新增的不确定性实验底座已经打通一个可执行的最小闭环：有限假设 belief state → 成本感知预期信息增益选证 → 接收显式 observation → posterior 更新与实际熵下降记录 → commit/acquire/defer 决策。独立评估器可输出 Brier、ECE、risk–coverage/AURC、成对排序准确率，以及证据移除/替换的 necessity flip rate 与 contrastive sensitivity；混用 calibration/test split 时禁止给出校准声明，未知 observation 则失败关闭。仓库示例数据只用于验证计算与报告链路，不构成模型效果证据。真实结论仍需冻结数据集、真实模型轨迹、独立 calibration/test 划分和专业标注。

该底座现已进入主审核链路：D 独立质量门之后由 G 不确定性治理节点生成 `uncertainty_trajectory.json`，API 与审核报告同时返回所选下一动作、证据摘要哈希、诊断估计、反事实重放说明和声明边界。新运行若缺少该产物，或在没有拟合校准器时允许 calibration claim，assurance case 将以 `uncertainty_claim_governance` 失败并阻断机器就绪结论。当前 G 节点执行的是诊断信号到治理动作的编译；真实概率仍待独立数据拟合。

G 选择 `acquire_evidence` 且调用方提供受治理 catalog 时，H 节点现可在硬预算内消费一项 observation，并重新进入 B/C/D/G。准入条件包括 `approved_for_research` 状态、项目与文档 ID 精确匹配、`semantic_review` 类型以及规范化 payload 的 SHA-256 一致；失败项保留机器可读拒绝原因。终态不保留原始 catalog，只记录 evidence/source ID 与 digest。该机制验证了安全回环和防篡改边界，但 catalog 仍由实验调用方准备，尚不是面向外部文献库的自主检索器。

为避免先看结果再定义比较方式，仓库新增六臂配对消融评估器与预注册式协议。评估器拒绝 case 跨 calibration/test 泄漏、重复 arm-case、各臂 case 集不一致和冻结摘要漂移；只有 held-out test、A–F 六臂完整且达到预先声明的最小 case 数时，才允许 `result_claim_allowed=true`。当前尚无达到该门槛的真实六臂数据，因此新增内容属于实验基础设施，不新增任何真实性能主张。

主链路现进一步支持受治理取证执行：API 可接收显式允许列表中的 evidence action 与 observation model，在最大步骤和最大成本内选择正净信息价值动作，记录来源 ID、版本、内容 SHA-256、预期/实际信息增益和 posterior。未在允许列表中的动作、重复取证、未知 observation 与预算耗尽均失败关闭。即使取证后的 belief 超过候选提交阈值，G 也只返回 `deliberate`，要求新证据重新进入 B/C/D；belief 分数不能越过独立质量门或专业决策债务。当前演示 observation 为合成数据，尚不构成真实来源取证效果。

###### 3.2 原始盲评为何不可信

第一次盲评中，语言模型给出 85 分，而按其分项 rubric 重新计算得到 98.18 分。该结果表面很高，但随后由三个相互独立的模拟审核员分别从医学安全、统计和注册质量角度复核，发现多项严重问题。这说明原始 LLM Judge 存在明显的宽松评分和自洽性偏差。

###### 3.3 模拟专业审核员的主要发现

**医学与安全审核：**

- “所有随机化受试者”的 estimand 与要求至少用药一次且有基线后数据的 FAS 不一致。
- 死亡、截肢和血运重建不能简单作为 MAR 缺失值统一插补。
- AE 分级和部分安全管理规则存在错误或跨章节冲突。
- 研究停止规则、DMC 文件时间和造影剂停药规则仍需专业修订。

**统计审核：**

- 在双侧检验、效应差 20 m、标准差 65 m、85% power、15% 脱落的生成假设下，220 例无法复现；正态近似需要约 447 至 448 例，220 例对应 power 约 55.7%。
- 20 m 效应假设证据不足，MMRM 与 MI 的实现描述不完整。
- 多重性控制在不同章节存在直接矛盾。

**注册与质量审核：**

- 测试包中的模拟 FDA/CDE 沟通纪要被下游文本错误提升为监管认可或要求。
- 中国人群数量、桥接 PK、505(b)(2) 路径和 NMPA 2.4/CTA 依据不足。
- 部分来源定位和 TBD 管理尚不足以形成申报级追溯链。

以上审核均为模拟审核，不构成医学、统计、注册或质量专业批准。

###### 3.4 新增确定性机器门禁

为避免语言模型用流畅文本掩盖基础矛盾，Phase 2 现增加不可被 LLM Judge 覆盖的确定性检查：

- `DET-STAT-001`：样本量和 power 的可复现性检查。
- `DET-REG-001`：合成监管材料被表述为真实监管认可的检查。
- `DET-STAT-002`：estimand 人群与分析集定义一致性检查。
- `DET-STAT-003`：多重性策略跨章节矛盾检查。
- `DET-STAT-004`：死亡等终末事件被不当纳入 MAR 插补的检查。

对冻结的 Phase 2 产物重新执行机器门禁后，运行状态由表面的高分结果改为：

```text
status: blocked_machine_gate
quality_gate: blocked_missing_evidence
```

本次冻结产物实际命中 `DET-STAT-001`、`DET-REG-001`、`DET-STAT-002` 和 `DET-STAT-004`。这意味着系统现在会诚实地阻止该方案进入专业人员批准阶段，而不是继续展示误导性的“98.18 分”。

###### 3.5 2026-07-18 机器门禁整改闭环

机器门禁现不再只输出错误列表。每条阻断 finding 都会编译成结构化 remediation work item，明确责任角色、必须同步修改的事实与产物、强制修订动作、可机器复核的退出条件，并禁止程序自动代替专业人员批准设计决策。

固定闭环为：`qualified decision → fact reconciliation update → cross-artifact propagation → deterministic revalidation → independent quality review`。冻结运行已生成 4 个整改工单，状态为 `blocked_pending_qualified_revision`；只有所有退出条件满足且确定性重检返回零阻断 finding，才允许进入后续独立质量审核。

可通过 `scripts/build_phase2_remediation_plan.py` 对任意冻结 Phase 2 `run.json` 重建整改包，避免依赖原生成模型的自评结论。

##### 8. 2026-07-19：50 个真实公开 Protocol+SAP 案例扩展验证

为解决“单案例结果无法代表跨研究稳健性”的暴露问题，本轮没有直接复制 NCT04683926
的 F1，而是先建立可审计的真实公开语料底座。构建器通过 ClinicalTrials.gov API v2
检索同时具有 Protocol 与 SAP 标记的研究，冻结 50 个唯一 NCT 案例、50 份注册快照与
65 份官方上传 PDF 的 URL、字节数、页数和 SHA-256。入选文档均通过 PDF 头、非空页数
和至少 100 个可提取正文字符的检查；1 个额外候选因正文不可提取被拒绝并由后续案例
替补，因此最终数量不是“下载成功数”，而是“通过完整性门的有效案例数”。

该 50 例覆盖 48 个干预性研究和 2 个观察性研究；35 例使用合并 Protocol/SAP，15 例
使用分立文档。65 份文档合计 3,025 页，单次文档读取成本中位数为 33 页、P90 为 99
页、最大为 241 页。这使主动取证不再只面对等成本的合成 action，而可以按真实文档
页数比较信息增益与证据成本。50 个 case contract 均保留 enrollment 与 primary
outcomes 注册字段，并明确 `gold_status=not_annotated`。

验证过程中还修复了两个方法缺陷。第一，原构建器会在达到 50 例前并发提交全部候选，
造成无必要下载与长尾等待；现改为分批取证、达到门槛后停止。第二，固定关键词曾被错误
用作文档角色真值，导致官方合并 Protocol/SAP 被大量假阴性淘汰；现由 ClinicalTrials.gov
的 `hasProtocol/hasSap` 元数据承担角色依据，正文关键词仅作为辅助审计，同时保留正文
可提取门。最终 65 份文档中 Protocol 关键词信号为 59 份、SAP 关键词信号为 48 份；
该差异被保留在构建报告中，不被伪装成 100% 命中。

本轮验证解决的是“真实来源完整性、跨案例覆盖与取证成本契约”缺口，不是模型效果金标准。
在独立盲标 finding、patch validity 和反事实证据依赖之前，50 例不得用于宣称准确率、
校准效果或六臂消融效应量。六臂实验底座会拒绝未配对 arm、split 泄漏、冻结摘要漂移
以及低于预注册样本量的结果声明；因此当前成果是把下一步实验从不可执行变为可执行且
不可越界，而不是提前制造性能数字。

##### 9. 2026-07-19：首批 130 个官方元数据金标签与 held-out 结果

held-out test 包含 10 个案例、14 份文档和 28 个二元标签。正文语义基线得到
Precision=82.61%、Recall=95.00%、F1=88.37%、Accuracy=82.14%；规范文件名基线在本批

##### 10. 50 例受控缺陷金标签与核心检测结果

为避免仅用角色识别结果说明核心效果，本轮在 50 个真实冻结案例上新增两类受控缺陷：
canonical NCT identifier 冲突，以及 enrollment proposed change 未传播至文档。每例为每类
缺陷各设置一个正例和一个匹配负对照，共形成 200 个缺陷级标签，其中正例 100、负对照
100。负对照不仅是空白文本：identifier 负对照包含正确当前 NCT 和另一个仅作为相关研究
引用的 NCT，enrollment 负对照则包含已经正确传播的新值。

切分仍按 NCT 冻结为 30/10/10。Held-out test 的 10 个案例、40 个标签上，当前
TrialCompiler 确定性检测器得到 TP=20、FP=0、FN=0、TN=20，Precision、Recall、F1 与
Accuracy 点估计均为 100%；Accuracy Wilson 95% 区间为 91.24%–100%。全量 200 标签全部
判断正确，Accuracy Wilson 95% 区间为 98.12%–100%。因此可以得出受范围约束的积极结论：
系统在 50 个真实案例派生的标识符一致性和 enrollment 变更传播任务上表现稳定，既检出
全部注入缺陷，也未破坏匹配负对照。

上述数据不能外推为自然临床语义缺陷总体准确率。这两类任务属于可确定裁决的受控缺陷；
自然发生的 estimand、缺失数据、多重性、医学安全性与注册策略问题仍需合格人员独立标注。
标签与结果分别保存在 `gold/seeded_defect_labels.jsonl` 和
`gold/seeded_defect_results.json`。

##### 11. 真实自然缺陷候选挖掘与专家裁决边界

受控缺陷达到 100% 后，本项目没有继续扩大同构注入任务来维持高分，而是直接解析 50 个
案例的 65 份真实 PDF，并从真实页码与原文中高召回挖掘自然差异候选。初始队列共 1,393
条，覆盖 48 个案例，包括 analysis population 65 条、missing data 51 条、multiplicity
18 条、primary endpoint/time 105 条、terminal event 105 条、trial identifier scope 45
条，以及 enrollment scope/state difference 1,000 条；另发现少量 estimand/intercurrent
event 词面候选。

大量 enrollment 差异并不等于缺陷，可能分别表示 planned/actual、筛选/随机化、总样本/
亚组或每组/合计；不同 NCT ID 也经常只是相关研究引用。为避免关键词命中被直接包装为
gold，系统按每个案例、每个类别保留上下文质量最高的一条，形成 197 条、48 个案例、8
个类别的多元裁决集。每条均保存真实 PDF 文件名、页码、短原文、注册事实、候选摘要和
所需裁决角色，但 `label_status` 固定为 `candidate_not_gold`，裁决字段为空。

只有同时满足双来源范围匹配、明确语义关系、裁决理由、合格角色和第二复核的候选才能进入
自然缺陷 gold。当前 200 个受控标签仍可用于确定性回归；197 条真实自然候选用于下一步
医学/统计/注册裁决，尚不纳入性能分母。这一治理阻止系统用自己挖掘的候选自行宣布缺陷，
从而避免循环评价和虚高成功率。

###### 11.1 裁决前冻结测试

为避免在看到自然 gold 后再调整系统，本轮已先对 197 条真实候选冻结逐条 disposition。
系统完成 197/197 条证据定位，来源文件名与页码完整率为 100%；0 条被自动宣布为已确认
缺陷，197/197 条均保留合格人员复核要求。其中 43 条被判为 enrollment scope/state
ambiguous，69 条进入统计语义裁决，80 条需要医学与统计上下文，5 条跨 NCT 候选进入范围
复核（其中 1 条具有明显 related-study 线索）。预测文件整体 SHA-256 为
`59E16BEAEB7B6C07ABD9F0601300CC96EF4F41D3AB890F41DE94F1DC7F239FA2`。

这次运行可以评价覆盖率、路由行为、自动提交率和证据定位完整率，但在 gold 尚未裁决时

##### 12. 模拟人工批准后的修订闭环验证

人工复核不是系统的缺陷，而是临床语义变更进入下一阶段的授权边界。为验证“发现问题后
是否真的能正确修改”，本轮在 50 个真实公开案例派生的 200 个受控场景上执行完整闭环：
系统发现候选问题，模拟具备相应职责的数据治理人员或统计人员作出批准/驳回决定，系统仅
对支持确定性最小修订的项目生成补丁，在沙箱应用后重新运行审查。100 个缺陷场景被模拟
批准，100 个匹配负对照被模拟驳回。

100 个批准项中，50 个 enrollment 旧值传播问题具备安全的确定性补丁。50/50 个补丁均
满足：新规范值出现在目标章节、旧值被移除、只改变提案指定章节、事实及来源引用链保持
不变、原 finding 在复检后关闭，且未引入新的 finding。修订闭环成功率点估计为 100%，
Wilson 95% 区间为 92.87%–100%。冻结 held-out test 的 10 个案例中，10/10 个可修订问题
同样正确闭合。100 个驳回的负对照均保持字节级文档摘要不变，意外修改数为 0。

另外 50 个 canonical NCT identifier 冲突虽然获得模拟批准，但当前修订器不将试验身份
字段列入自动替换白名单，因此全部返回 `approved_but_manual_edit_required`，没有擅自修改
正文。原因是错误 NCT ID 可能来自文档身份错误、相关研究引用或来源范围错误，仅凭字符串
替换无法决定哪一种解释成立。这一结果表明批准是启动受控处理的必要条件，但批准本身不
会绕过修订能力边界：可证明安全的修改自动闭环，不能证明安全的修改继续由人工编辑并再
次复检。

逐项记录与汇总分别位于
`results/simulated_human_repair_records.jsonl` 和
`results/simulated_human_repair_report.json`。以上“人工”均为受控实验中的模拟角色决定，
验证的是审批后修订机制，不构成真实医学、统计、注册或质量授权，也不能证明 197 条自然
语义候选的补丁有效性。

#### NCT04683926 可复现测试流程

##### 1. 测试目的

本测试验证 TrialCompiler 是否能把公开临床研究文件转换为可追踪事实层，识别单文档与跨文档不一致，并在一项关键事实变化后完成影响定位、候选修订、沙箱复检和人工审核门控。

案例为 NCT04683926 / OMNI-PAIN-103。它是一项已完成的 Phase 1、随机、开放、四治疗、四周期、四序列交叉试验。测试只使用公开 study-level 文件和明确标记的合成变更，不使用患者级数据。

本测试不是对临床设计作医学批准。AI 输出始终是候选意见；医学、统计、注册和质量判断必须由具备相应职责的人员完成。

##### 2. 测试边界

系统允许：

- 提取带来源定位的候选事实；
- 区分计划值、执行值、结果值和历史版本；
- 检测数字、时间点、术语、日程和跨文件语义冲突；
- 建立事实到章节、表格和文件的依赖关系；
- 生成操作级候选修订；
- 在不覆盖正式文件的沙箱中重新检查；
- 形成 Decision Request 和人工审核包。

系统禁止：

- 将公开资料中的差异直接判为某一来源错误；
- 用模型猜测替代缺失的专业决策；
- 将合成变更伪装成真实方案修订；
- 在未解决专业决策时批准或覆盖正式版本；
- 把模型输出写入可复用经验库而不经过人工批准。

```powershell
  --request-id <REQUEST_ID> `
  --decision require_recompile `
  --reviewer qualified-reviewer `
  --note "现有证据不足以授权唯一文本；补充证据后重新编译。"
```

另一合法选择是 `accept_documented`，表示审核者接受已记录差异，并提供非空理由。未解决的 Decision Request 必须 100% 阻断 `approve`。

参考运行的 3 个请求涉及：

- Protocol、SAP 与 CRF 的双时间轴解释；
- 饮水规则的操作性冲突；
- SAP safety-only 描述与 PK 分析范围。
