# 第六部分 技术方案、系统架构与正确性机制

本部分说明 TrialCompiler 如何把大模型的语义能力约束在一个可追溯、可验证、可由专业人员
接管的临床文档流程中。系统的目标不是让模型独立完成临床设计，而是减少资料整理、重复核对
和变更定位工作，同时保留医学、统计、注册、临床运营与质量人员的最终决策权。技术路线由
四个相互依赖的核心构成：版本化临床事实、显式文档依赖图、不确定性感知的多 Agent 工作流，
以及不能被模型高分绕过的人工与机器质量门。

## 6.1 总体分层技术架构

### 6.1.1 架构目标与设计约束

TrialCompiler 的技术架构首先服务于临床文档的责任边界，而不是单纯追求模型调用吞吐量。
临床方案中的终点、样本量、人群、给药、访视和统计原则会跨多份文件重复出现；一项变化既
可能需要传播，也可能因为文档目的不同而合法保留差异。因此，系统必须同时解决五个问题：
来源是否可信、事实当前处于什么状态、哪些文档依赖该事实、系统对判断有多大不确定性、谁有
权批准下一步动作。普通“上传文件—调用模型—返回文本”的架构无法稳定回答这些问题。

架构遵守七项设计原则。第一，来源先于生成，任何关键事实都要能回到授权资料；第二，事实
先于章节，将同一设计信息从重复文本提升为共享对象；第三，增量修订优于整章重写，只改变
受影响的最小范围；第四，生成、评价与批准职责分离；第五，硬约束不能被语言模型软评分抵消；
第六，不确定性必须能够触发取证、拒答或人工升级；第七，所有状态变化必须能够回放、复检和
回滚。这些原则同时决定了系统的模块边界、数据契约和工作流状态机。

### 6.1.2 三层业务架构

系统采用“飞书 Aily 协同入口—TrialCompiler 智能处理后端—企业正式质量体系”的三层业务
架构。三层之间通过结构化任务、候选结果、人工决定和受控文件接口连接，但不共享同一种权威。

飞书 Aily 是面向业务角色的协同入口。医学、统计、注册、临床运营和质量人员可以在飞书中
发起方案编制、既有文件审核或事实变更任务，上传授权资料，回答系统补问，查看事实确认卡、
风险卡和影响清单，并完成任务认领与审批留痕。飞书保存的是协作状态和人机交互，不承担正式
电子签名、受控版本发布或法规记录归档。即使用户在飞书中点击“同意处理”，系统也必须区分
这是事实批准、修订批准、仅同意继续取证，还是要求人工编辑。

TrialCompiler 后端承担智能处理和工程控制。它把飞书输入编译为类型化任务，登记来源，构建
Trial Fact Sheet 和 Clinical Document Graph，编排 Agent，调用确定性校验器与受治理语义
模型，生成 finding、影响矩阵、Repair Proposal、Decision Request 和审计记录。后端输出
始终是候选对象或验证结果，不具有正式文件权威。后端可以证明补丁是否满足已编码约束，但
不能替医学人员决定治疗风险、替统计人员选择 estimand、或替注册人员承诺监管路径。

企业 DMS 或既有质量体系是正式文件权威层。经相应角色审核的候选内容进入该层后，继续执行
组织已有的权限、电子签名、受控发布、归档、培训和版本治理。TrialCompiler 不重新发明这些
成熟能力，而是向正式系统提供来源充分、影响范围明确、已通过沙箱复检的候选修订和审计包。
这种分层降低了替换现有系统的实施风险，也防止交互平台或 AI 模块被误用为最终批准系统。

### 6.1.3 智能后端的八个核心模块

后端不是一个从输入到文本的单模型黑箱，而是由八个相互约束的模块组成。

1. **任务接入与契约模块。** 将用户输入转换为包含项目、任务类型、文档范围、适应证、阶段、
   允许来源、责任角色和输出要求的任务契约。缺少关键字段时先补问，而不是让模型猜测。
2. **来源登记模块。** 为文件和外部记录生成稳定 `source_id`，保存定位地址、版本、权威等级、
   适用范围、文件摘要和访问边界。相同文件重复上传时可通过摘要识别，内容变化则形成新版本。
3. **事实治理模块。** 将终点、样本量、研究人群、入排标准、给药方案、访视安排和统计原则
   表示为 FactRecord，保留单位、来源、状态、责任人、前值和版本。
4. **文档依赖图模块。** 建立事实到 Protocol、SoA、SAP、ICF、CRF 和注册记录的显式边，
   支持从事实找章节、从 finding 找事实、从修订找证据。
5. **Agent 编排模块。** 通过状态图组织上下文确认、证据整理、候选生成、缺陷扫描、修订建议、
   独立复核和人工请求。每个节点读取和写入受约束字段，避免任意覆盖全局状态。
6. **规则与语义审核模块。** 规则负责标识符、数字、日期、严格边界、旧值残留和来源状态；
   语义模型负责发现逻辑歧义、跨文件含义差异和需要专业判断的候选。
7. **不确定性与取证模块。** 维护候选解释的 belief，比较提交、暂缓、读取新证据和请求专家的
   预期收益与成本，并记录为何采取某个动作。
8. **质量、审计与复现模块。** 运行独立 scorer、负对照和回归测试，保存模型与 Prompt 摘要、
   输入摘要、Git commit、finding、补丁、人工决定、前后文档摘要和门禁结果。

模块之间不通过未经约束的长文本传递权威状态。例如语义审核可以提交新 finding，但不能直接
把 FactRecord 改为 approved；修订模块可以提交 Repair Proposal，但不能直接覆盖正式文件；
经验模块可以产生候选经验，但必须等待批准后才能进入下一项目上下文。

### 6.1.4 控制面、数据面与评测面的分离

从系统内部看，可进一步划分为数据面、控制面和评测面。数据面承载来源、事实、章节、文档图、
候选文本和补丁；控制面承载任务状态、角色权限、人工决定、质量门、重试和失败关闭；评测面
承载 Gold、隐藏参考资料、独立 scorer 和实验 split。三者必须隔离。

数据面中的模型输出不能改变控制面的批准状态；控制面中的人工批准必须指向具体事实或补丁，
而不能成为无范围的“全部通过”；评测面的 Gold 不能流入生成上下文，否则模型准确率将因
答案泄漏失去意义。当前代码通过类型化对象、可见性目录、checkpoint 摘要和测试断言落实该
分离。未来企业部署还应把这种逻辑隔离扩展为服务权限和存储权限隔离。

### 6.1.5 组件接口与失败边界

飞书向后端提交的是 intake contract，而不是随意拼接的 Prompt；后端向飞书返回的是事实卡、
finding 卡、影响矩阵、Decision Request 和运行状态；后端向 DMS 提供的是经人工审核的候选
文件及审计包，而不是未经确认的模型响应。每个接口都应包含项目、版本、对象 ID、来源摘要、
状态和时间，支持幂等处理和重复请求识别。

当依赖文件缺失、摘要不一致、未知 observation、角色无权限、模型输出 schema 无效或机器门
失败时，系统采用 fail-closed：保留现有权威状态，记录失败原因，阻止继续发布，并把可恢复
问题转为补问或整改任务。模型超时可以重试，但专业决定缺失不能通过重试模型解决；语义解析
失败可以降级为人工阅读，但不能伪造结构化事实。

### 6.1.6 一次任务如何穿过各层

以“主要终点评估时间由 Week 24 候选变更为 Week 28”为例：用户在飞书发起变更并上传医学
决定依据；接入模块锁定项目和允许资料；来源模块登记新决定；事实模块建立 proposed_change，
保留 previous value；依赖图返回受影响的 Synopsis、endpoint 章节、SoA、SAP、ICF 和注册
记录；规则模块查找 Week 24 残留；语义模块判断不同文档中的表述是否同 scope；修订模块只对
证据充分的位置生成候选红线；沙箱复检检查 Week 28 是否出现、Week 24 是否在应改位置消失、
合法历史表述是否保留；医学、统计和注册角色分别处理自己的 Decision Request；全部硬条件
满足后，候选文件才进入企业 DMS 的正式签核。整条链上没有任何一步允许模型直接发布。

| 层级 | 主要组件 | 输入 | 输出 | 明确不承担的职责 |
|---|---|---|---|---|
| 协同入口 | 飞书 Aily、任务卡、审批卡 | 用户任务、授权资料、人工反馈 | 结构化任务、待确认事实、审批记录 | 不负责正式文件发布 |
| 智能后端 | Agent 编排、事实层、依赖图、规则与模型 | 结构化任务、来源、已批准经验 | 候选事实、草稿、finding、repair、审核包 | 不替代专业批准 |
| 质量与评测 | 确定性门、独立 scorer、回归测试 | 候选输出、Gold、负对照 | 指标、阻断状态、整改工单 | 不用软分覆盖硬失败 |
| 正式系统 | 企业 DMS、质量体系、电子签名 | 经专业审核的候选文件 | 受控版本、签核、发布与归档 | 不把 AI 输出默认视为批准内容 |

## 6.2 受控数据流

### 6.2.1 数据对象的生命周期

受控数据流的基本单位不是一次聊天消息，而是具有 ID、来源、版本和状态的对象。原始资料先
形成 SourceRef；从资料中识别出的信息形成 draft FactRecord；专业确认后事实可进入 approved，
发现冲突时进入 requires_human_review，提出新值时进入 proposed_change；事实再与章节形成
依赖边；审查产生 ReviewFinding；可安全修订的问题形成 RepairProposal；需要专业选择的事项
形成 DecisionRequest；最终运行生成审计记录和可复现摘要。每一对象的产生者、消费者和允许
状态迁移都应明确。

一条事实的典型生命周期为：`draft → requires_human_review → approved`，或
`approved → proposed_change → approved/superseded`。被拒绝候选进入 rejected 并保留理由，
而不是物理删除。若新事实生效，前一版本标记 superseded，但仍可用于解释历史文档和回滚。
文档章节也采用候选副本与正式副本分离，AI 不原地覆盖上传文件。

### 6.2.2 阶段一：任务边界与来源冻结

任务开始时，系统先确认项目、任务类型、文档范围、试验阶段、适应证、地域、责任角色、允许
访问的资料和期望输出。边界不完整时，Context Agent 生成缺失问题，例如“本次审核是否包含
SAP”“所给会议纪要是否代表已批准决定”“注册记录以哪个日期快照为准”。在回答前，后续
生成节点不启动，避免把默认假设写进方案。

随后对每份资料计算摘要，记录文件名、类型、版本、来源、日期、适用范围和访问权限。公开
案例还保存 URL、页数、字节数和 SHA-256。冻结摘要的作用是保证同一实验、人工复核和后续
评分指向同一内容；如果注册系统或远程 PDF 更新，应创建新 corpus 版本，不能静默覆盖已经
标注的资料。

### 6.2.3 阶段二：候选事实提取与专业确认

Evidence Agent 从授权资料中提取研究目的、终点、样本量、人群、入排标准、干预、时间点、
安全要求和统计原则。每个候选事实同时保存值、单位、来源 ID、原文定位和模型说明。多个来源
表达不一致时，不立即选择“最像正确”的一个，而是建立候选集合，标记相同 scope、不同 scope
或 scope 不明，并路由给医学、统计、注册或运营角色。

专业人员确认的是具体事实及其范围，而不是笼统确认整份模型输出。例如主要终点需要确认测量
变量、时间框架和目标人群；样本量需要确认总数/每组、计划/实际和脱落调整；给药时间需要
确认连续研究日还是周期内日。只有确认后的事实进入当前 Trial Fact Sheet。该事实表成为生成、
审核和变更传播共享的单一基线。

### 6.2.4 阶段三A：受控方案生成流

生成任务只读取当前阶段允许的来源、已确认事实、授权模板、适用规则和已批准经验。生成 Agent
按章节工作，每个章节输出正文、使用的 fact IDs、source IDs、未决问题和限制。系统先生成
Synopsis 或章节骨架，再逐节扩展，避免一次性长文本生成后无法定位事实来源。

每节生成后立即运行局部检查：是否引用未批准事实、数值和单位是否与事实表一致、必需字段是否
缺失、是否把模拟或历史资料提升为监管结论。Phase 2 新证据到达时，系统不重新生成整份方案，
而是先更新事实、计算影响集合、增量修订受影响章节。隐藏参考资料在文本冻结后才进入 evaluator，
不能作为生成答案来源。

### 6.2.5 阶段三B：既有文档审核流

审核任务先将 Protocol、SAP、ICF、CRF 和注册记录解析为带来源定位的文档单元，再把其中的
事实表达对齐到 Trial Fact Sheet。确定性规则扫描标识符、数值、日期、Week、严格不等式、
旧值和来源链；语义模型寻找分析人群、终点、缺失数据、终末事件和监管语义候选。

一个候选不能仅因两个数字不同就成为缺陷。系统必须检查它们是否属于同一研究、同一版本、
同一人群、同一时间框架和同一统计含义。不能确认 scope 时输出 insufficient evidence 或
requires qualified review，而不是自动确认。每条 finding 保存严重程度、目标章节、相关事实、
证据来源、解释和责任角色，形成可操作的审核队列。

### 6.2.6 阶段三C：事实变更与传播流

变更任务保存旧值、新值、理由、来源和提出者，把事实状态设为 proposed_change。依赖图计算
所有直接引用单元，规则扫描其中的旧值残留，语义模块补充可能存在的隐式影响。影响结果按
“必须更新、可能更新、需要专家判断、合法历史保留”分级，而不是把所有旧值一律替换。

Repair Proposal 对每个目标章节保存原文和候选文本。非重叠、事实唯一且规则支持的补丁可在
人工批准后进入沙箱；重叠编辑、身份字段、医学含义不唯一或证据不足的项目形成 Decision
Request。系统不会因为人工说“这个问题要改”就推断具体改法；授权对象和编辑内容必须分别
明确。

### 6.2.7 阶段四：沙箱复检与人工决策

补丁只应用于文档副本。复检至少回答七个问题：规范新值是否出现；旧值是否从应改位置移除；
修改章节是否等于批准范围；非目标章节是否保持摘要不变；事实和来源引用是否保持；原 finding
是否关闭；是否产生新的高风险 finding。任一条件失败，补丁不能进入下一阶段。

人工决策按职责拆分。医学角色判断研究设计和安全性，统计角色判断样本量、estimand、分析集、
缺失数据与多重性，注册角色判断申报口径和监管表达，运营角色判断访视及执行可行性，质量角色
检查状态、版本和审计完整性。系统可以推荐责任角色，但不能用一个角色的批准替代另一角色的
职责。多角色意见冲突时保留所有意见并升级，不自动多数表决。

### 6.2.8 阶段五：正式衔接、经验候选与审计

全部机器退出条件满足且人工决定完成后，系统输出候选受控文件、变更摘要、影响矩阵、来源清单、
复检结果和批准记录，交由企业 DMS 执行正式签核。TrialCompiler 记录“具备进入正式流程的
条件”，而不自行把文件标记为监管批准或正式生效。

运行中的 finding、修订方法和人工理由可以形成经验候选，但不会自动写入长期知识库。经验需
再次确认来源、适用范围、版本和责任人；只有 approved 且仍有效的经验可被后续任务检索。
审计记录保存每个对象的前后状态、操作者、时间、输入输出摘要和门禁结果，使整个过程能够重放。

### 6.2.9 异常流与失败关闭

受控数据流必须描述失败路径。来源缺失时停止事实批准；文件解析失败时保留原文件并请求人工
定位；模型返回无效 schema 时拒绝写入类型化状态；发现未知 observation 时停止 belief 更新；
case split 泄漏或冻结摘要变化时禁止输出评测声明；补丁回归失败时恢复沙箱前摘要；人工决定
超时则保持 pending，而不是默认同意。

对于可恢复技术错误，系统可按固定次数重试并记录每次结果；对于专业证据不足，增加模型调用
次数不能视为解决方案，必须获取新资料或请求相应人员。这样的 fail-closed 设计使“不知道”
成为一种合法、可审计的状态，而不是被语言模型补全掩盖。

### 6.2.10 数据可见性与隐私边界

不同阶段和角色只访问完成任务所需的最小数据。生成 Agent 不访问 Gold；评估器不把隐藏参考
资料回传生成节点；项目 A 的经验只有在被批准为更广范围且权限允许时才可用于项目 B；飞书
只展示当前用户有权查看的事实和文档定位。真实部署还应对原文件、结构化事实、模型调用日志、
审计日志和备份分别设置权限与保留期限。

当前验证只使用公开研究层资料和完全合成材料，不包含患者级数据。这一事实不能被外推为系统
已经通过真实敏感数据安全验证；它只说明原型阶段主动缩小了数据风险面。企业 POC 必须另行
验证脱敏、隔离、加密、访问控制、删除和事件响应流程。

### 6.2.11 三类主流程与共同质量链总结

生成流、审核流和变更流具有不同业务输入，但共享同一套事实、来源、依赖和门禁。生成流关注
“依据什么写”；审核流关注“哪些表达可能不一致”；变更流关注“修改应传播到哪里”。三类流程
最终都收敛到：确定性检查—语义候选—不确定性处置—沙箱修订—重新检查—人工决定—正式
衔接—审计归档。共享质量链保证系统不会在某一种任务中放弃来源、状态或人工边界。

| 阶段 | 主要输入 | 核心处理 | 主要输出 | 失败时状态 |
|---|---|---|---|---|
| 边界确认 | 用户目标、项目资料 | 补问、权限和 scope 确认 | Intake contract | blocked_missing_context |
| 来源冻结 | 文件、注册记录 | 登记、摘要、版本锁定 | Source manifest | blocked_source_error |
| 事实治理 | 来源与原文 | 提取、对齐、人工确认 | Trial Fact Sheet | pending_human_review |
| 生成/审核 | 事实、章节、规则 | Agent 与确定性检查 | 草稿、finding | blocked_machine_gate |
| 变更传播 | proposed change、依赖图 | 影响计算、候选补丁 | Impact matrix、repair | pending_qualified_decision |
| 沙箱复检 | 补丁与负对照 | 值、范围、来源、回归检查 | Revalidation report | repair_failed |
| 正式衔接 | 通过的候选与审批 | 打包审计证据 | DMS candidate package | release_not_authorized |

## 6.3 来源登记、事实基线与证据追溯

### 6.3.1 为什么要建立独立事实层

临床设计信息通常以不同形式重复出现在方案摘要、正文、SoA、SAP、ICF、CRF 和注册记录中。
如果系统只保存各章节文本，就无法区分“同一事实的多种表达”和“多个碰巧数值相同的事实”，
也无法回答某个值究竟来自哪份资料、是否已经批准、修改后应传播到哪些位置。Trial Fact Sheet
因此不是对文档的又一次摘要，而是文档生成、审核和修订共同依赖的规范事实层。

事实层采用“候选事实—专业确认—当前事实—历史版本”的治理方式。模型从来源中提取的是
candidate，不是权威答案；专业人员确认时需要同时确认值、单位、适用人群、时间框架、版本和
来源；当前事实用于驱动候选文档；旧版本继续保留，用于解释历史文件、审计变更和执行回滚。
这种设计将“文本是否流畅”和“事实是否有效”分离，避免模型通过更自然的措辞掩盖状态问题。

### 6.3.2 来源身份、权威等级与适用范围

每个来源表示为 SourceRef，至少记录 `source_id`、标题、locator、authority、version、
effective_date 和 access_scope。locator 可以是公开 URL、文件路径、页码、章节、表格或单元格；
authority 表示监管指南、企业 SOP、项目决定、公开注册记录、历史项目或合成测试资料等来源
性质；access_scope 表示企业、治疗领域、项目、阶段或 evaluator-only 等可见性边界。

来源权威不是一个简单的总排序。监管指南可能在一般原则上权威，但不能替代项目医学决定；
企业模板规定文档结构，却不能证明某项样本量；ClinicalTrials.gov actual enrollment 与方案
planned enrollment 可能同时正确；模拟 FDA/CDE 会议纪要只能作为测试输入，不能被提升为
真实监管认可。因此系统比较来源时必须同时考虑 authority、scope、version 和语义用途，而
不是永远选择“级别最高”或“日期最新”的文本。

对于二进制文件，系统保存 SHA-256、字节数和页数；对于注册记录，保存获取时间和快照摘要；
对于模型可见的文本片段，保留原始来源 ID 和定位。摘要用于检测静默变化，不能代替来源内容。
同一 locator 内容变化时建立新版本，已经用于 Gold 或人工裁决的旧版本不得被覆盖。

### 6.3.3 FactRecord 的字段与语义

FactRecord 除 `fact_id` 和 value 外，还记录 name、unit、status、source_ids、owner_role、version、
updated_at 和 previous_value。实际扩展时还需要记录人口范围、时间框架、估计目标组成、适用
地域和结构化数据类型。例如“220”只有与“总随机样本量”“planned”“双臂合计”“含 15% 脱落”
一起记录才构成可比较事实；“Week 24”需要说明是主要终点评估、筛选窗口还是随访结束时间。

同一事实可以关联多个来源，多个来源也可以支持同一事实。系统不要求所有来源措辞相同，而是
要求能够解释它们的关系：一致支持、补充说明、版本替代、合法范围差异或待调和冲突。当来源
冲突时，系统保留所有候选值和证据，不以简单投票决定事实。Decision Request 应说明冲突位置、
可选处理、影响文件和所需角色。

### 6.3.4 状态机与允许的迁移

事实状态集合包括 draft、proposed_change、requires_human_review、approved、rejected 和
superseded。draft 表示尚未完成专业确认；requires_human_review 表示证据或语义无法由当前
规则唯一裁决；approved 表示相应责任角色已经确认该事实可在指定 scope 中使用；rejected
表示候选不被采纳但仍保留理由；proposed_change 表示相对于当前批准事实提出了新版本；
superseded 表示旧版本已经被新批准版本取代，但仍属于审计历史。

允许的主要迁移为：

```text
draft → approved / requires_human_review / rejected
requires_human_review → approved / rejected
approved → proposed_change
proposed_change → approved / rejected / requires_human_review
旧 approved → superseded（仅在新版本正式批准后）
```

不允许的迁移包括：模型置信度直接使 draft 变 approved；新事实尚未批准就把旧事实标记为
superseded；补丁应用成功就自动批准事实；缺少 owner role 或来源时进入 approved。状态迁移
必须记录操作者、时间、理由和前后摘要，以支持审计和重放。

### 6.3.5 文档单元、事实引用与证据路径

DocumentSection 表示可独立定位和修订的文档单元，记录 section_id、title、text、document_type、
fact_refs 和 source_ids。单元粒度可以是章节、段落、表格行、单元格或注册字段；粒度过粗会
导致影响范围扩大和无关改写，粒度过细则增加解析和维护成本。当前原型以章节和页级定位为主，
企业 POC 应逐步扩展至 SoA 单元格、SAP 规则和 CRF 字段级。

一条完整证据路径可以表示为：

```text
原始文件/注册快照
→ SourceRef 与 locator
→ FactRecord 与状态
→ DocumentSection.fact_refs
→ ReviewFinding
→ RepairProposal
→ DecisionRequest
→ 沙箱复检与最终人工决定
```

审核人员因此能够从一个报警向上追溯到原始证据，也能从一个事实向下查看所有受影响文档。
如果路径中任一必要边缺失，系统可以继续保存候选，但不能声称具备完整可追溯性。

### 6.3.6 事实冲突调和规则

系统发现不同表达后，先进行规范化而不是直接比较字符串。规范化包括单位转换、大小写、日期
格式、Week 表达、范围边界和标识符标准化。随后判断 scope：是否同一研究、同一人群、同一
版本、同一时间框架、同一估计目标和同一文件用途。只有 scope 相同且规范化值不兼容时，才
有条件成为 confirmed conflict。

差异至少分为五类：完全一致；语义等价；合法 scope 差异；版本先后关系；证据不足或真实
冲突。例如 Protocol 的 planned enrollment 与注册结果的 actual enrollment 属合法状态差异；
Protocol 连续研究日与 CRF 周期内 Day 1 可由映射解释；旧方案与已批准修订值属于版本关系；
主要终点时间在同一版本中分别写 Week 24 与 Week 28 才可能是真实冲突。无法分类时进入人工
调和，而不是默认选择一个值。

### 6.3.7 证据完整性与公开语料实现

当前 50 例公开语料对 65 份文件保存官方 URL、字节数、页数和 SHA-256，并对 50 份注册记录
保存快照和摘要。自然候选进一步保存文件名、页码、短原文、candidate ID 和 digest。一个 PDF
因正文不可提取被拒绝并由后续案例替补，说明来源进入 corpus 前需要通过完整性门，而不是只要
下载成功就算有效。

系统分别评价来源完整、定位完整和语义正确。文件摘要一致只能证明评测对象没有变化；页码
完整只能证明候选可查找；二者都不能证明候选是临床缺陷。反过来，即使专业人员认为问题真实，
若没有记录判断所用的来源与范围，该结果也不适合作为可复现 Gold。

| 对象 | 核心字段 | 在流程中的作用 |
|---|---|---|
| SourceRef | source_id、locator、version、authority、scope | 锁定证据身份和适用边界 |
| FactRecord | fact_id、value、unit、status、source_ids、previous_value | 形成可版本化事实基线 |
| DocumentSection | section_id、text、fact_refs、source_ids | 把事实连接到具体文档单元 |
| ReviewFinding | type、severity、section_ids、fact_ids、evidence | 表达可审计的问题候选 |
| RepairProposal | original、proposed、rationale、fact_ids | 表达待批准的最小修订 |
| DecisionRequest | owner role、evidence、options、status | 将专业决定交还相应责任人 |

系统设置三项基本不变量：任何已批准事实必须具有有效来源；任何正式候选修订必须追溯到
finding 和事实；任何 `requires_human_review` 状态不能由模型置信度直接提升为 approved。
若不变量失败，工作流必须阻断并形成审计记录。

## 6.4 多 Agent 工作流与职责分离

### 6.4.1 多 Agent 的必要性

TrialCompiler 使用多 Agent 不是为了让多个模型互相讨论后取多数意见，而是为了实施职责分离、
上下文最小化和独立复核。单个长 Prompt 同时承担来源理解、事实提取、章节生成、质量评价和
最终建议时，容易产生三类偏差：生成者会为自己的文本寻找合理解释；评价者看到生成思路后
难以保持独立；模型可能把高置信度误当成批准。多 Agent 将这些行为拆成可测试节点，并用共享
状态和确定性门约束节点之间的关系。

Agent 也不是组织岗位的替代品。“统计审核 Agent”可以发现样本量无法复现或分析集定义不一致，
但不能成为法定统计负责人；“注册 Agent”可以发现监管表达缺乏来源，但不能代表监管机构。
Agent 名称表示检查视角和工具权限，最终专业决定仍由对应人员作出。

### 6.4.2 Agent 角色与职责矩阵

| Agent/节点 | 主要职责 | 允许读取 | 允许写入 | 明确禁止 |
|---|---|---|---|---|
| Context Agent | 确认任务、scope、角色和缺失信息 | Intake、项目元数据 | 任务契约、补问 | 猜测缺失临床事实 |
| Evidence Agent | 登记来源、提取候选事实与定位 | 授权原文、来源清单 | SourceRef、draft facts | 批准事实 |
| Generation Agent | 基于已确认事实生成候选章节 | approved facts、模板、批准经验 | candidate sections | 读取 Gold、正式发布 |
| Review Agent | 发现确定性和语义候选问题 | 候选文档、事实、来源 | ReviewFinding | 自行修订并批准 |
| Repair Agent | 生成最小修订和影响清单 | finding、事实、依赖图 | RepairProposal | 覆盖正式文件 |
| Quality Agent | 在沙箱中独立复检 | 原文、补丁、负对照 | revalidation result | 用软分忽略硬失败 |
| Governance Agent | 路由人工决定、形成审计包 | finding、proposal、角色规则 | DecisionRequest、状态建议 | 冒充专业批准人 |
| Experience Agent | 整理可复用经验候选 | 运行结果、人工理由 | experience candidate | 未批准即写入长期记忆 |

同一底层模型可以在原型阶段承担多个 Agent 的语言推理，但必须使用不同节点契约、可见上下文、
提示和输出 schema；否则只是给同一个模型换名字，不能形成有效职责分离。高风险生产场景还可
进一步使用不同模型、规则引擎或人工 reviewer 形成更强独立性。

### 6.4.3 共享状态而非自由聊天

各 Agent 通过受控 workflow state 交换信息，而不是依赖完整对话历史。共享状态至少包含
task_context、sources、facts、sections、findings、repair_proposals、decision_requests、
quality_results、audit_events 和 workflow_status。节点只能更新被授权字段，状态合并时检查
对象 ID、版本和 schema。

例如 Review Agent 返回的是带 finding_type、severity、section_ids、fact_ids、source_ids
和 explanation 的结构化列表，而不是一句“方案可能有问题”；Repair Agent 必须引用 finding
ID 和事实 ID；Quality Agent 必须报告目标 finding、剩余 finding、新增 finding 和负对照状态。
结构化状态让每个节点可以被单测、重放和替换，也使人工能够理解责任链。

### 6.4.4 工作流节点与路由条件

主审核工作流可以概括为 A—H 八类节点：A 确认上下文，B 获取证据与经验，C 建立事实和依赖，
D 执行确定性与语义审核，E 生成候选修订，F 在沙箱独立复检，G 形成专业决策请求，H 生成
审计与报告。实际执行不是每次都机械经过全部节点，而是由状态条件路由。

如果 Context Agent 发现关键字段缺失，路由回飞书补问；来源解析失败，路由人工定位；没有
finding 时仍需检查覆盖范围，不能直接 vacuous pass；finding 可由批准事实唯一修复时进入
Repair Agent；finding 涉及医学、统计或注册选择时跳过自动补丁，直接生成 Decision Request；
沙箱出现新问题时回到 Review/Repair；所有退出条件满足后才能进入最终报告。

典型路由可表示为：

```text
context incomplete → ask human → resume
source invalid → blocked_source_error
facts uncertain → qualified fact review
deterministic repairable finding → proposal → approval → sandbox
semantic/non-unique finding → Decision Request
regression detected → repair_failed → revise/reject
all gates satisfied → candidate package（仍非正式发布）
```

### 6.4.5 并行检查、汇合与冲突处理

在来源和事实基线冻结后，医学、统计、注册、运营和质量视角可以并行执行，以减少等待并扩大
覆盖。并行节点只能新增 finding 或提出候选建议，不能各自修改同一正文。汇合节点按 finding ID、
事实 ID、章节和语义范围去重；表述不同但指向同一根因的 finding 可聚合，证据或结论相反的
finding 保留为冲突。

修订操作在应用前执行区间冲突检查。非重叠补丁可按高偏移到低偏移合并；重叠补丁、相同事实
的不同候选值或不同角色的相反决定不自动投票，而是形成冲突包。冲突包包括双方理由、来源、
影响范围和可选方案，由有权角色调和。这样并行 Agent 提高发现能力，却不会造成“最后写入者
覆盖前者”的非确定行为。

### 6.4.6 人工中断、恢复与 Decision Request

工作流遇到专业问题时进入可持久化中断状态，而不是结束整个任务或让模型继续猜测。Decision
Request 记录问题、选项、建议责任角色、相关事实、证据、受影响文档、系统不确定性和各选项
后果。人工可以批准候选事实、拒绝候选、要求补充资料、选择具体修订或指派其他角色。

恢复执行时，系统验证决定者角色、决定对象版本和输入摘要。如果等待期间来源或文档已变化，
旧决定不能无条件应用，必须重新确认 scope。恢复节点只消费明确决定，不从自由文本中推断
“大概同意”。所有人工输入进入审计日志，后续复检能够区分机器发现、机器建议和人工授权。

### 6.4.7 独立质量复核与反自洽机制

项目早期暴露出 LLM Judge 自洽性偏差：生成文本可在语言、结构和表面完整性上获得高分，但
模拟医学与统计审核仍发现样本量、estimand、终末事件和监管依据问题。因此 Quality Agent
不能只询问另一个模型“这份文档好吗”，而要组合独立规则、Gold scorer、负对照、补丁前后
摘要和未决人工状态。

质量门至少检查：来源完整性、事实状态、阻断 finding、补丁局部性、负对照保护、摘要一致性、
人工授权和回滚信息。软评分只作为补充观察，不能进入 Release 的补偿项。生成 Agent 的思维或
自评也不直接提供给独立 scorer，以减少评价被生成叙事锚定。

### 6.4.8 错误恢复、重试与幂等性

不同错误采取不同恢复策略。网络超时、临时模型故障和可重试解析错误可有限重试；重复调用使用
task ID、node ID 和输入摘要识别，避免产生两份事实或重复应用补丁。schema 错误先经过严格
解析与一次受限修复，仍无效则失败关闭。节点输出写入 checkpoint，恢复时从最后一个完整状态
继续，而不是重新运行所有前序节点。

专业证据不足、角色冲突或依赖边未知不属于技术重试问题。对此应获取新来源、补充人工判断或
保持 deferred。系统还设置循环上限和进展条件：若连续迭代没有关闭 finding 或降低不确定性，
停止自主循环并升级人工，避免 Agent 反复改写同一内容。

### 6.4.9 经验候选与记忆准入

Experience Agent 在任务结束后可以将“问题—证据—决定—修订—结果”整理为经验候选，记录
适用治疗领域、试验阶段、文档类型、事实类别、来源、版本和批准者。经验候选与事实一样具有
draft、approved、rejected、superseded 状态。只有来源明确、scope 匹配、仍有效且获批准的
经验才能进入新任务检索。

检索时先过滤权限、状态和 scope，再进行语义相似度排序；相似并不等于适用。模型必须看到
经验的来源和限制，不能只看到去上下文的结论。监管规则更新、企业 SOP 变化或项目结束后，
经验可以被 supersede 或缩小范围。该机制防止错误修订因“过去用过”而被自动复制。

### 6.4.10 为什么该架构仍然是 AI 系统而非普通流程引擎

确定性工作流负责状态、安全和可复现边界，但系统的核心难点仍包含 AI 推理：从长篇非结构化
文档提取候选事实；识别不同表述是否语义等价；发现未显式建边的潜在依赖；在多个解释之间
维护不确定性；根据预期信息增益选择下一份证据；生成与原文风格一致的局部候选修订；形成供
不同专业角色理解的解释。流程引擎无法仅靠固定字段完成这些任务。

TrialCompiler 的研究贡献正是让这些概率性能力运行在显式状态、证据和人工边界内。规则层
不取代 AI，AI 也不绕过规则；Agent 提供主动语义能力，图和状态机提供可验证约束。最终形成
的是“概率推理负责发现和建议、确定性系统负责约束和复检、专业人员负责授权”的混合系统。

## 6.5 临床文档依赖图、影响计算与变更传播

前文的事实层解决“哪个值在当前版本中有效”，本节进一步解决“这个值改变后，哪些文档必须
跟着改变、哪些内容只能提示专家、怎样证明修订没有越界”。因此依赖图不是一个用于展示的
静态关系图，而是编译、审查、整改和回归验证共同使用的中间表示。原目录中“影响计算”和
“原子修订”本质上属于同一传播链，本节将两者合并说明；第 6.8 节只讨论人工授权和发布治理，
第 6.9 节则证明这里定义的传播与补丁操作在明确假设下具有何种数学性质。

### 6.5.1 图对象、边类型与证据状态

设来源、规范事实、文档单元、finding、修订提案和人工决定的集合分别为
$\mathcal{S},\mathcal{F},\mathcal{U},\mathcal{I},\mathcal{R},\mathcal{H}$，完整图为

$$
G=(V,E),\qquad
V=\mathcal{S}\cup\mathcal{F}\cup\mathcal{U}\cup\mathcal{I}\cup\mathcal{R}\cup\mathcal{H}.
$$

文档单元不是整份文件，而是可定位、可单独比较和可局部修订的最小业务单元，例如 Protocol
中的入排标准条款、SoA 的一个访视行、SAP 的分析集定义、ICF 的风险段落、CRF 的字段约束或
注册页面中的 endpoint 字段。边至少包括 `source_of`、`depends_on`、`identifies`、`repairs`
和 `authorized_by`。每条边保存

$$
e=(from,to,type,status,evidence,created\_by,version),
$$

其中 `status` 分为 `explicit`、`candidate`、`approved` 和 `rejected`。明确字段引用、结构化
映射或人工确认的关系可作为有效传播边；仅由向量相似度或大模型推断得到的关系保持 candidate，
不能直接驱动正式改文。这样可避免把“语义上看起来有关”误当成已获授权的事实依赖。

### 6.5.2 依赖图的构建与增量维护

依赖图按四层证据逐步建立。第一层是结构化显式边，例如事实 `primary_endpoint.timepoint` 被
Protocol endpoint 段、SoA 第 24 周行、SAP 主分析时间点和注册结果指标共同引用。第二层是
可确定抽取边，通过 NCT 标识符、日期、周数、样本量、剂量、访视编号以及表格字段映射建立。
第三层是自然语义候选边：模型从自由文本中发现可能涉及 estimand、分析人群、缺失数据、终末
事件或多重性的段落，但只记录候选、页码、原句和理由。第四层是专家裁决与人工补边；被确认的
候选升级为 approved，被否决的候选保留为 rejected 负样本，供后续回归测试使用。

图随事实版本增量维护，而不是每次重新扫描并覆盖历史。新增事实产生新节点；事实替代使用
`supersedes` 关系连接旧、新版本；文档换版时，单元通过稳定章节标识、标题路径和内容摘要进行
重定位。若定位失败，系统将相关边标记为 `orphaned` 并阻断自动补丁，而不是把旧偏移量套到
新文件。由此可以区分“事实已经改变但文档尚未同步”和“文档结构改变导致原依赖位置失效”。

### 6.5.3 直接影响、传递影响与有效影响集

定义事实到文档单元的直接依赖矩阵

$$
A\in\{0,1\}^{n\times k},\qquad
A_{ij}=1\iff u_j\text{ 明确引用或依赖 }f_i.
$$

事实 $f_i$ 的直接影响集为

$$
I_{dir}(f_i)=\{u_j\in\mathcal{U}:A_{ij}=1\}.
$$

若某文档单元的结论又被另一单元引用，可再定义单元间依赖矩阵
$B\in\{0,1\}^{k\times k}$。在最大传播深度 $L$ 内，传递可达集为

$$
I_{reach}(f_i)=\left\{u_j:\left[A(I+B+B^2+\cdots+B^L)\right]_{ij}>0\right\}.
$$

实际传播不能无条件使用所有候选边。设 $E_{exp}$ 为显式边，$E_{app}$ 为已批准语义边，
$E_{cand}$ 为尚未裁决的候选边，则有效影响集和待复核影响集分别为

$$
I^*(f)=Reach(f;E_{exp}\cup E_{app}),\qquad
I_{review}(f)=Reach(f;E_{cand})\setminus I^*(f).
$$

$I^*(f)$ 可进入确定性检查和补丁规划；$I_{review}(f)$ 只能生成“可能受影响”的审查任务。
当前原型中的 `fact_to_sections` 和 `impact_set` 实现的是显式直接影响集；传递边与候选语义边
采用上述统一定义，但只有在对应边真实登记或获批后才纳入有效传播。因此公式描述了可扩展的
计算框架，同时没有把尚未实现或未裁决的语义关系包装成已有能力。

### 6.5.4 变更表示、旧值残留与差异分类

一次事实变化不是简单的字符串替换，而表示为

$$
\Delta f=(f,v_0\rightarrow v_1,reason,source,actor,time),
$$

其中旧值、候选新值、变更原因、授权来源、发起角色和时间戳均不可缺失。系统首先计算
$I^*(f)$，再对每个受影响单元计算旧值残留、规范值缺失、边界冲突、来源失配和范围失配等
谓词。以旧值残留为例：

$$
Stale(u,f)=\mathbf{1}[u\in I^*(f)]\cdot
\mathbf{1}[ContainsSameFact(u,v_0)].
$$

这里必须使用 `ContainsSameFact`，而不是无上下文的字符串 `Contains`。例如数字“24”可能
同时表示第 24 周、24 名受试者或表格序号；只有实体、单位、字段和语境均指向同一事实时才能
判为旧值残留。对 NCT ID 也必须区分当前研究身份与相关研究引用，避免把合法引用错误改写为
主试验编号。

每个受影响单元被分到四类处置：`required_update` 表示新值已批准、依赖明确且文本可唯一修改；
`possible_update` 表示存在语义关联但范围或表达尚不唯一；`expert_decision` 表示必须由医学、统计
或注册角色选择策略；`history_only` 表示该单元属于冻结历史版本，只记录差异而不回写。分类同时
保存触发谓词和反证条件，审核人员可以看到它为什么被归类，以及满足什么条件后可以改变处置。

### 6.5.5 确定性传播与专业判断的边界

数字、日期、周数、试验标识符、严格不等式、枚举值和同一事实的旧值残留适合确定性计算。
例如给药发生在第 1、4、7、10 天，相邻间隔均为 3 天，因此“给药间隔大于 3 天”与计划表
冲突，而“间隔不小于 3 天”不冲突。此处必须显式区分 $>3$ 与 $\ge3$，不能交给模糊语义
相似度处理。

相反，estimand、分析人群、缺失数据、多重性、终末事件和注册策略通常不存在唯一的字符串
修复。例如“死亡后缺失值采用多重插补”可能涉及 composite、while-on-treatment 或
hypothetical 等不同 intercurrent-event 策略。系统可以定位矛盾链、列出受影响章节并生成
Decision Request，但不能自动选择统计策略。由此，影响计算负责完整暴露决策面，专业人员负责
确定业务语义；二者不能相互替代。

### 6.5.6 从影响集到最小原子补丁

对 `required_update` 单元，系统生成 Repair Proposal，而不是直接生成一份无法审计的“新文档”。
每个原子编辑表示为

$$
o=(section,start,end,before,after,fact\_ids,source\_ids,precondition),
$$

并记录触发 finding、候选文本、修改理由和目标版本。`precondition` 至少要求目标区间仍等于
`before`、事实仍处于批准状态、来源版本未变化。只要任一前置条件失效，补丁就必须重新计算。
这能阻止多人并行编辑后把旧补丁错误应用到新文档。

补丁遵循最小范围原则：只修改能够由批准事实唯一确定的表达，不顺带润色邻近段落。多个补丁
区间不重叠且前置文本一致时，可以按偏移量从后向前合并；区间重叠、同一位置存在不同候选值、
目标范围不清或存在多种合理解释时，系统产生冲突对象并停止自动应用。补丁先作用于沙箱副本，
原始文档及其摘要保持不变。

### 6.5.7 传播后的重新编译与闭环判定

“补丁成功执行”不等于“问题已经解决”。沙箱文档必须重新进入抽取、规范化、依赖计算和全部
确定性检查，相当于一次局部重新编译。对补丁 $p$ 定义六项闭环条件：规范新值已出现
$C_1$；需要移除的位置不再残留旧值 $C_2$；实际修改范围等于获批目标范围 $C_3$；事实—来源—
修订链完整 $C_4$；原 finding 已关闭 $C_5$；没有产生新的阻断 finding $C_6$。只有

$$
Closed(p)=\bigwedge_{r=1}^{6}C_r=1
$$

时，修订才可进入独立质量审核。若新值出现但旧值仍在另一受影响章节，状态是
`partially_repaired`；若 finding 关闭但产生新冲突，则状态是 `regression_detected`；二者均不能
被“补丁已应用”这一技术状态掩盖。

### 6.5.8 端到端传播示例

假设经授权来源确认主要终点时间点由 Week 24 改为 Week 28。系统建立事实变化
`primary_endpoint.timepoint: 24→28`，直接影响集包含 Protocol endpoint 段、SoA 访视行、SAP
主分析规则和注册结果指标；ICF 中若只描述研究总时长且未引用主要终点时间点，则不应因关键词
“week”被纳入。Protocol、SoA 与注册字段中的同一事实旧值可形成 `required_update`；SAP 若还
涉及分析窗、缺失处理或 estimand，则进入 `expert_decision`。获批补丁只改三个确定位置，随后
重新运行 Week 值一致性、访视窗、来源链和旧值残留检查。只有四个依赖单元全部获得明确处置、
三个补丁通过六项闭环条件、SAP 决策由合格统计角色完成，整项变更才能结束。

### 6.5.9 完备性边界与覆盖扩展

依赖图的完备性是条件化的：只有已登记的事实和有效边才能保证传播。遗漏事实、扫描质量差、
章节重定位失败、隐含语义未被发现或候选边未获裁决，都可能造成漏传；过宽的语义边则会造成
误传。因此报告不声称对任意临床文档的隐含依赖具有理论上的完全召回。

工程上通过来源完整性检查、结构化字段映射、自然候选挖掘、人工补边、rejected 负样本、图覆盖
统计和回归测试持续扩大边集合。每次人工确认或否决都进入可追溯样本库，用于检验后续版本是否
仍能找到同一依赖、是否错误扩散到负对照。未知依赖被显式保留为风险和待审任务，而不是由一个
高置信度数字隐藏。

## 6.6 不确定性与可解释性机制

TrialCompiler 的 AI 研究核心不是让模型为一段答案附加一个 confidence，而是研究概率型 Agent
如何在多文档、证据不完整且来源可能冲突的环境中形成竞争假设，主动选择下一项证据，并在无法
安全收敛时拒绝提交。系统因此把不确定性视为能够改变行动和发布状态的计算对象；可解释性则不是
自动生成一段理由，而是要求 finding、动作和候选补丁携带可重放、可干预和可证伪的证据。

### 6.6.1 先区分诊断信号、不确定性与授权状态

来源冲突、证据缺失、模型分歧、影响范围未知和权限不足在界面上都可能表现为“无法确定”，但
它们并不是同一种量。来源冲突和证据缺失是观测到的诊断信号；权限不足是治理状态；多次采样
不一致是估计信号；只有针对明确事件、明确层级定义的分布或风险，才是不确定性估计。否则一个
没有语义对象的“87% confidence”既不能校准，也不能决定该读取哪份文件。

本项目从两个正交维度描述不确定性。第一维是性质：epistemic uncertainty 来自知识、来源或
模型能力不足，原则上可以通过取证、验证或专家判断降低；aleatoric uncertainty 来自环境或观测
过程本身的随机性和不可约歧义，不能承诺通过更多推理消除。第二维是位置：不确定性可以位于
observation、belief state、next action 或 trajectory outcome。例如 OCR 对一个数值的识别风险
属于 observation 层；两个规范事实解释的竞争属于 belief 层；下一步应读 SAP 还是注册记录属于
action 层；整条轨迹最终能否生成通过独立复检的补丁属于 outcome 层。

任何数值输出必须同时声明 target event。可接受的事件包括“该候选事实经专业裁决为真”“该
finding 与冻结 Gold 匹配”“该补丁通过独立复检”或“该轨迹无错误提交”；不可接受的写法是脱离
事件只报告“系统置信度”。只有估计器在独立 calibration split 上拟合或选阈值，并在未参与调参
的 test split 上评价后，报告才允许使用 calibrated probability。模型自报概率、reviewer rule
score 和演示数据中的手工 posterior 均不能据此冒充真实校准结果。

### 6.6.2 跨文档事实调和的有限假设状态

对一个候选差异，系统首先建立有限且互斥程度足够的假设集合
$\mathcal{H}=\{h_1,\ldots,h_q\}$。例如 NCT ID 不一致可以对应：$h_1$ 当前文件身份错误，
$h_2$ 文中是合法的相关研究引用，$h_3$ 来源范围配置错误，$h_4$ 证据不足。主要终点时间点
不同则可能对应真实冲突、历史版本、亚组与总人群差异或计划值与实际值差异。假设必须附有支持
证据、反对证据、可证伪条件和允许的下一动作，而不是由模型自由扩写成无限故事。

在时刻 $t$，Agent 维护 belief state

$$
b_t(h)=P(H=h\mid o_{1:t},a_{1:t-1}),\qquad
\sum_{h\in\mathcal{H}}b_t(h)=1,
$$

其熵为

$$
H(b_t)=-\sum_{h\in\mathcal{H}}b_t(h)\log b_t(h).
$$

观察 $o_{t+1}$ 到达后，系统依据显式 observation model 更新 posterior：

$$
b_{t+1}(h)=
\frac{P(o_{t+1}\mid h,a_t)b_t(h)}
{\sum_{h'}P(o_{t+1}\mid h',a_t)b_t(h')}.
$$

该公式要求实验明确给出 prior 和 likelihood，不能让大模型临场编造似是而非的小数。当前原型
使用有限假设和显式 observation-to-posterior 配置验证状态更新、治理和报告链路；这证明算法
管道可执行，不代表 posterior 已由真实临床数据拟合。

### 6.6.3 不确定性估计信号及其适用边界

系统可以组合多种估计信号，但不把它们互相等同。token likelihood 或 NLL 成本低，却会受到
文本长度和措辞影响；verbalized confidence 对黑盒模型易取得，但通常过度自信；多次采样的
self-consistency 可以发现答案不稳定，却无法发现“稳定地答错”；semantic entropy 在语义等价类
上度量采样分散，能减少字面差异的干扰，但仍依赖多次采样和可靠的语义聚类；来源或 verifier
分歧贴近本任务，却必须考虑多个 verifier 的相关错误，不能把多数票直接当概率。

若未来训练 trajectory calibrator，可把早期风险、跨步不确定性梯度、工具失败和证据稳定性作为
特征预测最终补丁是否正确，但必须防止数据泄漏和分布漂移。conformal 方法可以在明确的校准总体、
exchangeability 假设、nonconformity score 和 coverage event 下提供覆盖或风险保证；如果这些
条件没有写清，报告不能只因为使用了 conformal 一词就声称具有分布无关安全保证。

对于长轨迹，还必须说明怎样聚合步骤风险。设第 $t$ 步错误概率估计为 $r_t$，可以报告终态风险
$r_T$、平均风险 $T^{-1}\sum_t r_t$，或最弱步骤风险 $\max_t r_t$。TrialCompiler 的发布门禁优先
使用最弱步骤和关键动作风险，因为平均值可能掩盖一次错误的来源选择、事实覆盖或高风险补丁。

### 6.6.4 从被动置信度显示到主动取证

Agentic UQ 的关键是让不确定性改变下一步动作。系统的动作空间包括读取指定来源、请求补充附件、
增加语义采样或 verifier 预算、调用确定性工具、询问医学/统计/注册专家、生成候选补丁、降低主张
具体度、defer 和 abstain。对候选取证动作 $a$，预期信息增益定义为

$$
EIG(a)=H(b_t)-\mathbb{E}_{o\sim P(o\mid a,b_t)}[H(b_{t+1}\mid a,o)].
$$

若动作成本为 $c(a)$，风险权重为 $\lambda$，则在允许动作集合 $\mathcal{A}_{safe}$ 中选择

$$
a^*=\arg\max_{a\in\mathcal{A}_{safe}}
\{EIG(a)-\lambda c(a)\}.
$$

成本不仅是模型 token，还包括 PDF 页数、外部查询、人工分钟数、等待时间和敏感数据暴露面。
高 EIG 但越权的动作不属于 $\mathcal{A}_{safe}$；低成本但不能区分竞争假设的重复读取也不应被
选择。每个动作记录选择前 belief、预期收益、实际 observation、更新后 belief、实际熵下降、来源
版本和内容摘要，使评估者能判断 Agent 是否真的因不确定性而取证。

### 6.6.5 提交、继续取证、暂缓与人工转交

设最高后验假设为 $h^*$，候选补丁通过独立验证的估计风险为 $R_t$，剩余预算为 $B_t$。停止策略
不是单一 confidence 阈值，而是受硬门禁约束的选择性决策：

$$
Decision_t=
\begin{cases}
commit, & b_t(h^*)\ge\tau_h,\ R_t\le\tau_r,\ HardGate=1;\\
acquire, & \max_a(EIG(a)-\lambda c(a))>0,\ B_t>0;\\
defer, & QualifiedHumanAction\ available;\\
abstain, & \text{otherwise}.
\end{cases}
$$

即使 belief 越过阈值，候选结果仍必须重新进入第 6.5 节的来源、影响范围、原子补丁和沙箱复检，
不能绕过确定性质量门。若连续若干轮没有关闭 finding、没有带来实际熵下降，或重复访问同一来源，
系统应停止自循环并转交人工。这样“不知道”会产生明确的安全动作，而不是变成更长的自然语言
推理。

### 6.6.6 可解释性的五个层次与携证输出

本项目区分五种经常被混用的概念：provenance 说明用了哪些来源；observability 记录实际执行了
哪些动作和工具；plausibility 表示解释在人看来是否合理；faithfulness 要求解释确实反映影响决策
的因素；causal/mechanistic explanation 则声称内部机制或结构因果关系。前两者是工程可审计性，
并不自动证明后两者。大模型生成的流畅 rationale 或 Chain-of-Thought 也不能因为“听起来合理”
就被视为忠实解释。

TrialCompiler 因而输出 Proof-Carrying Clinical Compilation（携证临床文档编译）对象，而不是
只输出修改后的文字。一个携证结果至少包含：可证伪主张、规范事实及版本、引用来源及定位、竞争
假设和剩余不确定性、实际 action/tool trace、依赖影响集、候选原子补丁、补丁前置条件、沙箱复检
结果、负对照结果、人工授权状态和未清偿的专业决策债务。该对象既可以证明“机器结果可进入合格
人员审批”，也可以证明“当前证据不足，系统正确阻断”。

解释面向角色分层呈现。医学审核者需要看到临床语义和风险证据；统计人员需要看到 estimand、
分析集和缺失处理的逻辑；QA 需要看到来源、版本、改动范围和复检结果；系统维护者需要看到工具
轨迹和失败代码。分层只改变呈现，不改变底层证据对象，防止为不同用户生成彼此不一致的理由。

### 6.6.7 用反事实重放检验解释忠实性

对黑盒模型，当前可以验证的是 behavioral counterfactual faithfulness，而不是模型内部神经机制。
若解释声称证据 $e$ 对决定 $d$ 是必要的，则在固定模型、提示、工具和随机配置下移除 $e$ 并重放；
决定应发生预期变化。若把 $e$ 替换为支持竞争假设 $h'$ 的对立证据，系统的 belief、动作、finding
或补丁应向 $h'$ 方向移动。若输出不变，则该证据可能只是被展示出来，却没有真正影响决策。

设 $N$ 个案例均声明某证据必要，necessity flip rate 定义为

$$
NFR=\frac{1}{N}\sum_{i=1}^{N}
\mathbf{1}[d_i^{-e}\neq d_i].
$$

设对立证据干预后的期望方向为 $q_i$，contrastive sensitivity 定义为

$$
CS=\frac{1}{N}\sum_{i=1}^{N}
\mathbf{1}[Direction(d_i^{e\rightarrow e'},d_i)=q_i].
$$

还应报告只保留解释证据时的 sufficiency、移除解释证据时的 comprehensiveness、随机重放下干预
效应的 replay variance，以及解释中无法在实际 trace 或来源找到支撑的 unsupported-rationale
rate。上述指标只能支持行为层的忠实性结论；没有结构因果模型与受控干预设计时，不声称证明了
模型内部因果机制。

### 6.6.8 校准、选择性预测与实验指标

若 $p_i$ 表示目标事件成功的预测概率、$y_i\in\{0,1\}$ 为独立标签，则 Brier score 为

$$
Brier=\frac{1}{N}\sum_{i=1}^{N}(p_i-y_i)^2.
$$

ECE 将样本按置信度分箱并比较每箱平均预测与经验准确率；样本有限或分箱不稳定时，同时报告
binning 方案、样本数和置信区间。若系统只提供风险排序而没有可靠概率，应报告 pairwise rank
accuracy 或 rank-calibration，而不是强行计算“校准概率”。

对拒答或人工升级策略，在阈值 $\tau$ 下定义 coverage 为系统自动提交的样本比例，selective risk
为这些已提交样本中的错误率。risk–coverage 曲线展示自动化覆盖扩大时风险如何变化，AURC 汇总
整条曲线。一个有用的不确定性机制应在降低 coverage 时优先排除高风险样本，而不是随机拒答。
此外还需报告错误自动提交率、defer recall、每个正确闭环的证据成本、单位成本实际熵下降、取证后
成功增益、负对照保护率和 trajectory weakest-link 校准。

实验应比较 deterministic rules only、single-shot LLM、fixed RAG + LLM、fixed RAG + passive
UQ/abstention、uncertainty-guided acquisition，以及 acquisition + counterfactual explanation gate。
只有这种消融才能回答提升来自普通检索、被动拒答、主动取证还是解释门禁，而不能把整个系统的
改进笼统归因于“使用了 AI”。

### 6.6.9 当前实现范围与声明边界

当前仓库已经具备有限假设 belief state、显式 observation-to-posterior 更新、成本感知 EIG 选证、
commit/acquire/defer 决策、受控取证预算、轨迹记录，以及 Brier、ECE、risk–coverage/AURC、排序
准确率、necessity flip rate 和 contrastive sensitivity 的可复现实验底座。未知 observation、
重复访问、越权动作和预算耗尽采用失败关闭；混用 calibration/test split 时禁止产生校准声明。

但当前尚未完成真实多采样模型 UQ、冻结且由专业人员裁决的独立 calibration/test 数据集、learned
trajectory calibrator、来自真实外部工具或专家的 observation model、多轮真实来源主动取证对照
实验，以及具备统计功效的解释忠实性 benchmark。因此现阶段可以声称“已经实现并测试 UQ/XAI
实验与治理机制”，不能声称“真实临床文档上的概率已经校准”或“主动取证策略已经证明优于所有
基线”。这一边界使研究亮点建立在可复现机制和明确待验证假设上，而不是把演示分数包装成临床
有效性结论。

## 6.7 大模型与确定性规则协同

大模型用于自然语言理解、候选事实归纳、章节草拟和复杂语义线索发现；确定性规则用于数值、
时间、标识符、来源、状态和补丁范围等可形式化约束。两者不是相互替代关系：只用规则会遗漏
表达多样的语义问题，只用大模型则难以保证重复运行稳定、边界精确和状态安全。

语义模型发现的内容保持 candidate，除非有独立证据和相应角色裁决。确定性规则的结果可以
直接作为工程 finding，但仍不自动等于医学结论。最终报告同时保存规则类型、触发事实、目标
章节、来源、严重程度和所需角色，使审核人员能够理解系统为什么报警、需要看什么以及哪类
证据可以关闭问题。

## 6.8 人工授权、整改治理与发布边界

第 6.5 节已经给出影响计算、补丁生成和重新编译方法，本节不再重复计算过程，而回答“谁有权
让哪一种状态发生变化”。TrialCompiler 将四种容易混淆的批准明确分开：事实批准确认某个规范
值可进入事实层；finding disposition 决定问题成立、误报、接受风险或需要补证；repair approval
允许指定补丁作用于指定版本和范围；release authorization 才允许修订后的文档进入下一阶段。
前一批准不会自动推出后一批准。

授权采用“对象—动作—角色—范围—期限”五元组。医学事实与风险表述由医学角色负责，分析集、
estimand、缺失数据和多重性由统计角色负责，注册字段和公开披露由注册角色负责，文档质量与版本
完整性由 QA 负责。系统管理员可以维护账号和工作流，但不能借技术权限替代专业签字。每次决定
保存决定人、资格角色、输入证据、可作用范围、时间戳、理由和被授权对象摘要；对象或来源换版后，
旧授权不得静默继承。

当系统无法唯一确定修改内容时，生成 Decision Request，而不是伪造一个自动答案。请求至少包含：
冲突事实、相互支持的来源、受影响单元、可选处置、每个选项的下游影响、当前不确定性以及所需
专业角色。例如 NCT ID 不一致可能来自文件身份错误、相关研究合法引用或来源范围配置错误；即使
人工确认“该问题需要处理”，仍可能返回 `approved_but_manual_edit_required`，因为系统尚不知道
应改编号、改文件分类还是改来源范围。

整改工单依次经历 `open → triaged → decision_pending → approved_for_repair → repaired_in_sandbox
→ revalidated → qa_approved → released`。`rejected`、`accepted_risk`、`needs_evidence`、
`manual_edit_required` 和 `regression_detected` 是明确分支，不能压缩成一个“已处理”状态。
释放门禁同时要求：阻断 finding 为零、所有高风险决定具有合格角色签字、补丁通过第 6.5.7 节
闭环检查、独立 QA 完成审核、目标文档摘要与审核对象一致。由此形成“合格人员决定—事实调和—
关联传播—沙箱修订—确定性复检—独立质量审核—发布”的受控链路。

人工在这里不是模型失败后的兜底，而是系统架构中的授权主体。Agent 的价值在于把分散材料压缩成
可核查的决策包、计算每个选择的传播范围并执行获批后的机械工作；最终专业责任仍由相应角色承担。
这种设计允许提高自动化程度，同时不把模型概率、不确定性评分或语言流畅度冒充医学、统计与注册
授权。

## 6.9 数学与技术正确性说明

设软质量分为 `S(x)`，硬约束为 `g_1(x),…,g_m(x)∈{0,1}`，人工授权为 `A(x)`。候选进入
下一阶段的必要条件定义为 `Release(x)=A(x)∧∧_i g_i(x)`。因此，无论 `S(x)` 多高，只要
任一硬约束失败或授权未完成，`Release(x)=false`。这就是硬门禁对软评分的非补偿性。

对单个事实补丁，若所有编辑操作仅作用于 `I(f)`，且补丁前后非目标章节摘要相同，则满足
局部性；若补丁后规范值存在、旧值不存在、来源引用不变、目标 finding 消失且没有新增高风险
finding，则在已编码谓词范围内满足修订闭环条件。这些命题证明的是状态、来源和修改范围的
工程性质，不证明所有医学或统计判断正确。

系统还用 Wilson 区间表达有限样本的不确定性，避免只报告 100% 点估计。例如 50/50 个
修订成功对应的 Wilson 95% 区间仍为 92.87%—100%，说明样本内没有失败不等于总体零风险。
这一报告方式贯穿角色识别、受控缺陷和修订闭环实验。

### 6.9.1 形式化对象与状态空间

设授权来源集合、事实集合和文档单元集合分别为

$$
\mathcal{S}=\{s_1,\ldots,s_m\},\qquad
\mathcal{F}=\{f_1,\ldots,f_n\},\qquad
\mathcal{U}=\{u_1,\ldots,u_k\}.
$$

一个来源表示为

$$
s_i=(id_i,locator_i,version_i,authority_i,scope_i),
$$

一个事实表示为

$$
f_i=(id_i,key_i,value_i,status_i,source_i,owner_i,version_i,previous_i).
$$

其中事实状态属于有限集合

$$
\Sigma=\{draft,proposed\_change,requires\_human\_review,
approved,rejected,superseded\}.
$$

文档单元可以是 Protocol 章节、SoA 表格、SAP 规则、ICF 表述、CRF 字段或注册记录。沿用
第 6.5 节定义的依赖矩阵 $A$、有效边集合和影响集 $I^*(f)$，完整文档图写为

$$
G=(V,E),\qquad
V=\mathcal{S}\cup\mathcal{F}\cup\mathcal{U}\cup\mathcal{I}\cup\mathcal{R}\cup\mathcal{H},
$$

其中 $\mathcal{I}$、$\mathcal{R}$、$\mathcal{H}$ 分别表示 findings、repair proposals 和人工
决定。本节不再重述图构建与业务分类，而是在该既定计算模型上证明来源、状态、传播和补丁的
工程性质。

### 6.9.2 三个系统不变量

**不变量一：来源可追溯性。** 对任何进入正式审核状态的事实 $f_i$，必须满足

$$
source_i\neq\varnothing,
\qquad
\forall s\in source_i, s\in\mathcal{S}\land locator(s)\neq\varnothing.
$$

因此，只有模型生成文字而没有可定位来源的内容不能成为有效事实。

**不变量二：状态不能被置信度替代。** 设模型对事实正确性的置信度为 $p_i$，事实可驱动
正式修改的必要条件定义为

$$
Effective(f_i)=
\mathbf{1}[status_i=approved]\cdot
\mathbf{1}[source_i\neq\varnothing]\cdot
\mathbf{1}[QualifiedOwner(owner_i)].
$$

即使 $p_i=0.99$，只要相应专业角色尚未批准，$Effective(f_i)=0$。这一区分防止系统把
模型自信误写为组织授权。

**不变量三：硬门禁不能被软评分抵消。** 设语言质量分为 $Q_{LLM}\in[0,100]$，阻断
finding 数为 $B\ge0$，则

$$
Release(X)=\mathbf{1}[B(X)=0]\cdot\mathbf{1}[HumanGate(X)=1].
$$

**命题 1（非补偿性）**：不存在任何有限的语言质量增量 $\Delta Q$，能够使含阻断 finding
的状态自动释放。

**证明**：若 $B(X)>0$，第一个指示函数为 0，所以 $Release(X)=0$。$Q_{LLM}$ 不出现在
释放函数的乘积项中，任意 $\Delta Q$ 均不改变结果。证毕。该命题对应 Metformin-PAD 中
`blocked_machine_gate` 对高语言评分的强制覆盖。

### 6.9.3 变更传播计算的条件完备性与健全性

设事实 $f$ 从旧值 $v_0$ 变为候选值 $v_1$：

$$
\Delta f=(f,v_0\rightarrow v_1),
$$

系统保存 `previous(f)=v_0`、`value(f)=v_1` 和
`status(f)=proposed_change`。沿用第 6.5 节的有效影响集，对 $u\in I^*(f)$ 定义旧值残留谓词

$$
Stale(u,f)=\mathbf{1}[Contains(u,v_0)],
$$

以及 finding 集合

$$
F_{stale}(f)=\{u\in I^*(f):Stale(u,f)=1\}.
$$

**假设 A1**：依赖矩阵记录了所有真实依赖；**假设 A2**：`Contains` 对目标原子旧值无
漏检。在 A1、A2 下，所有依赖 $f$ 且仍包含 $v_0$ 的单元都会进入 $F_{stale}(f)$。
证明很直接：任取真实依赖且包含旧值的 $u_j$，由 A1 得 $A_{fj}=1$，所以
$u_j\in I^*(f)$；由 A2 得 $Stale(u_j,f)=1$，故 $u_j\in F_{stale}(f)$。

反过来，若增加 **A3**：`Contains(u,v_0)=1` 仅当 $v_0$ 在 $u$ 中代表同一事实，而不是
无关同值，则 $F_{stale}(f)$ 中不存在与 $f$ 无关的单元。实际自由文本不能保证 A1、A3
恒成立，因此系统同时使用合法差异负对照、scope 判断和人工复核，而不宣称绝对完备。

### 6.9.4 最小修订与原子补丁

设原文档为 $Y$，目标 findings 为 $I$，候选文档为 $Y'$。理想修订是约束优化：

$$
\min_{Y'}D_{edit}(Y,Y')
$$

满足

$$
Unresolved(Y',I)=0,\qquad
NewRegression(Y',Y)=0,\qquad
Traceable(Y')=1.
$$

每个编辑操作表示为

$$
o=(section,start,end,before,after,fact\_ids,source\_ids).
$$

**命题 2（非重叠补丁可合并）**：若所有操作区间两两不重叠，且 `before` 与原文一致，
则按位置从高偏移到低偏移应用不会互相覆盖。

**证明**：不相交区间修改不同字符集合；从高偏移开始可避免前一替换改变后一替换的坐标。
因此每个操作的目标字符保持可定位。若区间重叠，交换律不再成立，系统不自动选择，而生成
Decision Request。这个命题解释了为何系统采用局部红线而不是让多个 Agent 顺序重写整章。

### 6.9.5 数值边界与合法双时间轴

NCT04683926 一处文本声明给药间隔 $\Delta d>3$ days，而给药日为

$$
D=(1,4,7,10),\qquad
\nabla D=(4-1,7-4,10-7)=(3,3,3).
$$

因此

$$
\min(\nabla D)=3\not>3.
$$

“第 1、4、7、10 天给药”与“间隔大于 3 天”不能同时为真。该问题无需大模型，可由规则
稳定复现；同时必须区分 $>3$ 与 $\ge3$，否则会发生边界漏检。

但 Protocol 连续时间轴与 CRF/SAP 周期内 Day 1 可能只是不同坐标。设

$$
\phi(period_i,day\ 1)=D_i,
\qquad D_i\in\{1,4,7,10\}.
$$

只要映射 $\phi$ 被明确记录且两种时间轴均获批准，它们是坐标变换而不是矛盾。因此系统设置

$$
Approved(F022)\land Approved(F023)\land MappingExists
\Longrightarrow Conflict=0.
$$

该负对照使系统既能检出严格边界错误，又不会把合法双时间轴一概报错。

### 6.9.6 样本量可复现性

对等比例双臂连续型结局，设均值差为 $\delta$、共同标准差为 $\sigma$、双侧显著性水平为
$\alpha$、目标 power 为 $1-\beta$，每组可评价样本量近似为

$$
n_{arm}=2\left(\frac{(z_{1-\alpha/2}+z_{1-\beta})\sigma}{\delta}\right)^2.
$$

若脱落率为 $r$，总随机样本量为

$$
n_{total}=\left\lceil\frac{2n_{arm}}{1-r}\right\rceil.
$$

Metformin-PAD 冻结产物给出 $\delta=20$ m、$\sigma=65$ m、$\alpha=0.05$、power=0.85、
$r=0.15$。使用 $z_{0.975}\approx1.960$、$z_{0.85}\approx1.036$，得到

$$
n_{arm}\approx189.6,
\qquad
n_{total}\approx\left\lceil\frac{2\times189.6}{0.85}\right\rceil\approx447.
$$

所以生成文本中的 $n=220$ 无法由其自身参数复现；在同一简化近似下，$n=220$ 的 achieved
power 约为 0.557，而不是 0.85。该计算只检查文本内部可复现性；若正式方案使用 MMRM、
协变量调整或模拟法得到其他样本量，必须提供相应假设和输出，系统不能用近似公式替代统计师。

### 6.9.7 Estimand、分析集与终末事件

设 estimand 目标人群为全部随机化受试者

$$
P_E=\{x:x\text{ 被随机化}\},
$$

而 FAS 要求至少用药一次且至少有一次基线后测量：

$$
P_{FAS}=\{x:x\text{ 随机化}\land dose(x)\ge1\land postbaseline(x)\ge1\}.
$$

显然 $P_{FAS}\subseteq P_E$；只要存在随机后未用药或无基线后测量者，就有严格包含
$P_{FAS}\subsetneq P_E$。因此，若文本声称 estimand 面向所有随机化受试者，却用排除部分
随机化受试者的 FAS 实现，二者不能被视为自动一致。系统触发 `DET-STAT-002`，但不自行
决定修改 estimand 还是 FAS，这属于统计决策。

对终末事件，令 $Y_t$ 为功能结局，$D_t=1$ 表示受试者在 $t$ 前死亡。死亡后的 $Y_t$ 可能
在临床定义上不存在，而不只是“存在但未观测”。直接设

$$
Y_t\mid D_t=1\sim MAR
$$

并进行普通多重插补，隐含死亡后结局可从存活者分布合理生成的假设。系统因此在 death、MAR
和 multiple imputation 同时出现时触发 `DET-STAT-004`，要求说明 composite、
while-on-treatment、hypothetical 或其他 intercurrent-event 策略及敏感性分析。这是保守
风险规则，不是对所有 estimand 策略的自动裁决。

### 6.9.8 Benchmark 指标推导

定义

$$
Precision=\frac{TP}{TP+FP},\qquad
Recall=\frac{TP}{TP+FN},\qquad
F1=\frac{2PR}{P+R}.
$$

NCT04683926 初始结果为 $TP=4,FP=2,FN=4,TN=1$，所以 Precision=0.6667、Recall=0.5000、
F1=0.5714。加入数值边界、跨来源时长/筛选窗口、来源链完整性和合法时间轴负控后，结果为
$TP=8,FP=1,FN=0,TN=2$，所以 Precision=0.8889、Recall=1.0000、F1=0.9412。

绝对 F1 增量为

$$
\Delta F1=0.9412-0.5714=0.3698,
$$

相对增幅约 $0.3698/0.5714\approx64.7\%$。同时负对照准确率由 0.5 提升到 1.0，说明
提升不是单纯通过“多报问题”换取 Recall。不过该 Gold 规模很小，正确结论只是在同一冻结
案例和 scorer 下修复了已知 FN 与负控误报，不能外推总体性能。

### 6.9.9 整改闭环的终止条件

设第 $t$ 轮阻断 finding 集为 $B_t$，每条 finding 对应工单 $w_i$，其退出条件为
$Exit(w_i)$。进入独立质量审核的条件是

$$
Ready_t=\mathbf{1}[B_t=\varnothing]\prod_i\mathbf{1}[Exit(w_i)=1].
$$

高风险工单设置 $AutomaticApply(w_i)=0$，所以系统可自动生成整改要求与重检结果，但不能
自动批准样本量、estimand 或监管路径。若每轮至少关闭一个 finding 且不产生新 finding，
$|B_t|$ 严格递减，最多经过 $|B_0|$ 轮达到空集；若产生新问题、证据不足或专业意见冲突，
流程转入人工决策，而不是无限自循环。

### 6.9.10 证明覆盖与未证明事项

当前代码和测试支持：类型化来源、事实、章节、finding、repair 与 decision request；显式
影响集合；NCT 标识符、Week、严格边界和旧值残留检查；原子补丁与重叠冲突升级；分阶段
可见性隔离；样本量、合成监管权威化、estimand/FAS、多重性和终末事件门；独立 scorer、
负对照和 50 例修订闭环。

当前不能证明：任意临床文档的事实抽取完备；任意语义冲突的高 Precision/Recall；规则覆盖
全部统计或监管风险；模型具有稳定跨领域校准；系统能够替代专业人员；公开或合成 benchmark
可以代表生产性能。准确表述是：TrialCompiler 是状态转换系统

$$
X_{t+1}=T(X_t,a_t;G,S,H),
$$

其中 $G$ 为依赖图，$S$ 为带来源和状态的事实，$H$ 为人工治理状态。它优先保证来源可追溯、
授权状态不可被置信度替代、硬门禁不可被软评分抵消，并在这些明确边界内提供可验证控制。

## 7.2 原型验证总体设计

验证遵循“来源先冻结、预测先于 Gold、案例级隔离、正负对照并存、生成与评分分离”的原则。
完整性测试回答资料是否真实、可读和可复现；角色 Gold 回答文档角色识别；受控缺陷 Gold

人工审批实验回答批准后的补丁是否真正正确。不同证据层不能相互冒充。

| 证据层 | 数据来源 | 回答的问题 | 可以报告 | 不能报告 |
|---|---|---|---|---|
| 完整性层 | 50 例注册快照与 65 份 PDF | 资料是否真实、可读、可冻结 | 数量、页数、摘要、覆盖 | 模型准确率 |
| 角色层 | 官方 hasProtocol/hasSap | 文档角色是否识别正确 | 角色分类指标 | 临床缺陷能力 |
| 受控缺陷层 | 两类注入缺陷与负对照 | 确定性检查是否稳定 | 混淆矩阵、Wilson 区间 | 自然语义总体性能 |

| 修订闭环层 | 模拟批准与沙箱补丁 | 批准后是否改对且不越界 | 正确性、局部性、回归 | 真实专业授权有效性 |

## 7.3 NCT04683926 一致性审核案例

公开案例 NCT04683926 / OMNI-PAIN-103 包含 Protocol、SAP、ICF 和注册资料。项目建立 27 条
人工事实基线、正例、合法差异负对照和独立 Gold Scorer。初始规则在严格数值边界、跨来源
时长和合法双时间轴方面存在漏报与误报；增加相应规则后，冻结结果为 TP=8、FP=1、FN=0、
TN=2，Precision=0.8889、Recall=1.0000、F1=0.9412、负对照准确率=1.0000。

额外 FP 来自一条未匹配 Gold 的可报告语义提醒，而不是把两个负对照判错。该结果证明当前
案例上的工程改进，不代表跨疾病、跨文档类型的临床总体准确率。

独立 scorer 的计分单位是预先冻结的 Gold test，而不是模型自行生成的 finding 名称。一个
阳性 Gold 只有在事实、章节、语义谓词和必要证据都匹配时才计为 TP；负对照若被错误报告则
计为 FP；未匹配任何正例的额外可报告 finding 也计为 FP。这样可以防止系统通过改写 finding
名称或输出大量模糊提醒获得虚高 Recall。初始版与整改版对比如下：

| 指标 | 初始版 | 整改版 |
|---|---:|---:|
| TP / FP / FN / TN | 4 / 2 / 4 / 1 | 8 / 1 / 0 / 2 |
| Precision | 0.6667 | 0.8889 |
| Recall | 0.5000 | 1.0000 |
| F1 | 0.5714 | 0.9412 |
| 负对照准确率 | 0.5000 | 1.0000 |

整改新增严格给药间隔、跨来源住院时长、筛选窗口包含边界和来源链完整性规则，并为合法
Protocol/CRF 双时间轴增加 scope-aware 负对照。由此，已知漏报被补齐，同时已知误报被
消除；这比单纯扩大报警数量更能说明规则改进的有效性。

## 7.4 Metformin-PAD 受控生成案例

该工作流以完全合成或历史重建测试包验证分阶段可见性。Phase 1 仅读取初始资料并生成候选
事实、Synopsis 和章节；Phase 2 在冻结前期结果后加入新证据并增量修订；隐藏参考资料只在
生成结束后交给独立评估器。每次运行保存输入与提示摘要、模型、来源边界和 checkpoint。

首次软评分表面较高，但模拟医学、统计和注册审核发现样本量无法复现、estimand 与分析集
不一致、终末事件被不当纳入 MAR 插补、模拟监管材料被权威化等问题。新增机器门后，运行被
标记为 `blocked_machine_gate` 和 `blocked_missing_evidence`。这一失败整改说明系统亮点不在
“生成得像”，而在发现高分不可信后能够阻断并形成角色化整改工单。

机器门将问题编译为结构化整改工单，而不是只输出一段批评。每条工单记录责任角色、受影响
事实和产物、必须执行的修订动作、所需证据及可机器检查的退出条件。例如样本量工单要求统计
人员提供可复现公式或模拟输出，并同步 Synopsis、统计章节和相关登记字段；estimand 工单
要求明确人群、变量、治疗条件、伴发事件策略和汇总量；模拟监管材料工单要求删除“监管已
认可”类无依据表达并恢复资料的合成属性。只有退出条件全部为真，工单才可关闭。

这一案例也验证了分阶段资料隔离：Phase 1 运行不能看到 Phase 2 新证据，Phase 2 只能在
前期摘要冻结后增量处理，Evaluator-only 资料不能回流生成上下文。每次运行保存 checkpoint
digest，因此可以判断后续结果是合法增量还是历史输入被静默改变。

## 7.5 50 例真实公开语料与文档角色 Gold

项目通过 ClinicalTrials.gov API v2 冻结 50 个真实研究、50 份注册快照和 65 份官方 PDF，
共 3,025 页。48 个为干预性研究、2 个为观察性研究；35 个案例使用合并 Protocol/SAP，
15 个使用分立文件。入选文件均通过 PDF 头、非空页数和正文可提取检查，URL、页数、字节数
和 SHA-256 均可复核。

官方 `hasProtocol/hasSap` 元数据形成 130 个二元角色标签，50 个案例按 30/10/10 分为
train、calibration 和 held-out test。测试集规范文件名基线在当前 28 个标签上得到 100%
点估计，但 Wilson 95% 区间为 87.94%—100%。该指标仅说明官方文档角色，不是缺陷检测性能。

语料构建不是下载前 50 个搜索结果。构建器逐项核对官方文档元数据、文件头、页数、正文

## 7.6 200 个受控缺陷 Gold 与检测结果

在 50 个真实案例上派生两类可确定缺陷：canonical NCT identifier 冲突和 enrollment 变更
未传播。每例每类各有一个正例和匹配负对照，共 200 个标签，正负各 100。标识符负对照包含
正确 NCT 和作为相关研究引用的另一 NCT；enrollment 负对照包含已经正确传播的新值，因此
不是容易通过的空文本。

Held-out test 的 10 个案例、40 个标签上得到 TP=20、FP=0、FN=0、TN=20，Precision、
Recall、F1 和 Accuracy 点估计均为 100%，Accuracy Wilson 95% 区间为 91.24%—100%。全量
200 标签判断正确，区间为 98.12%—100%。结论严格限定为两类受控确定性任务。

Wilson 区间用于避免在分母较小时把 100% 点估计解释成确定规律。对成功数 $k$、样本数 $n$
和正态分位数 $z$，区间中心与半宽为

$$
center=\frac{\hat p+z^2/(2n)}{1+z^2/n},
$$

$$
half=\frac{z}{1+z^2/n}
\sqrt{\frac{\hat p(1-\hat p)}{n}+\frac{z^2}{4n^2}},
\qquad \hat p=k/n.
$$

因此 40/40 的下界仍只有约 91.24%，200/200 的下界约 98.12%。报告区间的目的不是削弱
结果，而是明确现有样本对总体结论提供了多强的证据。

## 7.7 真实自然候选与不确定性处理

系统从 65 份真实 PDF 中高召回挖掘 1,393 条自然差异候选，经案例和类别平衡后形成 197 条、
48 个案例、8 个类别的队列，覆盖分析人群、缺失数据、多重性、主要终点时间、终末事件、
试验标识符和入组范围等。每条具有文件名、页码、短原文和所需角色。

数字差异可能表示计划/实际、筛选/随机化、总样本/亚组；另一 NCT 可能只是相关研究引用。
因此系统没有自行制造 Gold：197 条均保持 `candidate_not_gold`，0 条自动确认，197 条全部
路由专业复核。裁决前预测摘要已冻结。由于当前没有时间进行高成本外部裁决，本次不计算这
部分的 Precision、Recall、F1 或 Accuracy，仅把它作为后续 POC 资产。

候选分布为：analysis population 30 条、enrollment scope/state 43 条、estimand/intercurrent
event 2 条、missing data 27 条、multiplicity 10 条、primary endpoint/time 39 条、terminal
event 41 条、trial identifier scope 5 条。系统冻结 disposition 中，43 条属于入组范围或
状态不明确，69 条需要统计语义裁决，80 条需要医学与统计上下文，5 条需要 NCT 范围复核。
所有 197 条都有文件名和页码，来源定位完整率为 100%。

这组数据的价值不是提供一个新高分，而是暴露真实文本中的困难分布：同一数字可能代表不同
人群和时间状态，“death”可能是 SAE 定义而非 estimand 缺陷，另一个 NCT 可能是相关研究，
“intention-to-treat”出现也不自动证明分析集矛盾。系统选择 defer 而不是 auto-confirm，体现
了不确定性治理在自然数据上的实际作用。

## 7.8 模拟人工批准后的修订闭环

项目在 50 例上构造 200 个“发现—决定—修订—复检”场景，模拟统计或数据治理角色批准
100 个缺陷、驳回 100 个负对照。50 个 enrollment 问题具备安全确定性补丁：50/50 新值
写入、旧值移除、仅改变目标章节、来源链保持、finding 关闭，新增 finding 为 0；100 个负
对照均未变化。成功率点估计为 100%，Wilson 95% 区间为 92.87%—100%。

另外 50 个 NCT ID 冲突虽获“需要处理”的模拟批准，系统仍拒绝未经证明的字符串替换并转
人工编辑。这验证了授权与执行能力是两个不同门槛。模拟角色只验证流程机制，不构成真实
医学、统计或注册批准。

每条闭环记录同时保存 before/after digest、finding ID、提案数、改变章节、规范值是否出现、
旧值是否消失、来源链是否保持、复检后剩余 finding 和新增 finding。验收不是只比较最终
字符串，也不是只检查原报警消失。50 个可修订场景的 `semantic_patch_correct`、
`minimal_scope_patch`、`source_trace_preserved` 和 `finding_closed_after_patch` 均为真；
`new_finding_after_patch_count`、`negative_control_changed_count` 和 `unexpected_change_count`
均为 0。held-out test 的 10 个可修订问题同样全部闭合。

## 7.9 失败案例与机器门禁整改

验证过程中修复了权威状态与摘要漂移、0 finding 的 vacuous pass、软评分掩盖硬失败、固定
关键词误作角色真值、过度并发取证和只检查报警消失而不检查补丁语义等问题。当前摘要由权威
状态重新物化；未闭合 finding 进入决策队列；角色 Gold 使用官方元数据；补丁复检同时检查
新旧值、范围、来源链和新增问题。报告保留这些失败，是为了证明系统能够从暴露的问题形成
可执行整改，而非只展示成功案例。

| 暴露问题 | 风险 | 整改 | 回归证据 |
|---|---|---|---|
| state 与 summary 漂移 | 旧摘要误导为可发布 | 摘要由权威状态重新物化 | assurance 一致性测试 |
| 0 finding 得到满分 | 空结果成为 vacuous pass | 无有效覆盖时禁止通过 | 质量门负例测试 |
| 软评分掩盖严重矛盾 | 流畅文本获得虚高评价 | 硬门对软分非补偿 | Metformin 机器阻断 |
| 关键词充当角色真值 | 合并文档产生假阴性 | 官方元数据定角色 | 130 个角色标签 |
| 到达目标前提交全部下载 | 成本与等待失控 | 分批取证、达标即停止 | 50 例构建器测试 |
| 只检查 finding 消失 | 错误改写也可能消警 | 检查值、范围、来源与新问题 | 50 个修订闭环 |

## 7.10 GitHub、Demo 与复现方式

仓库地址为 `https://github.com/ZiliShao222/TrialCompiler`。确定性复现可执行：

```powershell
python -m ruff check src tests
python -m pytest -q
python scripts/build_public_protocol_sap_corpus.py --limit 50 --workers 8
python scripts/profile_public_protocol_sap_corpus.py
python scripts/build_and_score_public_role_gold.py
python -m scripts.build_and_score_seeded_defect_gold

python scripts/build_natural_defect_adjudication_set.py
python scripts/evaluate_natural_defect_candidates.py
python -m scripts.evaluate_simulated_human_repair_loop
```

仓库保存公开语料清单、注册快照、case contract、Gold labels、逐项预测、逐项修订记录、汇总
JSON 和 GitHub Actions。API 密钥不进入仓库；公开 benchmark 和确定性测试无需模型密钥。

关键证据位置包括：`benchmarks/trialdocbench/public_corpus_050/` 保存 50 例语料、Gold、自然
候选和修订结果；`docs/final_submission/engineering/` 保存技术证明、验证报告与运行说明；
`tests/` 保存自动化检查；`.github/workflows/` 保存 CI 配置。独立 scorer、受控缺陷构建器
和模拟人工闭环脚本均可单独运行，使汇总数字能够下钻到逐条记录。

复现时应同时记录 Git commit、Python 与依赖版本、模型名称、Prompt 摘要、输入文件摘要、
运行时间和输出目录。ClinicalTrials.gov 是持续更新的注册系统，重建语料可能得到更新文件；
因此完成标签后不得静默覆盖冻结 corpus，而应创建新版本并保留两个摘要。涉及外部模型的
运行还需固定模型和提示版本；确定性 benchmark 则可在无 API key 环境中重复执行。

## 7.11 当前局限、下一阶段计划与验收条件

当前仍缺少自然候选的外部专家 Gold、跨治疗领域和跨语言验证、真实 Agent 轨迹校准、企业
模板与 SOP 适配、生产安全验证和真实业务 ROI。下一阶段应冻结版本后，由医学、统计和注册
人员独立裁决自然候选，再对已冻结预测评分；随后开展人工流程与辅助流程的配对 POC，记录
发现率、错误修订率、影响范围召回率、审核时间和返工。

扩围条件包括阻断性问题零漏放行、正式文件零静默覆盖、未批准事实不进入正式流程、来源和
审计记录完整、回滚演练通过，以及错误修订率和人工审核时间达到预注册阈值。任何硬条件未
满足，系统保持研究或受控试点状态。基于现有证据，可以将 TrialCompiler 表述为具有真实
公开数据底座、可运行代码、数学控制、不确定性治理、独立评测和安全责任边界的临床文档 AI
研究原型；不能表述为已经替代专业人员、获得临床生产批准或证明总体临床准确率的系统。
