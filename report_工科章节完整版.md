# 第六部分 技术方案、系统架构与正确性机制

临床试验方案及其关联文件具有多来源、多版本、跨专业和强约束特征。同一项研究设计事实可能同时
出现在 Protocol、SAP、ICF、CRF、注册记录及内部决策材料中；不同文件又可能因人群、时间状态、
文档职责或监管范围不同而保留合法差异。人工逐份核对容易遗漏变更影响，直接使用大模型改写又
难以稳定保证来源、版本、数值边界和专业授权。

TrialCompiler 采用“概率型 AI 推理 + 确定性临床编译 + 人工治理”的技术路线：以版本化事实和
文档依赖图统一跨文件信息，由多智能体发现语义问题并根据不确定性选择取证或转交，再通过规则、
原子补丁和沙箱复检把候选结果编译为可追溯、可验证、可阻断的审核材料。系统辅助专业人员形成和
执行决策，但不替代医学、统计、注册及质量角色的正式授权。

本部分依次说明总体架构、事实与证据底座、多智能体协同、不确定性驱动的自然语言事实计算、
携证临床编译、人工发布治理及私有化部署路径。

## 6.1 设计目标与总体分层架构

### 6.1.1 架构目标与原则

TrialCompiler 面向同一临床设计事实跨文档重复、变更传播容易遗漏以及语义判断难以形式化的问题，
需要同时回答五个问题：依据来自哪里、当前事实是什么、哪些内容受其影响、系统对判断有多大不确定
性、谁有权使结果生效。架构据此遵循四项原则：关键结论必须绑定来源和版本；事实先于章节并通过
依赖关系传播；模型生成、机器验证和人工授权相互分离；任何状态变化均可复检、阻断和回滚。

### 6.1.2 总体架构

TrialCompiler 采用“五层主链 + 独立评测面”的总体架构。五层主链依次为协同接入层、证据与事实
层、智能体（Agent）推理层、确定性编译与质量层、人工治理与正式系统层；独立评测面从冻结运行中
读取产物，但不向生成与审核过程回传预先冻结的金标准标签（Gold Standard，以下简称“金标准”）。
各层并非简单的软件组件堆叠，而是对应临床文档从“资料”转变为
“候选结论”、再转变为“可审批修订”的权威升级过程。

```text
医学/统计/注册/运营/质量人员
                │
                ▼
① 协同接入层：任务契约、资料提交、范围确认、人工补问
                │
                ▼
② 证据与事实层：来源冻结 → 文档单元化 → 候选事实 → 版本化事实基线
                │
                ▼
③ 智能体推理层：角色化审核 → 语义候选 → 竞争假设 → 不确定性决策
                │                         │
                │              证据不足时触发受控取证
                │                         └──────→ 返回证据与事实层
                ▼
④ 确定性编译与质量层：规则检查 → 依赖传播 → 原子补丁 → 沙箱复检 → 机器门禁
                │
                ▼
⑤ 人工治理与正式系统层：事实批准 → 问题处置 → 修订批准 → 质量保证复核 → 文档管理系统发布

独立评测面：冻结金标准 / 负对照 / 独立评分程序 / 校准集—测试集划分 / 回归测试
              └──只评价冻结产物，不向①—⑤提供答案
```

第一层是协同接入层。医学、统计、注册、临床运营和质量人员通过飞书 Aily、命令行界面
（Command-Line Interface，CLI）或应用程序接口（Application Programming Interface，API）发起任务，
提交公开或经授权的文件，并说明项目、任务类型、文档范围、研究阶段、允许来源和期望产物。系统先
将自然语言请求转换为结构化任务契约；范围、责任角色或关键资料缺失时先补问，不让模型自行猜测。
该层负责协作和留痕，不承担正式电子签名或文件发布。

第二层是证据与事实层。系统为每份文件和外部记录登记稳定来源标识、版本、适用范围、定位信息和
内容摘要，再将文档拆分为可定位的章节、表格和字段。模型与规则可以从中提出候选事实，但候选须
经过来源核对和专业确认后才能进入版本化临床试验事实表（Trial Fact Sheet）。事实层将终点、人群、
样本量、给药、访视和统计原则从重复文本中提升为共享对象；临床文档依赖图（Clinical Document
Graph）则记录各文档单元对事实的依赖。后续智能体读取的是带来源和状态的事实，而不是每次重新从
全部原文猜测当前设计。

第三层是智能体推理层。不同角色智能体在共享状态上执行证据审查、语义差异发现、候选修订和独立
复核，而不是通过自由聊天形成不可追踪共识。大模型重点处理跨文档语义等价、隐含冲突、适用范围判断、
竞争解释和候选表达。遇到证据不足时，不确定性治理模块比较继续读取资料、调用工具、请求专业确认、
暂缓或拒答的收益与成本；若选择取证，新观测证据（observation）必须重新登记并返回第二层，随后
重新运行相关
审核节点，不能在模型内部直接把新信息写成批准事实。

智能体审核采用 A–H 受控状态机，而不是让多个模型自由讨论后给出一份综合答案。一次既有文档审核
按下列顺序运行：

```text
A Context Lock（锁定上下文）
→ B Evidence Review（证据审查与问题识别）
→ C Repair Proposal（修订提案）
→ D Independent Quality Gate（独立沙箱复检）
→ G Uncertainty Governance（决定提交、取证、讨论或转交）
      ├─ 需要新证据：H Governed Evidence Acquisition
      │                → 新证据登记 → 返回 B/C/D/G
      └─ 证据充分或需要人工：E Review Packet（审核包）
                              → F Experience Candidate（经验候选）
                              → 合格人员决定
```

其中，A 节点把项目、文档类型、允许来源、当前批准事实和任务范围写入上下文锁，防止后续智能体
跨项目或跨版本取证。B 节点同时运行确定性预检和语义审查，输出带来源定位的问题记录（finding），
而不是直接改文档。C 节点只对证据充分且事实状态允许的问题记录生成修订提案（Repair Proposal）；
存在多种合理解释时改为专业决策请求（Decision Request）。D 节点在独立沙箱副本中应用候选补丁，
检查目标问题是否关闭、是否改变了非目标范围、来源链是否保留以及是否产生新问题，避免生成智能体
自己评价自己。

G 节点汇总 B 的证据、C 的候选和 D 的复检结果，维护竞争假设及剩余不确定性。它可以选择提交
候选（`commit`）、获取新证据（`acquire`）、增加受控讨论（`deliberate`）或转交人工（`defer`）。
选择获取新证据时，H 节点只能获取允许列表中、范围匹配且摘要可验证的证据，并把观测证据写回共享
状态；随后必须重新经过 B、C、D 和 G，不能因取证后置信度升高而直接提交。选择转交人工时，系统
生成面向指定医学、统计或注册角色的专业决策请求，并暂停相关状态迁移。

机器阶段结束后，E 节点将事实、问题记录、影响范围、候选补丁、复检结果、剩余不确定性和所需决定
汇总为审核包（Review Packet）。F 节点只提取可能复用的经验候选（Experience Candidate），不自动
写入组织记忆；经验必须经过专业
人员批准、限定适用范围并去除项目敏感信息后才可在后续任务中检索。整个流程共享类型化状态和审计
事件，每个智能体只能读写其职责内的字段，因此“多智能体”表示职责分离和受控交接，而不是多个
聊天机器人轮流生成文本。

对于全新方案生成，系统不复用“先发现缺陷再补丁”的审核链，而采用分阶段生成链。第一阶段中，
`G1 Evidence Planner`（证据规划智能体）整理证据、关键问题和候选事实表，`G2 Section Writer`
（章节撰写智能体）在同一事实约束下生成各章节，`G3 Quality Judge`（质量评审智能体）检查来源边界、
未支持声明和跨章节一致性；第一阶段冻结后，第二阶段才允许接收新增申办方、监管或中心反馈，由
`G4 Fact Reconciler`（事实调和智能体）调和当前有效事实，`G5 Incremental Reviser`（增量修订智能体）
修订受影响章节，`G6 Artifact Compiler`（关联产物编译智能体）同步访视活动计划表（Schedule of
Activities，SoA）、估计目标（estimand）、统计和实施类关联产物，`G7 Quality Judge`（第二阶段质量
评审智能体）执行复核。隐藏参考资料仅供最终独立评估器使用，不得回流 G1—G7。因此，既有文档审核
与全新方案生成共享来源、事实、门禁和审计底座，但使用不同的智能体状态机，
避免把生成任务和整改任务混成同一种流程。

第四层是确定性编译与质量层。智能体产生的问题记录和修订提案进入类型、来源、状态、
标识符、数字、日期、严格边界和旧值残留检查。事实发生变化时，系统沿依赖图计算受影响章节，只对
事实明确、范围唯一的位置生成原子补丁；补丁先应用于沙箱副本，再重新执行事实抽取、规则检查、
负对照保护和新增问题扫描。该层输出通过或阻断的机器结论以及可复核证据，不因大模型语言评分较高
而放宽硬约束。

第五层是人工治理与正式系统层。机器通过只表示候选结果具备进入专业审批的条件，不表示内容已经
获得临床或组织授权。系统分别收集事实批准、问题处置、修订批准和发布授权；需要医学、统计、注册
或质量保证（Quality Assurance，QA）判断的事项以专业决策请求进入相应角色。只有阻断问题关闭、
补丁复检通过、签字对象与当前版本摘要一致并完成独立质量保证复核后，候选文件才进入企业文档管理
系统（Document Management System，DMS）的受控签核、发布和归档流程。

独立评测面与上述业务主链隔离。它保存冻结金标准、负对照、实验划分和独立评分程序，用于评价文档
角色识别、受控缺陷检测、自然候选路由、修订局部性和不确定性策略。评测面可以读取冻结输入、预测、
补丁和审计记录，但金标准与隐藏参考不得进入智能体上下文。由此，业务流程回答“结果能否进入审批”，
评测流程回答“在预先定义的任务上表现如何”，二者不能相互代替。

### 6.1.3 关键控制关系

系统将数据、控制和评测三类状态分离。来源、事实、文档单元和补丁属于数据状态；角色权限、人工
决定和发布门禁属于控制状态；金标准、隐藏参考和实验划分属于评测状态。模型输出不能改变批准状态，
无范围的人工“同意”不能替代对象级授权，评测答案也不能进入生成上下文。

各部分通过包含项目、版本、对象标识、来源摘要和状态的结构化接口连接。依赖缺失、摘要变化、权限
不足、输出结构无效或机器门失败时，系统保留既有权威状态并失败关闭；可恢复技术错误允许有限重试，
专业证据或授权缺失则转为补问、专业决策请求或人工处理。后续各节分别展开事实、推理、传播、
治理和正确性控制的具体实现。

## 6.2 可信计算底座：来源、知识库与版本化事实

### 6.2.1 证据来源与知识库构成

TrialCompiler 将“项目事实依据”和“通用参考知识”分开管理。项目事实依据用于判断当前研究中
什么内容有效，包括经授权的方案、统计分析计划、知情同意书、病例报告表、注册记录、申办方决定
和专业人员确认；通用参考知识用于提供法规原则、文档结构、方法和风险线索，不能直接替代项目
决定。通用文献和模板的检索结果仅用于形成候选问题，不直接改变当前研究事实。

当前知识库按用途分为五类：

| 来源类别 | 主要内容 | 在系统中的作用 | 不得越过的边界 |
|---|---|---|---|
| 项目授权材料 | 当前项目文件、正式决定、批准版本和专业回复 | 建立和更新项目事实 | 未批准草稿不得冒充当前事实 |
| 公共注册与监管来源 | 美国 ClinicalTrials.gov 临床试验注册平台快照，国际人用药品注册技术协调会、美国食品药品监督管理局、欧洲药品管理局等公开资料 | 核对公开字段、法规原则和来源范围 | 通用指南不能替代项目医学或统计决定 |
| 标准、模板与关联文档 | 干预性试验方案标准条目（SPIRIT）、TransCelerate 及机构公开的方案、统计分析计划和知情同意模板 | 提供章节结构、字段覆盖和检查线索 | 模板示例值不能写入项目事实 |
| 科学方法与技术文献 | 多智能体、不确定性量化、可解释性、临床文档自动化等论文 | 支持算法设计、指标选择和研究假设 | 论文结论不等于本系统已经验证有效 |
| 经批准的组织经验 | 已确认的合法差异、规则适用条件、误报模式和整改经验 | 为后续任务提供受范围约束的经验检索 | 单次运行经验未经审核不得跨项目复用 |

知识库通过统一来源目录管理资料身份、版本、原始链接、内容摘要、使用目的和适用边界，并按监管
伦理、标准模板、数据规范、人工智能方法及项目材料等主题组织。进入验证流程的公开研究资料另行
冻结注册快照、原始文件和案例范围。检索结果必须能够返回来源登记信息和原文位置；主题分类、文本
相似度或模型摘要本身不构成事实依据。

### 6.2.2 来源登记、权威与适用范围

来源登记用于建立资料身份、原文定位、版本状态、适用范围和访问权限之间的稳定关联。所有进入事实
提取、语义审核或评测流程的资料均须先形成来源记录；仅有文件名、文本片段或模型摘要而缺少来源
定位的内容，不得用于支撑正式事实。

来源记录的主要登记要素如下：

| 登记维度 | 主要内容 | 控制目的 |
|---|---|---|
| 资料身份 | 来源编号、标题、文件类型、提供方 | 区分同名文件及不同来源 |
| 原文定位 | 网页地址、本地文件、页码、章节、表格或单元格 | 支持原文核查和证据追溯 |
| 版本信息 | 版本号、生效时间、获取时间、内容摘要 | 识别历史版本和内容变化 |
| 适用范围 | 研究、地区、人群、阶段、时间状态和文档用途 | 限定来源能够支持的结论 |
| 访问范围 | 项目权限、跨项目复用范围、评测专用标记 | 防止越权检索和评测信息泄漏 |

系统使用 256 位安全散列算法（SHA-256）生成文件内容摘要，并将摘要与来源编号和版本绑定。相同
下载地址或文件名对应的内容发生变化时，系统建立新版本，不覆盖已经参与事实批准、人工裁决或金
标准评价的历史文件。内容摘要用于验证评审对象是否发生变化，不用于判断文件内容是否正确。

来源权威采用“来源性质与适用范围联合判定”，不设置脱离用途的全局优先级。监管指南用于约束通用
原则，企业模板用于约束文档结构，项目批准文件用于确定当前设计，公开注册记录用于核对公开字段和
实际状态，科学文献用于支持方法选择和风险识别。通用指南、模板示例和技术论文均不能直接替代项目
医学、统计或注册决定。

来源差异须在研究、版本、人群、时间状态和文档用途一致的条件下进行比较。例如，批准方案中的计划
入组人数 220 与公开注册结果中的实际入组人数 198 分别描述计划状态和实际状态，不构成需要自动
改写的事实冲突；统计模板中的示例样本量 200 仅提供格式参考，不具备项目事实效力。只有适用范围
相同且规范化内容不兼容时，系统才生成冲突候选并进入专业调和。

知识库检索沿用上述来源控制。智能体只能访问任务契约允许的来源集合，检索结果必须返回来源编号、
版本及原文位置。仅有摘要、原文不可定位或版本无法确认的内容，只能作为补充取证线索。独立评测
使用的金标准和隐藏参考与业务知识库隔离，不向生成或审核智能体开放。

### 6.2.3 版本化临床事实模型

系统将跨文档重复出现的设计信息提升为版本化临床事实，而不是仅保存章节摘要。每条事实记录
（Fact Record）包括事实编号、名称、规范值、单位、适用人群、时间框架、状态、来源、责任角色、
版本、更新时间和前一值。例如数字“220”只有与“计划总随机样本量、双臂合计、包含 15% 脱落”
共同记录时才可比较；“第 24 周”也必须说明它表示主要终点评估、筛选窗口还是随访结束。

事实状态包括草稿、候选变更、需要人工复核、已批准、已拒绝和已被新版本替代。模型抽取的内容
首先进入草稿或需要人工复核状态；只有具备相应职责的人员确认规范值、单位、范围、版本和来源后，
事实才可成为指定范围内的已批准事实。新版本正式批准后，旧版本转为历史状态并继续保留，以支持
历史文件解释、审计和回滚。模型置信度、补丁执行成功或多数来源投票均不能直接改变批准状态。

同一事实可以由多个来源共同支持，来源之间可以是一致支持、补充说明、版本替代、合法范围差异或
待调和冲突。发生冲突时，系统保留各候选值及其证据，生成专业决策请求，说明冲突位置、可选处理、
影响文件和责任角色，而不由模型静默选择一个值。

### 6.2.4 文档单元与证据追溯链

系统将临床试验方案（Protocol）、访视活动计划表、统计分析计划、知情同意书、病例报告表和注册
记录拆分为可独立
定位的文档单元。单元可以是章节、段落、表格行、单元格或注册字段，并记录文档类型、文本、来源和
引用的事实。单元过粗会扩大修改范围，过细则增加维护成本；当前原型以章节和页级定位为主，企业
试点可进一步扩展至表格单元格和字段级。

完整证据链为：

```text
原始文件或注册快照
→ 来源记录及原文定位
→ 版本化事实及批准状态
→ 引用该事实的文档单元
→ 问题记录
→ 修订提案或专业决策请求
→ 沙箱复检
→ 最终人工决定
```

审核人员可以从问题向上追溯原始依据，也可以从事实向下查看受影响文件。任何必要环节缺失时，
系统仍可保存候选，但不得将其表述为来源完整或可自动修订的结论。

### 6.2.5 事实调和与完整性控制

系统比较不同文档前，先统一单位、日期、周数、大小写、范围边界和试验标识符，再判断是否属于同一
研究、同一人群、同一版本、同一时间状态、同一估计目标和同一文档用途。差异被划分为完全一致、
语义等价、合法范围差异、版本替代以及证据不足或真实冲突。只有适用范围相同且规范化结果不兼容，
才进入冲突处置。

例如，方案中的计划入组人数与注册结果中的实际入组人数可以同时成立；临床试验方案的连续研究日与
病例报告表按周期重新编号的“第 1 天”可以由时间映射解释；同一版本中主要终点分别写为第 24 周和
第 28 周，才构成需要进一步裁决的候选冲突。无法分类时，系统转交相应专业角色，而不是默认选择
最新值。

当前 50 个公开案例包含 65 份官方文件和 50 份注册快照。系统保存原始链接、字节数、页数、获取
时间和 SHA-256 摘要；自然候选进一步保存文件名、页码、短原文、候选编号和预测摘要。一个正文
不可提取的文件在构建时被拒绝并由后续案例替补，说明“下载成功”不等于“证据可用”。

该底座维持三项基本不变量：已批准事实必须具有可定位的有效来源；正式修订提案必须追溯到问题记录
和事实；需要人工复核的状态不能由模型置信度直接升级为已批准。来源完整、定位完整和语义正确分别
评价：文件摘要一致只能证明对象未改变，页码齐全只能证明内容可查找，二者都不能代替专业正确性
判断。
## 6.3 多智能体组织层：职责、状态与执行控制

多智能体组织层通过职责分离、结构化状态交换和条件路由协调各项任务，并通过并行汇合、独立复核、
人工授权和异常恢复机制，控制并行修改、模型自评及异常重试可能引发的执行风险。

### 6.3.1 职责分离

多智能体设计用于分离证据整理、内容生成、问题识别、修订建议、质量复核和治理处置。不同智能体
代表不同的任务契约、可见上下文、工具权限和输出类型，不代表其具有相应组织岗位的批准权。同一
底层模型可以在原型阶段承担多个节点的语言推理，但各节点仍须保持独立输入、结构化输出和状态权限。

| 职责类型 | 主要任务 | 允许形成的结果 | 不得执行的动作 |
|---|---|---|---|
| 上下文与证据 | 锁定任务范围、登记来源、提取候选事实 | 任务契约、来源记录、候选事实 | 猜测缺失事实或批准事实 |
| 生成与修订 | 依据已确认事实形成章节或最小修订 | 候选章节、修订提案 | 覆盖正式文件 |
| 专业审核 | 从医学、统计、注册、运营和质量视角识别问题 | 带证据的问题记录 | 自行批准其发现的问题 |
| 独立质量复核 | 在沙箱中检查来源、范围、回归和负对照 | 复检结果、阻断状态 | 用语言质量分覆盖硬失败 |
| 不确定性与治理 | 选择取证、提交或转交，整理专业决策事项 | 取证请求、审核包、专业决策请求 | 冒充专业批准人 |
| 经验管理 | 整理经确认的适用条件、误报和整改经验 | 经验候选 | 未经批准写入组织知识 |

医学审核智能体可以发现终点或安全性表述风险，统计审核智能体可以发现样本量、估计目标或分析集
问题，注册审核智能体可以发现公开字段和监管表述问题；这些输出均为机器候选。正式专业结论仍由
具备相应职责的人员作出。

### 6.3.2 共享状态与条件路由

各智能体通过统一工作流状态交换信息，不传递无约束的完整对话。共享状态包含任务范围、来源、
事实、文档单元、问题记录、修订提案、专业决策请求、质量结果、审计事件和当前流程状态。每个对象
具有稳定编号、版本和状态；每个节点只能更新其职责允许的字段。

例如，专业审核节点提交的问题记录必须包含问题类型、严重程度、文档位置、关联事实和来源；修订
节点必须引用已有问题和事实；质量节点必须分别报告目标问题是否关闭、是否出现新问题以及负对照
是否保持。结构化交接使节点能够被独立测试、重放和替换，也使人工审核能够区分机器发现、机器建议
和人工决定。

流程根据状态条件路由，而不是机械执行所有节点。任务范围不完整时返回补问；来源无效时阻断并要求
重新定位；事实未确认时进入专业事实审核；问题能够由已批准事实唯一修复时进入修订与沙箱复检；
涉及医学、统计或注册选择时生成专业决策请求；复检产生新问题时返回审核与修订；全部机器条件满足
后才形成候选审核包。没有发现问题时仍须检查资料覆盖范围，不能将空结果直接判为通过。

### 6.3.3 并行汇合、人工中断与独立复核

在来源和事实基线冻结后，医学、统计、注册、运营和质量视角可以并行检查。并行节点只允许新增问题
记录或提出候选建议，不得同时改写正式文本。汇合时按事实、文档位置和语义范围合并重复问题；证据
或结论相反的记录保留为冲突，不通过多数投票自动消除。

多个修订提案在应用前接受区间和事实冲突检查。非重叠且前置文本一致的补丁可以合并；修改范围重叠、
同一事实存在不同候选值或不同专业角色提出相反处置时，系统形成冲突包，列出各方证据、影响范围和
可选方案，并转交有权角色决定。该机制避免并行执行产生“最后写入者覆盖前者”的结果。

涉及专业判断时，工作流进入可恢复的人工中断状态。专业决策请求记录待决问题、证据、选项、影响
文件、所需角色和剩余不确定性。恢复执行前，系统核对决定者角色、决定对象版本和输入摘要；等待
期间来源或文件发生变化时，原决定须重新确认适用范围。

独立质量复核不读取生成节点的自我评价，而以实际候选文件、批准事实、确定性规则、负对照和未决
人工状态为依据。质量门检查来源完整性、事实状态、阻断问题、补丁局部性、摘要一致性、新增问题、
人工授权和回滚信息。语言质量评分仅作为辅助观察，不参与硬门禁补偿。

### 6.3.4 异常恢复与经验准入

网络超时、临时模型故障和可重试解析错误可以在限定次数内重试。系统通过任务编号、节点编号和输入
摘要识别重复调用，防止重复创建事实或重复应用补丁；节点结果写入检查点，恢复时从最后一个完整
状态继续。结构化输出修复失败、摘要不一致或权限校验失败时采用失败关闭。

专业证据不足、角色意见冲突和未知依赖不属于技术错误，不通过重复调用模型解决。此类情况应补充
来源、请求人工判断或保持转交状态。若连续迭代未关闭问题、未降低不确定性或反复读取同一证据，
系统终止自主循环并升级人工。

任务结束后，可将已经确认的“问题—证据—决定—修订—结果”整理为经验候选，并记录治疗领域、
试验阶段、文档类型、适用条件、来源、版本和批准者。经验只有在来源有效、适用范围匹配且经过专业
批准后才可供后续任务检索；法规、企业程序或来源发生变化时，原经验应失效或缩小适用范围。语义
相似仅用于排序，不构成经验适用性的判定。
## 6.4 不确定性驱动的自然语言事实计算

本节解决概率型人工智能推理的核心问题：临床文档中的自然语言事实如何转化为可比较、可计算且
保留不确定性的对象，并在证据不足或解释竞争时主动决定下一步行动。其困难不只是识别关键词和
数字：相同数值可能属于不同实体、单位、人群和时间坐标，不同措辞可能表达同一事实，同一句话也
可能对应多种合理的医学或统计解释。若直接把整段文本压缩为相似度或模型置信度，系统既无法确定
该关系能否驱动正式修订，也无法说明判断依据和剩余风险。

TrialCompiler 将这一过程组织为“文档单元化—类型化候选—竞争假设—主动取证—可解释验证”。
规则负责稳定识别结构化字段和数值约束，大模型负责提出语义归属、跨句关系和竞争解释；两者的
输出均先成为带来源和状态的候选，不能直接写入正式事实或文档。确定内容进入约束计算，歧义内容
进入有限假设和不确定性计算，无法安全收敛的任务转交相应专业人员。

### 6.4.1 从自然语言到类型化候选

系统首先按标题路径、段落、表格行列和字段边界切分文档，为每个文档单元绑定文件版本、页码或
结构位置、原文、内容摘要和适用范围。规则抽取器识别日期、周数、剂量、样本量、试验标识符、
比较符和枚举值；大模型提出实体归属、语义角色、跨句指代、潜在事实关系和证据缺口。两类结果
统一写入类型化事实候选：

$$
z=(e,a,v,u,c,t,p,s,l,r),
$$

其中 $e$ 为实体，$a$ 为属性，$v$ 为规范值，$u$ 为单位，$c$ 为比较关系，$t$ 为时间锚点，
$p$ 为适用人群，$s$ 为作用域，$l$ 为来源定位，$r$ 为证据状态。例如“主要终点在第 24 周评估”
被表示为“主要终点—评估时间—24—周—等于—随机化后—目标人群—当前版本—来源位置—候选或
已批准”。数字“24”只有与上述字段共同出现时才具有可比较意义。

规范化根据内容类型生成不同的计算对象：时间表达转成时间点或区间，剂量和样本量转成带单位数值，
“大于”和“不小于”转成不同不等式，分析人群转成集合谓词，策略选项转成有限状态。无法唯一
规范化的语句保留多个解释，不强制选择单一值。

例如，试验方案写明“主要终点在随机化后第 24 周、面向全部随机化受试者评估”，统计分析计划却
写明“主要分析在第 28 周、基于至少接受一次治疗且具有基线后测量的全分析集实施”。为便于展示，
省略两者相同的来源定位和证据状态后，系统分别形成

$$
z_P=(\text{主要终点},\text{评估时间},24,\text{周},\text{随机化后},\text{全部随机化受试者}),
$$

$$
z_S=(\text{主要终点},\text{分析时间},28,\text{周},\text{随机化后},\text{全分析集}).
$$

两个对象在终点实体和时间原点上可以对齐，但在时间点和人群范围上不同。系统据此生成“第 24 周
与第 28 周是否属于同一分析目标”“全部随机化人群与全分析集是否允许不同”两项可回答的问题，
而不是仅凭数字不同直接判错。若来源确认主要终点时间已经由第 24 周变更为第 28 周，时间差异可
进入影响传播；人群差异仍需统计人员判断是合法分析范围、文本不一致还是估计目标定义缺失。

候选关系采用确认、待确认和否决三种证据状态：

$$
R_{ij}\in\{confirmed,candidate,rejected\}.
$$

结构化字段匹配、明确引用或人工确认形成已确认关系；仅由语义模型提出的关系保持待确认，只能
触发复核和取证；已经确认的合法差异或无关关系进入否决状态，作为防止误传播的负对照。由此，
自然语言输出先成为可审计候选，再经过来源核验和状态升级进入后续计算。

### 6.4.2 竞争假设与不确定性状态

当同一文本存在多个合理解释时，系统建立有限假设集合

$$
H_u=\{h_1,\ldots,h_m\}.
$$

若先验和观测模型均有明确依据，可维护信念状态

$$
b_t(h)=P(h\mid o_{1:t}),\qquad \sum_{h\in H_u}b_t(h)=1,
$$

并在获得新证据后按照贝叶斯规则更新。若缺少可辩护的先验、似然或校准数据，系统只记录候选排序、
支持证据、竞争解释和“概率未校准”状态，不把模型即时生成的小数解释为真实正确率。

系统区分诊断信号、不确定性状态和授权状态。规则告警、来源冲突、模型自一致性、语义熵、验证器
分歧和工具失败可以提示风险，但不能直接等同于错误概率；候选概率较高也不等于证据充分，更不等于
已经获得专业授权。模型的词元似然容易受到长度和措辞影响，自述置信度通常存在过度自信，多次采样
的一致性无法发现“稳定地答错”，语义熵仍依赖采样和语义聚类，多个验证器的相关错误也不能通过
简单多数票消除。因此，不确定性信号必须记录计算方式、适用任务、校准状态和已知失效模式。

对跨文档事实调和，可将候选解释定义为“同一事实”“合法范围差异”“版本差异”“来源错误”或
“证据不足”等有限状态。新来源、确定性规则结果和人工回答作为观测更新该状态。未知观测、来源
越权或校准集与测试集混用均采用失败关闭；系统保留原假设和新证据，不允许大模型通过改写理由
追认先前结论。

### 6.4.3 主动取证与停止策略

不确定性在本系统中不是展示分数，而是改变下一步行动。候选行动包括读取指定来源、请求缺失附件、
调用确定性工具、增加语义采样或验证预算、询问医学或统计人员、降低主张具体程度、暂缓处理和
拒绝形成结论。对行动 $a$，期望信息增益定义为

$$
EIG(a)=H(b_t)-
\mathbb{E}_{o\sim P(o\mid a,b_t)}
[H(b_{t+1}\mid a,o)].
$$

考虑证据成本 $c(a)$、风险权重 $\lambda$ 和允许行动集合 $\mathcal A_{safe}$，系统选择

$$
a^*=\arg\max_{a\in\mathcal A_{safe}}
\{EIG(a)-\lambda c(a)\}.
$$

成本不仅包括模型调用，还包括文档页数、外部查询、人工时间、等待时间和敏感数据暴露。信息增益
较高但越权的行动不属于允许集合；不能区分竞争假设的重复读取也不应继续消耗预算。每次行动记录
选择前的假设状态、预期收益、实际观测、状态更新、来源版本和成本，使评估者能够判断智能体是否
真正因为不确定性而取证。

停止策略由证据、风险、预算和硬门禁共同决定：

$$
Decision_t=
\begin{cases}
commit, & Risk_t\le\tau_r\ \land\ HardGate=1;\\
acquire, & \max_a(EIG(a)-\lambda c(a))>0\ \land\ Budget_t>0;\\
defer, & QualifiedHumanAction\ available;\\
abstain, & \text{otherwise}.
\end{cases}
$$

这里的提交只表示候选可以进入确定性编译，不代表可以正式发布。若连续若干轮没有关闭问题、没有
降低风险或反复访问同一来源，系统停止自循环并转交人工。这样，“不知道”会产生明确的取证、暂缓
或拒绝动作，而不是变成更长的自然语言推理。

### 6.4.4 可解释输出与反事实验证

TrialCompiler 不把流畅的解释文字视为可信解释，而是输出可重放的推理记录。每个候选至少携带：
主张及适用范围、规范事实和来源版本、引用位置、竞争解释、剩余不确定性、实际工具与取证轨迹、
候选依赖、建议责任角色和未清偿的专业决策。医学审核者可查看临床语义和风险证据，统计人员可
查看估计目标、分析集和缺失处理逻辑，质量人员可查看来源、版本和轨迹；不同角色看到的呈现可以
不同，但底层证据对象保持一致。

系统区分来源可追溯、运行可观察、解释表面合理、行为忠实和内部机制因果五个层次。来源与工具轨迹
能够支持工程审计，但不自动证明解释忠实；大模型生成的推理文字也不能因为听起来合理就被当作模型
内部机制。当前可验证的是行为层忠实性：固定模型、提示、工具和随机配置后，移除被声明为必要的
证据，或替换为支持竞争假设的对立证据，判断和行动应发生预期变化。

对声明证据必要性的 $N$ 个案例，必要性翻转率可定义为

$$
NFR=\frac{1}{N}\sum_{i=1}^{N}
\mathbf{1}[d_i^{-e}\neq d_i].
$$

还应检查对立证据方向敏感性、随机重放方差和无法在真实轨迹中找到依据的解释比例。上述指标只支持
行为层忠实性，不证明模型内部神经机制。自然语言表示准确性、候选依赖发现率、不确定性排序、
选择性风险和反事实敏感性均在第七部分通过冻结数据和独立评分器评估；未完成真实校准时，不声称
输出概率已经代表临床总体正确率。

## 6.5 携证临床编译：影响传播、受控修订与安全保证

概率型推理产生的是带来源、不确定性和竞争解释的候选，不能直接改写正式文档。携证临床编译将
已确认候选转换为依赖图中的计算对象，以确定性规则计算影响、限制补丁范围、重新编译修改结果，
并形成可供人工治理的证明记录。大模型负责开放语义发现和取证规划，编译平面负责类型、来源、
状态、边界和发布门禁；两者通过类型化对象交接，而不是通过自然语言相互覆盖。

### 6.5.1 依赖图与有效影响范围

设来源、规范事实、文档单元、问题、修订提案和人工决定的集合分别为
$\mathcal S,\mathcal F,\mathcal U,\mathcal I,\mathcal R,\mathcal H$，完整图为

$$
G=(V,E),\qquad
V=\mathcal S\cup\mathcal F\cup\mathcal U\cup\mathcal I\cup\mathcal R\cup\mathcal H.
$$

文档单元是可定位、可比较和可局部修订的最小业务对象，例如方案中的入排标准条款、访视活动计划
表的一行、统计分析计划的分析集定义、知情同意书的风险段落或注册记录中的终点字段。边记录来源、
依赖、问题识别、修订和授权关系，并保存关系类型、证据状态、创建者和版本。明确字段引用、结构化
映射或人工确认的关系可以进入正式传播，仅由模型提出的语义关系保持候选状态。

正式传播使用显式边和已经批准的语义边：

$$
I^*(f)=Reach(f;E_{explicit}\cup E_{approved}),
$$

待复核影响集为

$$
I_{review}(f)=Reach(f;E_{candidate})\setminus I^*(f).
$$

$I^*(f)$ 可以进入确定性检查和补丁规划，$I_{review}(f)$ 只能生成复核任务。图按事实和文档版本
增量维护；事实替代保留前后版本关系，文档换版后依据章节路径和内容摘要重新定位。定位失败的边
被标记为失效并阻断补丁，不能继续使用旧偏移量。该机制保证已经登记的依赖不会在可达性计算中
遗漏，但不声称所有自然语言隐含依赖均已被发现。

### 6.5.2 变更分类与原子修订

一次事实变化表示为

$$
\Delta f=(f,v_0\rightarrow v_1,source,scope,actor,time).
$$

系统计算 $I^*(f)$，并在实体、属性、单位、时间、人群和作用域一致的前提下检查旧值残留、规范值
缺失、边界冲突、来源失配和范围失配。无上下文字符串替换不属于有效检查：数字“24”可能表示周数、
人数或序号，另一个试验标识符也可能是合法的相关研究引用。

受影响单元分为四类：必须更新表示新值已批准、依赖明确且位置唯一；可能更新表示存在语义关联但
表达或范围不唯一；专业决策表示需要医学、统计或注册角色选择策略；历史保留表示冻结版本只记录
差异而不回写。数值、日期、单位、严格不等式和同一事实的旧值适合确定性处理；估计目标、分析集、
缺失数据、多重性、终末事件和注册策略通常只能定位矛盾并形成专业决策请求。

对必须更新的单元，系统生成原子修订提案：

$$
o=(section,start,end,before,after,fact,source,precondition).
$$

前置条件要求目标版本和目标区间未变化、修改前文本仍然一致、事实仍处于批准状态且来源版本有效。
补丁只修改由批准事实唯一确定的表达，不顺带重写邻近段落。区间不重叠且前置条件一致时，可以从
高偏移位置向低偏移位置合并；区间重叠、目标值竞争、范围不清或存在多种合理解释时，系统生成冲突
对象并停止自动应用。所有补丁先作用于沙箱副本，原文及其摘要保持不变。

### 6.5.3 重新编译与携证输出

补丁执行不等于问题解决。沙箱文档重新进入抽取、规范化、依赖计算和全部确定性检查，相当于一次
局部重新编译。闭环要求规范新值已经出现、应删除位置不再残留旧值、实际修改范围等于批准范围、
事实—来源—修订链完整、原问题关闭且没有新增阻断问题。任一条件失败时，状态保持为部分修复或
发现回归，不能被“补丁已应用”这一技术状态掩盖。

模型输出必须通过结构校验、引用存在性检查、作用域检查和敏感字段检查后才能进入编译。外部文档中
的指令性文字作为不可信数据处理，不能改变系统角色、工具权限、输出路径和发布状态。运行记录保存
模型与提示版本、采样配置、输入摘要、工具调用、类型化输出和失败代码；输入或模型发生变化时，
旧检查点不能无条件复用。

编译结果不是一份缺少依据的新文本，而是携证结果

$$
\Pi=(\pi_s,\pi_z,\pi_i,\pi_p,\pi_v,\pi_h),
$$

分别记录来源、类型化表示、影响范围、补丁、重新编译结果和人工授权状态。机器阶段可以形成来源
充分、范围明确、复检通过并携带剩余不确定性的候选，但不能自行赋予医学、统计、注册或发布效力。
相应效力由后续治理环节中的责任角色授予。

### 6.5.4 条件安全性与适用边界

携证编译具有三项范围明确的工程性质。第一，在有效边集合中已经登记的依赖必然进入
$I^*(f)$，但这一传播覆盖性不延伸到尚未发现或尚未确认的隐含关系。第二，若所有原子编辑区间均
位于批准范围 $T$、区间互不重叠且前置条件仍成立，则

$$
Changed(X,X')\subseteq T,
$$

即补丁不会修改非目标区域。第三，若硬约束为 $g_1,\ldots,g_m$，人工授权为 $H$，则

$$
Release(X')=H(X')\land\bigwedge_{j=1}^{m}g_j(X').
$$

任何硬约束失败或授权缺失都会使发布条件为假，语言质量分和模型置信度不能补偿。

给定当前状态 $X$ 和事实变更 $\Delta f$，编译过程写为

$$
\mathcal C(X,\Delta f)=(X',q,\Pi),
\qquad
q\in\{release,block,defer\}.
$$

若 $q=release$，则证明记录必须支持

$$
Traceable(X')\land
RegisteredImpactDisposed(X')\land
LocalPatch(X')\land
Revalidated(X')\land
HumanAuthorized(X').
$$

这一定义保证系统不会在已声明的证明义务未完成时静默释放结果。以主要终点时间点由第 24 周改为
第 28 周为例，方案终点段、访视活动计划表和注册字段中的同一事实可进入必须更新，统计分析计划若
同时涉及分析窗、缺失处理或估计目标则进入专业决策；只有所有已登记依赖均得到处置、原子补丁通过
重新编译且统计决策完成后，该项变更才可进入发布治理。

上述保证适用于已经识别、登记并纳入检查的对象，不证明任意临床文档均能被完整抽取，也不证明
所有隐含依赖、语义冲突和专业策略都能被机器正确识别。自然语言表示、候选依赖召回、不确定性排序
和解释忠实性由实验评估；样本量、估计目标、缺失数据和监管路径由相应专业角色裁决。准确率、
精确率、召回率、F1 值和 Wilson 区间属于第七部分的实验统计，不属于数学正确性证明。

## 6.6 治理出口：人工授权、整改状态与发布边界

机器输出只有在组织授权中才能获得正式效力。人工在这里不是模型置信度不足时临时加入的兜底，
而是事实生效、专业风险接受和正式发布的授权主体。系统自动化的是证据整理、影响计算、候选修订
和复检，不是专业责任本身。

### 6.6.1 批准层级、作用域与有效期

TrialCompiler 把四种容易混淆的批准分开。`fact approval` 确认某个值、定义或策略在指定范围内可
进入规范事实层；`finding disposition` 决定问题成立、误报、接受风险、需要补证或超出范围；
`repair approval` 允许特定补丁作用于特定文档版本和区间；`release authorization` 才允许修订后
材料进入下一正式阶段。

四者形成有向前置关系，而不是一个布尔 `approved=true`：

$$
FactApproved\not\Rightarrow RepairApproved,
\qquad
RepairApproved\not\Rightarrow ReleaseAuthorized.
$$

例如批准“主要终点时间点为 Week 28”，并不意味着批准系统生成的所有 Week 24→28 替换；SAP
中的分析窗可能需要另行统计判断。补丁在沙箱中通过也不等于可以发布，因为独立 QA、剩余 finding
和文档版本仍需核对。这一分层防止一次宽泛点击被解释成无限授权。

**授权对象、作用域与有效期。**

每项授权表示为

$$
Auth=(subject,role,object,action,scope,validity,evidence,signature),
$$

其中 `object` 可以是事实、finding、Decision Request、补丁或发布包；`scope` 限定项目、国家/
地区、文档类型、章节、版本和适用人群；`validity` 规定授权时点和失效条件。每次决定保存决定人、
资格角色、输入证据、理由、被授权对象摘要及电子签名或等效审计标识。

授权绑定内容摘要而不仅是文件名。对象文本、来源版本、事实版本、影响范围或候选补丁发生实质变化
后，旧签字状态转为 `stale_authorization`，必须重新确认，不能静默继承。撤销批准时不删除旧记录，
而是新增 `revoked` 事件并记录原因和替代决定，以保留完整时间序列。

### 6.6.2 专业职责与决策请求

医学负责人负责疾病定义、临床终点含义、安全风险和医学可接受性；统计负责人负责 estimand、分析
集、样本量、缺失数据、多重性和敏感性分析；注册负责人负责注册字段、地区路径和公开披露范围；
临床运营角色负责访视可执行性、中心流程和招募约束；数据管理角色负责 CRF 字段、编码和数据流；
QA 负责版本完整性、审计链、程序遵循和发布前独立复核。

系统管理员可以配置账号、运行环境和技术工作流，但不能借管理员权限代替医学、统计、注册或 QA
签字。候选补丁的提出者原则上不能独自完成同一补丁的最终 QA 放行；高风险事实的批准者与发布者
应保留可识别的职责分离。若小团队必须兼任角色，系统应显式记录角色兼任和补偿性复核，而不是
伪造独立性。

**将不确定性转成可回答的专业问题。**

当证据不足、多个解释均合理或修改方式不唯一时，系统生成 Decision Request。请求至少包括：
待决问题、竞争事实或假设、支持与反对证据、来源权威和版本、受影响文档单元、各选项的下游影响、
模型与规则分歧、剩余不确定性、建议责任角色、答复期限和不作决定的风险。选项必须允许“需要更多
证据”“转交其他角色”和“均不接受”，避免用预设选项迫使专家认可模型建议。

专家答复不能只有一段自由文本。结构化决定包括所选处置、理由、适用范围、生效事实、是否允许
机器补丁、必须人工编辑的部分、复检要求和失效条件。例如 NCT ID 不一致可能源于文件身份错误、
相关研究合法引用或来源范围错误。即使专家确认“需要处理”，仍可返回
`approved_but_manual_edit_required`；这表示问题处置已获批准，但机器尚不能唯一决定应改编号、
改文件分类还是改来源范围。

### 6.6.3 整改状态与发布门禁

主整改路径为：

```text
open
→ triaged
→ decision_pending
→ approved_for_repair
→ repaired_in_sandbox
→ revalidated
→ qa_approved
→ released
```

`rejected`、`false_positive`、`accepted_risk`、`needs_evidence`、`manual_edit_required`、
`partially_repaired`、`regression_detected`、`stale_authorization` 和 `withdrawn` 是独立分支，不能
压缩成一个“已处理”。每次状态转移具有明确前置条件、允许角色、必需产物和退出证据；非法跳转
被拒绝并写入审计日志。

`accepted_risk` 也不是关闭问题的快捷键。它必须包含风险说明、影响范围、批准角色、补偿措施、
复审日期和是否阻断发布。`false_positive` 必须保存负对照或范围解释，后续规则回归时用于防止同类
误报。`needs_evidence` 必须产生具体取证任务和预算，而不是让 finding 无限停留在开放状态。

**发布门禁与非补偿原则。**

设阻断 finding 数为 $B(X)$，高风险决定是否齐备为 $H(X)$，沙箱复检是否通过为 $V(X)$，独立
QA 是否完成为 $Q(X)$，审核对象摘要是否仍一致为 $D(X)$，则

$$
Release(X)=
\mathbf{1}[B(X)=0]\cdot H(X)\cdot V(X)\cdot Q(X)\cdot D(X).
$$

任一项为 0，发布结果均为 0。语言质量分、模型置信度、修订数量和截止日期压力都不出现在补偿项
中，因此不能用“整体看起来很好”抵消一个未解决的硬门禁。紧急情况下可以缩小发布范围、发布经
批准的部分版本或走有记录的例外流程，但不能把失败状态改名为通过。

发布包至少冻结：输入文件与摘要、批准事实快照、finding disposition、Decision Request 决定、
实际补丁、沙箱复检结果、剩余非阻断风险、角色签字、软件/规则/模型版本和最终输出摘要。任何正式
输出都可由这些对象重放到相同的治理结论；无法重放的结果不能作为受控发布证据。

### 6.6.4 审计回滚与完整授权示例

审计日志采用追加式事件记录，至少包含 actor、role、timestamp、before/after state、object digest、
reason、evidence 和 correlation ID。查看、取证、提出修改、批准、应用、撤销和发布分别记录，不能
只保存最终文档。敏感内容按最小必要原则展示，日志访问本身也应被审计。

若发布后发现来源错误、专业决定撤销或补丁产生延迟回归，系统根据事实版本、影响集和发布摘要定位
受影响产物，创建 recall/rollback 工单，并恢复到最近一个已批准状态；不能通过删除事件伪造从未
发生。事后确认的合法差异、误报模式和人工补边可以形成 Experience Candidate，但只有经过专业
复核、去除项目敏感内容并限定适用范围后，才可成为后续运行可检索的组织经验。

**完整授权示例。**

以 enrollment 从 120 调整为 150 为例：来源更新首先产生候选事实，临床运营或相应责任角色在指定
国家、版本和招募范围内完成事实批准；系统依据第 6.5 节计算方案、注册记录、预算假设和
中心计划的影响集。纯数字旧值可形成原子补丁，但若样本量 120 同时来自统计功效计算，系统不得仅
做全局替换，而需统计角色判断这是计划招募数变化还是统计假设变化。

随后 finding disposition 分别确认各处是旧值、合法历史还是需专家决定；repair approval 只覆盖
已确定的 Protocol 与注册字段；补丁进入沙箱并复检旧值残留、统计一致性和负对照。QA 核对实际
修改范围、来源链和未决统计问题。只有统计问题已关闭或被明确排除在本次发布范围外、所有摘要仍与
签字对象一致，release authorization 才能成立。该例说明人工授权并不降低自动化价值，而是让
Agent 的高效传播与专业责任形成可审计接口。

## 7.2 原型验证总体设计

原型验证围绕三个递进问题展开。第一，输入资料是否真实、可读并可冻结复现；第二，系统是否能够
完成文档角色识别、确定性缺陷检测和自然语言候选发现；第三，在候选问题获得模拟人工批准后，系统
能否完成局部修订、保持来源链并避免引入新问题。三个问题分别对应输入可靠性、任务有效性和修订
安全性，结论不得跨层外推。例如，文件下载成功不能证明缺陷识别准确，受控缺陷检测成功也不能证明
自然临床语义已经达到总体准确率。

评测遵循来源先冻结、预测先于金标准、案例级隔离、正负对照并存以及生成与评分分离的原则。同一
研究的注册快照、试验方案、统计分析计划、案例约束、标签和修订记录始终属于同一案例，避免同一
研究的信息跨集合泄漏。50 个案例划分为 30 个开发案例、10 个校准案例和 10 个独立测试案例。
开发集仅用于规则编写、错误分析和流程调试，不用于训练模型参数；校准集用于选择阈值和处置策略；
独立测试集在策略冻结后一次性评价。该划分名称反映实际用途，不将规则开发误写为模型训练。

输入可靠性由 50 份注册快照、65 份官方 PDF、文件摘要、页数和文本可提取性验证。任务有效性由
三组互不替代的实验构成：官方文档元数据支持角色识别评价；带正负对照的受控缺陷支持确定性检查
评价；真实 PDF 页级候选用于观察系统是否能够定位复杂语义并保留不确定性。修订安全性则在模拟
批准条件下检查新值写入、旧值移除、修改局部性、来源保持、问题关闭和新增问题。自然候选在外部
专业裁决完成前只报告候选数量、来源定位、处置路由和证据完整性，不报告准确率。

所有正文结果均由冻结机器产物复算。语料规模来自 build_report.json 和 corpus_profile.json，
角色识别结果来自 role_baseline_results.json，受控缺陷结果来自 seeded_defect_results.json，
自然候选摘要来自 natural_candidate_evaluation_pre_gold.json，修订闭环结果来自
simulated_human_repair_report.json。汇总结果可下钻到案例约束、逐条标签、预测和修订记录，
避免以手工录入数字代替可复核证据。

## 7.3 NCT04683926 一致性审核案例

公开案例 NCT04683926 / OMNI-PAIN-103 包含 Protocol、SAP、ICF 和注册资料。项目建立 27 条
人工事实基线、正例、合法差异负对照和独立金标准评分器。初始规则在严格数值边界、跨来源
时长和合法双时间轴方面存在漏报与误报；增加相应规则后，冻结结果为 TP=8、FP=1、FN=0、
TN=2，Precision=0.8889、Recall=1.0000、F1=0.9412、负对照准确率=1.0000。

额外的假阳性来自一条未匹配金标准的可报告语义提醒，而不是把两个负对照判错。该结果证明当前
案例上的工程改进，不代表跨疾病、跨文档类型的临床总体准确率。

独立评分器依据预先冻结的金标准条目计分，而不是依据模型自行生成的问题名称计分。一个阳性
金标准只有在事实、章节、语义谓词和必要证据均匹配时才计为真阳性；负对照被错误报告时计为
假阳性；未匹配任何正例的额外提醒也计为假阳性。这样可以防止系统通过改写问题名称或输出大量
模糊提醒获得虚高召回率。初始版与整改版对比如下：

| 指标 | 初始版 | 整改版 |
|---|---:|---:|
| TP / FP / FN / TN | 4 / 2 / 4 / 1 | 8 / 1 / 0 / 2 |
| Precision | 0.6667 | 0.8889 |
| Recall | 0.5000 | 1.0000 |
| F1 | 0.5714 | 0.9412 |
| 负对照准确率 | 0.5000 | 1.0000 |

整改新增严格给药间隔、跨来源住院时长、筛选窗口包含边界和来源链完整性规则，并为合法
Protocol/CRF 双时间轴增加 scope-aware 负对照。由此，已知漏报被补齐，同时已知误报被
消除；这比单纯扩大报警数量更能说明规则改进的有效性。

## 7.4 Metformin-PAD 受控生成案例

该工作流以完全合成或历史重建测试包验证分阶段可见性。Phase 1 仅读取初始资料并生成候选
事实、Synopsis 和章节；Phase 2 在冻结前期结果后加入新证据并增量修订；隐藏参考资料只在
生成结束后交给独立评估器。每次运行保存输入与提示摘要、模型、来源边界和 checkpoint。

首次软评分表面较高，但模拟医学、统计和注册审核发现样本量无法复现、estimand 与分析集
不一致、终末事件被不当纳入 MAR 插补、模拟监管材料被权威化等问题。新增机器门后，运行被
标记为 `blocked_machine_gate` 和 `blocked_missing_evidence`。这一失败整改说明系统亮点不在
“生成得像”，而在发现高分不可信后能够阻断并形成角色化整改工单。

机器门将问题编译为结构化整改工单，而不是只输出一段批评。每条工单记录责任角色、受影响
事实和产物、必须执行的修订动作、所需证据及可机器检查的退出条件。例如样本量工单要求统计
人员提供可复现公式或模拟输出，并同步 Synopsis、统计章节和相关登记字段；estimand 工单
要求明确人群、变量、治疗条件、伴发事件策略和汇总量；模拟监管材料工单要求删除“监管已
认可”类无依据表达并恢复资料的合成属性。只有退出条件全部为真，工单才可关闭。

这一案例也验证了分阶段资料隔离：Phase 1 运行不能看到 Phase 2 新证据，Phase 2 只能在
前期摘要冻结后增量处理，Evaluator-only 资料不能回流生成上下文。每次运行保存 checkpoint
digest，因此可以判断后续结果是合法增量还是历史输入被静默改变。

## 7.6 200 个受控缺陷 Gold 与检测结果

在 50 个真实案例上派生两类可确定缺陷：canonical NCT identifier 冲突和 enrollment 变更
未传播。每例每类各有一个正例和匹配负对照，共 200 个标签，正负各 100。标识符负对照包含
正确 NCT 和作为相关研究引用的另一 NCT；enrollment 负对照包含已经正确传播的新值，因此
不是容易通过的空文本。

独立测试集的 10 个案例、40 个标签上得到 TP=20、FP=0、FN=0、TN=20，Precision、
Recall、F1 和 Accuracy 点估计均为 100%，Accuracy Wilson 95% 区间为 91.24%—100%。全量
200 标签判断正确，区间为 98.12%—100%。结论严格限定为两类受控确定性任务。

Wilson 区间用于避免在分母较小时把 100% 点估计解释成确定规律。对成功数 $k$、样本数 $n$
和正态分位数 $z$，区间中心与半宽为

$$
center=\frac{\hat p+z^2/(2n)}{1+z^2/n},
$$

$$
half=\frac{z}{1+z^2/n}
\sqrt{\frac{\hat p(1-\hat p)}{n}+\frac{z^2}{4n^2}},
\qquad \hat p=k/n.
$$

因此 40/40 的下界仍只有约 91.24%，200/200 的下界约 98.12%。报告区间的目的不是削弱
结果，而是明确现有样本对总体结论提供了多强的证据。

这一层的“真实”仅指底层 case contract 和来源来自真实公开研究；缺陷本身是按预定义规则受控
注入的，标签由注入操作确定，而不是从自然临床文本中由专家发现。两类任务的判定边界与检测器
高度一致，适合验证标识符 scope、事实传播、负对照保护和回归稳定性，不适合证明大模型语义推理
能力。把这一限制与 7.7 的未裁决自然候选并列报告，正是为了避免用容易复现的确定性任务代替困难
的自然语义评价。

## 7.7 真实自然候选与不确定性处理

系统从 65 份真实 PDF 中按召回优先策略挖掘 1,393 条自然差异候选，再按案例与类别分层选择
上下文质量较高的候选，形成 197 条、
48 个案例、8 个类别的队列，覆盖分析人群、缺失数据、多重性、主要终点时间、终末事件、
试验标识符和入组范围等。每条具有文件名、页码、短原文和所需角色。

数字差异可能表示计划/实际、筛选/随机化、总样本/亚组；另一 NCT 可能只是相关研究引用。
因此系统没有自行制造 Gold：197 条均保持 `candidate_not_gold`，0 条自动确认，197 条全部
路由专业复核。裁决前预测摘要已冻结。由于当前没有时间进行高成本外部裁决，本次不计算这
部分的 Precision、Recall、F1 或 Accuracy；在没有对漏掉的真实问题进行 Gold 标注前，也不能
计算或声称候选挖掘 Recall。当前可报告的只有队列规模、类别覆盖、路由行为和来源定位完整性，
并把该队列作为后续 POC 资产。

候选分布为：analysis population 30 条、enrollment scope/state 43 条、estimand/intercurrent
event 2 条、missing data 27 条、multiplicity 10 条、primary endpoint/time 39 条、terminal
event 41 条、trial identifier scope 5 条。系统冻结 disposition 中，43 条属于入组范围或
状态不明确，69 条需要统计语义裁决，80 条需要医学与统计上下文，5 条需要 NCT 范围复核。
所有 197 条都有文件名和页码，来源定位完整率为 100%。

这组数据的价值不是提供一个新高分，而是暴露真实文本中的困难分布：同一数字可能代表不同
人群和时间状态，“death”可能是 SAE 定义而非 estimand 缺陷，另一个 NCT 可能是相关研究，
“intention-to-treat”出现也不自动证明分析集矛盾。系统选择 defer 而不是 auto-confirm，体现
了不确定性治理在自然数据上的实际作用。

## 7.8 模拟人工批准后的修订闭环

项目在 50 例上构造 200 个“发现—决定—修订—复检”场景，模拟统计或数据治理角色批准
100 个缺陷、驳回 100 个负对照。在 100 个模拟批准缺陷中，50 个 enrollment 问题具备安全
确定性补丁，另外 50 个 NCT ID 问题需要人工编辑。因此闭环成功率的预定义分母是 50 个
`auto_repair_available` 场景，而不是全部 100 个已批准缺陷。对这 50 个场景，50/50 新值
写入、旧值移除、仅改变目标章节、来源链保持、finding 关闭，新增 finding 为 0；100 个负
对照均未变化。成功率点估计为 100%，Wilson 95% 区间为 92.87%—100%。

另外 50 个 NCT ID 冲突虽获“需要处理”的模拟批准，系统仍拒绝未经证明的字符串替换并转
人工编辑。它们不是修订失败样本，而是按预定义安全策略正确进入 `manual_edit_required`；若把
它们排除在分母之外却不说明原因，就会形成选择性报告，因此本节同时报告 50 个可自动修订和
50 个必须人工编辑的处置。该结果验证授权与执行能力是两个不同门槛。模拟角色只验证流程机制，
不构成真实医学、统计或注册批准。

每条闭环记录同时保存 before/after digest、finding ID、提案数、改变章节、规范值是否出现、
旧值是否消失、来源链是否保持、复检后剩余 finding 和新增 finding。验收不是只比较最终
字符串，也不是只检查原报警消失。50 个可修订场景的 `semantic_patch_correct`、
`minimal_scope_patch`、`source_trace_preserved` 和 `finding_closed_after_patch` 均为真；
`new_finding_after_patch_count`、`negative_control_changed_count` 和 `unexpected_change_count`
均为 0。独立测试集的 10 个可修订问题同样全部闭合。

## 7.9 失败案例与机器门禁整改

验证过程中修复了权威状态与摘要漂移、0 finding 的 vacuous pass、软评分掩盖硬失败、固定
关键词误作角色真值、过度并发取证和只检查报警消失而不检查补丁语义等问题。当前摘要由权威
状态重新物化；未闭合 finding 进入决策队列；角色 Gold 使用官方元数据；补丁复检同时检查
新旧值、范围、来源链和新增问题。报告保留这些失败，是为了证明系统能够从暴露的问题形成
可执行整改，而非只展示成功案例。

| 暴露问题 | 风险 | 整改 | 回归证据 |
|---|---|---|---|
| state 与 summary 漂移 | 旧摘要误导为可发布 | 摘要由权威状态重新物化 | assurance 一致性测试 |
| 0 finding 得到满分 | 空结果成为 vacuous pass | 无有效覆盖时禁止通过 | 质量门负例测试 |
| 软评分掩盖严重矛盾 | 流畅文本获得虚高评价 | 硬门对软分非补偿 | Metformin 机器阻断 |
| 关键词充当角色真值 | 合并文档产生假阴性 | 官方元数据定角色 | 130 个角色标签 |
| 到达目标前提交全部下载 | 成本与等待失控 | 分批取证、达标即停止 | 50 例构建器测试 |
| 只检查 finding 消失 | 错误改写也可能消警 | 检查值、范围、来源与新问题 | 50 个修订闭环 |

该命令访问持续更新的 ClinicalTrials.gov，不属于冻结结果的位级复现，也不应覆盖已经标注的
`public_corpus_050`。在线重建得到的是新语料版本，需要重新生成摘要、重新检查案例划分，并在需要

JSON 和 GitHub Actions。API 密钥不进入仓库；公开 benchmark 和确定性测试无需模型密钥。

候选和修订结果；`docs/final_submission/engineering/` 保存技术证明、验证报告与运行说明；
`tests/` 保存自动化检查；`.github/workflows/` 保存 CI 配置。独立 scorer、受控缺陷构建器
和模拟人工闭环脚本均可单独运行，使汇总数字能够下钻到逐条记录。

复现时应同时记录 Git commit、Python 与依赖版本、操作系统、模型名称、Prompt 摘要、随机种子、
输入文件摘要、运行时间和输出目录。ClinicalTrials.gov 是持续更新的注册系统，重建语料可能得到
更新文件；因此完成标签后不得静默覆盖冻结 corpus，而应创建新版本并保留两个摘要。涉及外部模型
的运行还需固定模型和提示版本，并区分“服务端同名模型发生更新”和“本地代码完全相同”；确定性
benchmark 则可在无 API key 环境中重复执行。

## 7.11 当前局限、下一阶段计划与验收条件

当前仍缺少自然候选的外部专家 Gold、跨治疗领域和跨语言验证、真实 Agent 轨迹校准、企业
模板与 SOP 适配、生产安全验证和真实业务 ROI。下一阶段应冻结版本后，由医学、统计和注册
人员独立裁决自然候选，再对已冻结预测评分；随后开展人工流程与辅助流程的配对 POC，记录
发现率、错误修订率、影响范围召回率、审核时间和返工。

下一阶段不能只追求把 197 条候选“尽快标完”。至少需要双人独立裁决与分歧仲裁、明确
`confirmed_defect`、`legitimate_difference`、`insufficient_evidence` 和
`out_of_scope_reference` 的操作定义，并报告一致性。预测阈值和处置策略只能在校准集
上选择；独立测试标签在策略冻结后一次性评分。对于不确定性研究，还需预先定义 target event、主要
终点、证据成本、最大取证步数、错误提交和 defer 的代价，避免看到结果后改变成功标准。

扩围条件包括阻断性问题零漏放行、正式文件零静默覆盖、未批准事实不进入正式流程、来源和
审计记录完整、回滚演练通过，以及错误修订率和人工审核时间达到预注册阈值。任何硬条件未
满足，系统保持研究或受控试点状态。基于现有证据，可以将 TrialCompiler 表述为具有真实
公开数据底座、可运行代码、数学控制、不确定性治理、独立评测和安全责任边界的临床文档 AI
研究原型；不能表述为已经替代专业人员、获得临床生产批准或证明总体临床准确率的系统。
